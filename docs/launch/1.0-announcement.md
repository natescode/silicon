# Silicon 1.0 — a WebAssembly-targeting systems language where features are data

**TL;DR.** Silicon 1.0 ships today. It compiles to WebAssembly and (via QBE) to
native binaries on Linux + macOS. The differentiator: every operator,
control-flow construct, and definition keyword in the language is defined
as Silicon source under `src/strata/` and dispatched by a tiny core. Adding
`@my_keyword` to the language is the same kind of work as writing a function.

Install:

```sh
curl -fsSL https://raw.githubusercontent.com/NatesCode/silicon/main/scripts/install.sh | sh
sgl init hello
cd hello
sgl run
```

The whole tour is in [`docs/getting-started.md`](../getting-started.md);
read it after this post.

## What it looks like

```silicon
@type Shape := $Circle r Int | $Rect w Int, h Int;

\\ area (Shape) -> Int
@fn area s := {
    @match(s,
        $Circle r, { r * r * 3 },
        $Rect w h, { w * h })
};

@fn main := {
    c := Circle(5);
    r := Rect(4, 7);
    area(c) + area(r)
};

@export main;
```

`sgl run` compiles, executes in wasmtime, prints `103`. `sgl build --release`
produces a native ELF that does the same.

If you've used Roc, the sum-type syntax will feel familiar. If you've used
Zig, the compile-time stratum mechanism will feel like `comptime` with the
fences in different places. If you've used Rust, the `Option` / `Result` /
`Vec` / `HashMap` / `Rc` stdlib will be familiar; the `@defer` / `@try`
sugar lives somewhere between Go and Rust.

## Why a new language

Silicon's bet is on **language features as data**. In most languages,
adding a new operator or a new control-flow keyword means editing the
compiler — touching the parser, the elaborator, the typechecker, the
lowering pass. Silicon's compiler core does not switch on keyword names.
Every keyword lives in a `.si` file under `src/strata/`:

```silicon
@stratum Plus := {
    Compiler::register::operator('+');
    Compiler::on::lower('+', Plus_lower);
};
```

`@if`, `@loop`, `@match`, `@fn`, `@global`, `@struct`, `@type`, `@enum`,
`@defer`, `@try` — all of them are strata. The TypeScript compiler has
no special-case branches for them; it walks the AST, fires registered
handlers, and emits the IR the handlers built.

This means users (and future-you-editing-the-compiler) can add a keyword
without changing the grammar. The grammar is intentionally tiny —
`Definition` and `FunctionCall` carry every new keyword. The grammar is
also LL(1) (ADR 0010), which keeps the door open for a hand-written
parser in a future Silicon-in-Silicon bootstrap.

## Why not just use Zig / Rust / Roc?

The honest answers:

- **Zig.** If you want a tight, no-runtime systems language with `comptime`
  metaprogramming, Zig is excellent. Silicon's `@stratum` mechanism is
  closer to "macros as a phase" than to Zig's compile-time evaluation —
  the stratum handler runs at elaboration, not at the call site. If you
  already use Zig and like `comptime`, Silicon is not your replacement;
  it's a different point in the same design space.
- **Rust.** Silicon does not have a borrow checker. It has explicit arenas
  (`with_arena { … }`, `move_to_parent_arena value`) and `Rc<T>` for the
  shared-ownership case. If you need the static safety guarantees Rust
  provides, you need Rust. If you want a smaller mental model and you're
  fine with an arena discipline, Silicon is worth a look.
- **Roc.** Silicon and Roc share the HM-style inference flavour and the
  taste for sum types as a first-class feature. Roc has a much more
  mature effect system and a much more mature platform abstraction; if
  you're building a long-lived application, Roc is further along. Silicon
  is a smaller, lower-level language pointed at WebAssembly and native.

The thing that makes Silicon worth writing is the strata mechanism. If
that primitive doesn't excite you, the other three languages above are
all good answers to "I want a new systems-ish language."

## What 1.0 includes

- **Language.** `@struct`, parametric sum types with `@match` destructure,
  `Option[T]` / `Result[T,E]`, `@defer` / `@try`, HM-lite inference (no
  let-generalisation), unsigned int types (`u8`–`u64`), `Int64` for
  WASI surfaces, first-class functions + `call_indirect`, slices.
- **Memory.** Bump-allocated arenas with explicit `with_arena` scoping
  and tail-position parent-arena escape. Heap exhaustion is a clean
  trap; `--max-heap=N` caps it.
- **Backends.** WAT/WASM (primary), QBE → native for linux-x86_64,
  linux-aarch64, macos-aarch64, macos-x86_64. WasmGC opt-in via
  `--target=wasm-gc`. Self-host verified.
- **CLI.** `sgl init`, `sgl build`, `sgl run`, `sgl check`, `sgl fmt`.
  `--release` is an alias for `--native`.
- **CaaS.** A library-first compiler API. The `sgl` CLI is a thin
  consumer; you can call `parse / elaborate / typecheck / lower` from
  your own tooling. See `docs/compiler-as-a-service.md`.
- **Distribution.** GitHub Releases with SHA-256 checksums, Homebrew
  formula, apt/deb package, winget manifest (Windows via WSL), `curl |
  sh` installer.

## What 1.0 doesn't include

I want to be loud about this so you find out from me, not from a
frustrated GitHub issue:

- **No LSP.** The CaaS API is the foundation; a Sigil LSP is v1.1.
- **No package registry.** Local-path deps only at 1.0. `sgl add`,
  `sgl.lock`, and a git-backed index are designed and scheduled for
  v1.1.
- **No Silicon-native comptime interpreter.** Strata handlers run on the
  host process at compile time. v1.1 work pivots this to a Silicon
  interpreter (ADR 0003).
- **No incremental compilation.** Full re-parse + full typecheck on
  every change today. Architecture supports incremental; implementation
  is v1.1.
- **No code-action / quick-fix API.** Diagnostics ship as structured
  data; the `TextEdit`-producing layer is v1.1.

Full known-limitations list is in [`CHANGELOG.md`](../../CHANGELOG.md).

## What's next

v1.1 priorities, in order of how blocking they are for adoption:

1. **LSP server.** The CaaS API is the missing piece; the wrapper is
   straightforward.
2. **Package registry + lockfile.** Local-path only is a real friction
   point.
3. **Incremental typecheck.** The bench numbers in
   `docs/performance.md` show parser dominates at ~3K LOC; incremental
   work directly attacks that.
4. **Browser playground.** A tech demo, but a useful one for the docs
   site.

If you try Silicon and something is missing or broken, file an issue —
the worst-of-1.0 cases I most want to hear about are the ones I haven't
guessed.
