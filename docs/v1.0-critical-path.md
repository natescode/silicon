# Silicon v1.0 — Critical Path

> Status: planning artifact (living document). Last synced: 2026-06-05.
> Scope: the founding philosophy of Silicon, the one hard v1.0 gate (100% web/bun FFI
> coverage), the ADR chain that delivers it, the two adjacent pulls it leans on, the
> memory-mode decisions a v1.0 user actually makes, the honest risks, and a sequencing view.

This is the single page to read first. Everything below cross-links to the authoritative ADRs in
[`docs/adr/`](./adr/); where this doc and an ADR disagree, the ADR wins.

---

## 1. North star — Silicon is a multi-mode, gradually-memory-managed language

Silicon's founding bet is that a single source language should let a program choose **how memory is
managed**, rather than being locked into one global strategy for its whole life. That is not a Silicon
invention — it is the thesis of **Gradual Memory Management (GMM)**, articulated by **Jonathan Goodwin**
and prototyped in his language **Cone**. Silicon's "modes" are the deliberate realization of that thesis,
not a wart to apologize for.

### 1.1 The lineage, cited accurately

GMM is named by explicit analogy to Siek's *gradual typing*. Goodwin's framing: gradual memory management
"offers the programmer a choice over how memory is used **on an alias-by-alias basis**," promising a system
that is *not opinionated* (the language doesn't dictate the strategy), *not fussy* (friendly defaults, opt-in
constraints, no fighting the compiler to write simple programs), and *helpful* (it can "safely integrate
**multiple memory management strategies within the same program**").

The core architectural move is to decompose every alias (any reference) into **four orthogonal types**:

| Axis | What it governs | Goodwin's term | Silicon's analog |
|---|---|---|---|
| Value type | the data | value type | the type |
| Lifetime | the scope it's valid in | lifetime | scope hierarchy `local ⊂ arena ⊂ Rc` (ADR 0011) |
| **Strategy** | **how it's allocated/reclaimed** | **allocator type / region** | **target: arenas vs engine GC** (ADR 0008 / 0009) |
| Permission | aliasing & mutation constraints | permission (`uni`/`imm`/`mut`/`ro`/`opaq`) | capability borrowing `& / &mut / &uniq / &val` (ADR 0011) |

**Primary sources (load-bearing):**
- Jonathan Goodwin, *A Framework for Gradual Memory Management* (Sept 12, 2017) — https://jondgoodwin.com/pling/gmm.pdf
- Blog: *Gradual Memory Management* — https://pling.jondgoodwin.com/post/gradual-memory-management/
- Cone, *Memory Managed Your Way* — https://cone.jondgoodwin.com/memory.html ; permissions — https://cone.jondgoodwin.com/coneref/refperm.html
- Cone repo — https://github.com/jondgoodwin/cone

**Honesty about the source.** Goodwin himself flags the 2017 paper as a *working hypothesis* ("no language
specifications or detailed algorithms... no measurements or proofs"); Cone is a research compiler where, per
its docs, only mutability + coercion are currently enforced. So GMM is Silicon's **philosophical north star
and intellectual ancestor**, not a proven implementation blueprint. We cite it the way we ship features:
honestly scoped. (Terminology drift to be aware of: the 2017 paper's allocator-type names `local/lex/rc/tgc/pool/borrowed`
became Cone's region names `so/rc/gc/arena/pool/borrowed`, and its 7 permissions `mut/imm/unique/lock/const/mutx/id`
became Cone's 5 static permissions `uni/imm/mut/ro/opaq`. Same concepts, evolved naming.)

> **Lineage, now attributed.** [ADR 0008](./adr/0008-memory-management-arenas.md):30 calls Silicon's direction
> "**gradually memory-managed**: multiple allocator strategies selectable per scope, eventually Strata-defined."
> The Goodwin/Cone attribution for that phrase is now recorded in ADR 0008's Context (added 2026-06-05) and
> here; before that the repo cited neither.

### 1.2 What Silicon ships as its realization of GMM

Silicon decomposes the **strategy** axis into two whole-program memory **modes** that share one surface
language, plus an opt-in and a (proposed) borrow system:

- **Systems mode** — `--target=host` / `--target=wasix` (the native / wasm-mvp substrate): bump-allocated
  **arenas, no GC**, LIFO `@with_arena` / `@move_to_parent_arena`, deterministic latency. ([ADR 0008](./adr/0008-memory-management-arenas.md), Accepted.)
- **Managed mode** — `--target=wasm-gc`: managed reference types reclaimed by the **engine's GC** (V8 /
  SpiderMonkey / JSC). ([ADR 0009](./adr/0009-wasm-gc-target.md), Accepted.)
- **Rc opt-in** — `src/stdlib/rc.si`, single-threaded interior-immutable refcounting (Rust-`Rc`-shaped). On
  `wasm-gc` it is compile-time elided to identity (the engine owns the lifecycle).
- **Capability borrowing** — the proposed `local ⊂ arena ⊂ Rc` scope hierarchy + four reference capabilities
  (`& / &mut / &uniq / &val`) of [ADR 0011](./adr/0011-borrow-checker-and-rcaps.md) (Proposed, design exploration).

### 1.3 An honest positioning statement (do not overclaim)

Goodwin's strong definition of "gradual" requires *multiple strategies coexisting in one program at fine
(per-alias / per-scope) granularity, safely composed by the type system*. **Silicon v1.0 does not yet meet
that bar**, and we say so plainly:

| Honest claim for v1.0 | Why |
|---|---|
| "Two whole-program memory **modes** sharing one surface language, an `Rc` opt-in, and an allocator ABI designed for gradual MM" | True today. |
| "On a credible **path to** gradual MM" | The allocator ABI ([`compiler/wit/allocator.wit`](../compiler/wit/allocator.wit)) is literally Goodwin's "allocator type" made into an ABI; ADR 0011 capabilities + a future GC stratum are the path. |
| ~~"Gradual MM in Goodwin's sense, **today**"~~ | **Overclaim.** v1.0 is "switchable + opt-in" (the Nim-`--mm` + Rust-`Rc` quadrant), because only one of {arena, GC} is live per build — nothing coexists per-alias yet. |

Where Silicon sits in the landscape today: closest *true* peers are **Nim** (whole-program `--mm` switch) and
**Zig** (pluggable allocators, explicit no-GC arenas); closest *aspirational* peer is **Vale** (regions +
gen-refs, fine-grained, type-tracked) and Goodwin's own **Cone**. The piece that would move Silicon into the
gradual quadrant is letting ≥2 strategies coexist per-scope with type-enforced safe composition — i.e. ADR 0011
capabilities landing alongside a GC-as-a-stratum (ADR 0008 M-6 / v1.1).

**Design doctrine that falls out of this.** *Mode-tiering* — "grant each mode exactly the power its runtime
substrate honestly supports" — is the reusable Silicon pattern. Capability divergence between modes lives in
**what compiles**, never in **semantics**. ADR 0009's portability split (lifecycle elision vs `E0012`/`E0013`
rejection) and ADR 0019's closure tiers are both instances of it.

## 2. The one hard gate: 100% web/bun FFI coverage

**v1.0 has exactly one product gate that is non-negotiable: 100% of the web/bun API surface must be reachable
from Silicon `@extern` bindings.** This is a *hard* gate, not a soft target ([ADR 0018](./adr/0018-async-promise-ffi.md):6).

**Why it is the migration value-add.** Silicon's near-term users are people porting Node/Bun code. A language
that can call *most* of the platform forces every real program back into JavaScript at the first `Promise`,
DOM handle, or event callback — which means it can't actually replace the incumbent. 100% coverage is what
makes "rewrite this hot path / this whole service in Silicon" a real option instead of a toy. Anything less
than 100% leaves a permanent escape hatch back to JS and undermines the entire migration story. Hence: a gate.

The starting point is humbling and stated honestly: the binding generator's clean default output binds only
**~10%** of each surface ([ADR 0017](./adr/0017-ffi-binding-generator.md):99), because the boundary admits
only Tier-0 scalars + linear-`String` + `JSString` (externref) + `CharCodeArray`. The remaining ~90% is
"structurally unbindable" until the boundary machinery lands. The rest of this doc is the chain that closes
that 90%.

## 3. The critical-path chain: ADR 0017 → 0018 → 0019

Three ADRs (all currently **Proposed**) form the spine from 10% to 100%.

```
  ADR 0008 (Accepted)  arenas / AllocatorABI ──┐ (free-list still wanted for linear-mem
                                                │  data; NO LONGER gates closures)
  ADR 0009 (Accepted)  wasm-gc whole-program ──┤ (provides engine-GC'd externref/struct substrate)
                                                v
  ADR 0017 (Proposed)  bindgen, ~10% ceiling ──(ceiling lifted by)──▶
  ADR 0018 (Proposed)  boundary machinery: object handles, Asyncify @await (Bun-portable),
                        JSPI fast path ──(Phases 3-4 need closures)──▶
  ADR 0019 (Proposed)  mode-tiered closures  ==  ADR 0018 Phase 4  ──▶  100% on web AND bun
                          ▲
  ADR 0011 R4 (Proposed, low-priority, OFF critical path) ──(loosens 0019 classifier; unlocks &mut capture)
```

### 3.1 What each link delivers

**ADR 0017 — FFI binding generator.** Machine-generates Web/Node/Bun `@extern` bindings + the two host shims
(`cli/src/host/js-host.ts`, `playground/.../web-env.js`) from one Binding-IR, with a `bindgen.lock` and a CI
`check-shims` test. Headline is an *honest coverage ceiling*: ~10% today (Tier-0 only). **0017 is the generator;
its ceiling is lifted by 0018.**

**ADR 0018 — Async/Promise (+ object-handle + callback) FFI.** Lifts the ceiling by building the boundary
machinery for the three structural blockers: (A) no async/event loop, (B) no generic JS-object handle, (C) no
callbacks/closures. Key mechanism decision:

- **Asyncify-portable is the permanent baseline / floor.** Whole-program unwind/rewind, needs *no* engine async
  feature, needs *no* closures, runs everywhere now — including the primary Bun (JSC) host.
- **JSPI is a per-engine fast path** layered on behind a capability gate (~1µs stack switching). **As of June 2026,
  JSPI does not run on Bun** — Bun embeds JavaScriptCore, which hasn't shipped JS Promise Integration
  (tracked `oven-sh/bun#20878`). JSPI *is* standardized (Phase 4, Apr 2025) and ships on V8 (Chrome 137+ /
  Node 24+ / Deno 2.4+) and SpiderMonkey (Firefox 139+) — but it cannot be the baseline while Bun, a primary
  target, can't run it. Hence: Asyncify primary, JSPI detect-and-upgrade.

**ADR 0019 — First-class closures / capture (mode-tiered).** This *is* ADR 0018's Phase 4 — and per ADR 0018
its "largest single item." Mode-tiered decision:

1. **Non-escaping closures** — zero-cost, available on **every** mode. Comptime-monomorphized, unboxed env in
   frame/arena, lowered to a *direct* call. The empty-env case is byte-identical to today's `@fnref`.
2. **Escaping / host-callable closures** — **`--target=wasm-gc` only.** A managed fat record
   `(struct (field i32 fnIndex) (field (ref $Env)))` that crosses to the host as `externref` and is engine-GC'd
   (`__closure_release` is a no-op). On systems mode (`host`/`wasix`) this is **rejected at typecheck** with a
   new E-code ("host-callable escaping closure requires `--target=wasm-gc`", sibling to `E0012`/`E0013`/`E0014`).
3. **The Rc-on-linear-memory boxed escaping tier does not exist in any mode** — deleted, not documented. This
   *dissolves* the old "leaks until the free-list lands" caveat (see §4.2). The v1.0 escaping-closure invocation
   reuses the *existing* `call_indirect` + the shipped `JSString` externref path, so it needs no engine feature
   Bun lacks; `call_ref` is only a v1.x optimization.

### 3.2 The coverage ladder

| Phase (ADR 0018) | Delivers | Coverage | Notes |
|---|---|---|---|
| Phase 0 | Object handles (generic externref) | ~10% → ~35% | unblocks DOM/JS-object APIs |
| Phase 1 | Blocking `@await` via Asyncify-portable | ~35% → ~80% | **includes Bun** (no JSPI needed) |
| Phase 2 | JSPI fast path | (no new API coverage) | pure optimization; V8/SpiderMonkey only |
| Phase 3 | Closures → callbacks / events (**= ADR 0019**) | ~80% → ~97% | the largest single item |
| Phase 4 | Poll-reactor + tasks | ~97% → **100%** | the gate closes |

**All five phases are v1.0** ([ADR 0018](./adr/0018-async-promise-ffi.md):104). ADR 0019's internal order:
**C0** (generalize the `__fn_i_i` ABI) → **C1** (non-escaping, all modes) → **C2** (escaping, wasm-gc-only +
escape classifier — *the gate*) → **C3** (poll-reactor). C5 (intra-program escaping via regions on native)
is **post-v1.0**.

**Critical path to the gate:** 0018 P0/P1 (Bun-portable foundation) → 0019 C0/C1/C2/C3 (closures). Off the
critical path: JSPI (0018 P2, optimization) and ADR 0011 R4 (loosening). See §4.

## 4. The two adjacent v1.0 pulls

Two Accepted/Proposed ADRs sit next to the chain. Neither is *on* the critical path; both must be understood
to budget v1.0 honestly.

### 4.1 ADR 0011 — borrow checker + rcaps (the adjacent *loosener*)

Status: Proposed, "design exploration, no implementation timeline," explicitly low priority. The relevant
piece is **R4 escape promotion** (moving a value to a wider scope requires `&uniq` at the call site; escape
analysis picks the smallest scope that fits). ADR 0019 depends on R4 in two ways, **neither of which blocks the
gate**:

1. **Unlocks `&mut` capture.** v1.0 capture is by-value / `&`-immutable *only*. `&mut`/`&uniq` capture into an
   escaping closure is unsound without escape analysis (it could let a `&mut` outlive its frame), so v1.0
   **conservatively forbids it** until R4 exists.
2. **Loosens the escape classifier.** Because R4 is unbuilt, v1.0 ships a *sound conservative over-approximation*:
   any closure that syntactically reaches `@extern`/`@export_callback` is treated as escaping-to-host ⇒ requires
   `wasm-gc`. This **may reject some safe wasm-mvp programs**. R4 later only *relaxes* those false rejections
   (lets provably-non-retained `@extern` callbacks stay on systems mode).

**Takeaway:** v1.0 ships a sound-but-strict classifier now; ADR 0011 R4 is the later relaxation + `&mut`-capture
unlock. Keep 0011 off the critical path; document the dependency honestly.

### 4.2 ADR 0008 — arenas / free-list (the linear-memory reclaim pull)

Status: **Accepted** (2026-05-26; M-1 met, `wit/allocator.wit` landed). The allocator ABI exports `alloc` /
`realloc` / `enter-scope` / `exit-scope` / `promote` / `free-all` but **no individual `free`** — the bump
allocator leaks freed blocks (no free list); per the ADR this is "acceptable for 1.0; a real free-list
allocator is future work."

**The single most important sequencing correction this doc makes:** the free-list **no longer gates the FFI
100% gate.** Before ADR 0019's mode-tiering, the escaping closure env was Rc-on-linear-memory and *leaked* until
a free-list landed, so the gate appeared to wait on ADR 0008. After mode-tiering the escaping env is an
**engine-GC'd struct**, so:

> The free-list is **reclassified** from "blocks the FFI gate" to "still wanted for general long-running
> linear-memory data." It has **left the closure critical path.** ([ADR 0019](./adr/0019-first-class-closures-and-capture.md):80)

It remains wanted for ADR 0008's M-3…M-8 (general linear-memory reclaim, and the v1.1 mark-sweep-GC-as-a-stratum
at M-6) — but no v1.0 *FFI* milestone waits on it.

## 5. The memory-mode decisions a v1.0 user faces

The CLI exposes **two orthogonal axes**, each its own parser ([`cli/src/sigil_cli.ts`](../cli/src/sigil_cli.ts):749-763,
"Host platform — orthogonal to the wasm memory-model `target`"):

| Axis | Values | Selects | Set via |
|---|---|---|---|
| **`--target`** (memory model) | `host` (default) · `wasix` · `wasm-gc` | linear-memory bump allocator vs engine GC | `--target=` or `sgl.toml [build] target` |
| **`--platform`** (host) | `native` (default) · `web` · `bun` | native vs JS host (enables `JSString` externref + `wasm:js-string` builtins) | `--platform=` or `sgl.toml [build] platform` |

> Mapping note: ADR 0019's "mode" vocabulary is a projection of these axes. `--native` / `wasm-mvp` ≈
> `platform=native` on a linear-memory target (`host`/`wasix`); "wasm-gc" is `target=wasm-gc`. The
> escaping-closure gate keys on **`target=wasm-gc`**; the FFI host gate (`JSString`, js-string builtins) keys on
> **`platform ∈ {web, bun}`**.

Concretely, the decisions a v1.0 user makes:

1. **"I need to hand a callback to a host API."** ⇒ compile that module with **`--target=wasm-gc`**. Escaping,
   host-callable closures exist only there (engine-GC'd; leak-free).
2. **"I want deterministic latency / no GC for a hot loop."** ⇒ stay on systems mode (`host`/`wasix`, arenas).
   Escaping host callbacks are *rejected* there at typecheck (sound, see §4.1).
3. **The whole-program no-mixing constraint.** `--target=wasm-gc` is a **whole-program (per-module) decision**
   ([ADR 0009](./adr/0009-wasm-gc-target.md):73-77): "mixing managed refs and linear-memory pointers in one
   module has sharp edges (the GC scanner doesn't visit linear memory) that 1.0 doesn't try to solve." So
   **"arenas for my hot loop AND host callbacks in one module" is not expressible in v1.0.** This is an
   *accepted, coherent sharp edge*, not a defect — and ADR 0019 made it cheaper by collapsing the old
   two-story (Rc-on-mvp vs GC) lifetime model into a single engine-GC story behind the one flag ADR 0009
   already imposes.
4. **Async per engine: Asyncify vs JSPI.** `@await` works **everywhere now** via Asyncify-portable (including
   Bun). On V8/SpiderMonkey it transparently upgrades to the JSPI fast path. **Bun lacks JSPI but has wasm-gc**
   — so on Bun you get async (Asyncify) *and* escaping closures (wasm-gc) in v1.0; you just don't get JSPI's
   ~1µs path until JSC ships it.

This is the crux of the multi-mode design: escaping host-callable closures exist **only** where a JS host owns
a callback registry (`platform ∈ {web, bun}`) **and** that is exactly where `wasm-gc` lives. The two axes stay
formally orthogonal but correlate in practice — which is *why* the gate keys on `target=wasm-gc`.

## 6. Honest risks and candid magnitude

- **Closures are the largest single item on the path.** ADR 0018 hands Phase 4 to ADR 0019 and labels it
  "the largest single item." It is the highest-risk, highest-effort v1.0 work; the gate does not close without it.
- **The conservative escape classifier will reject some safe systems-mode programs.** Until ADR 0011 R4 lands,
  any closure syntactically reaching `@extern` is forced to `wasm-gc`. We ship sound-but-strict and accept the
  false rejections; users who hit them either move the module to `wasm-gc` or wait for R4.
- **The systems-mode capability line is real and permanent for v1.0.** Escaping host-callable closures are a
  `wasm-gc`-only capability; systems mode trades that away for no-GC determinism. This is mode-tiering working
  as designed, but it *is* a capability the systems user does not get.
- **Whole-program `wasm-gc` forecloses in-module mode mixing** (§5.3). Accepted v1.0 sharp edge; revisited only
  when the gradual-MM coexistence work (ADR 0011 + GC stratum) lands.
- **JSPI is engine-gated, not Silicon-gated.** Bun async stays on Asyncify until JSC ships JSPI (`bun#20878`);
  outside Silicon's control. The portable baseline ensures this is a perf gap, not a coverage gap.
- **GMM is aspirational, not shipped.** Per §1.3, v1.0 is "switchable + opt-in," not gradual-in-Goodwin's-sense.
  Stating otherwise unqualified would be an overclaim, contrary to the project's honest-scope habit
  (cf. ADR 0016 specifying a protocol before dispatching it).

## 7. Sequencing / milestone view

| Order | Work | ADR(s) | On critical path? | Gate effect |
|---|---|---|---|---|
| 0 | Arenas / AllocatorABI (done) | 0008 (Accepted) | foundation | — |
| 0 | wasm-gc whole-program substrate (done) | 0009 (Accepted) | foundation | — |
| 1 | Bindgen + `bindgen.lock` + `check-shims` | 0017 | yes | establishes ~10% baseline |
| 2 | Object handles (generic externref) | 0018 P0 | yes | ~10% → ~35% |
| 3 | Asyncify-portable `@await` (Bun-incl.) | 0018 P1 | yes | ~35% → ~80% |
| 3a | JSPI fast path | 0018 P2 | **no** (optimization) | no new coverage |
| 4 | Closures C0 (ABI) → C1 (non-escaping, all modes) | 0019 | yes | enables callbacks |
| 5 | Closures C2 (escaping wasm-gc-only + escape classifier) | 0019 | yes | **the gate**; ~80% → ~97% |
| 6 | Poll-reactor + tasks | 0018 P4 / 0019 C3 | yes | ~97% → **100%** |
| — | ADR 0011 R4 escape analysis | 0011 | **no** | loosens classifier; unlocks `&mut` capture |
| — | Free-list / individual `free` | 0008 M-3…M-8 | **no** (off closure path) | general linear-mem reclaim; v1.1 GC stratum |
| post-v1.0 | Intra-program escaping via native regions | 0019 C5 | post-v1.0 | — |
| post-v1.0 / v1.1 | GC-as-a-stratum (path to true gradual MM) | 0008 M-6 | post-v1.0 | enables strategy coexistence (§1.3) |

**The shortest honest description of v1.0's critical path:** *Bindgen (0017) → object handles + Asyncify
(0018 P0/P1, Bun-portable) → closures (0019 C0/C1/C2/C3) → poll-reactor → 100%.* JSPI, ADR 0011 R4, and the
ADR 0008 free-list are all deliberately **off** this path.
