# ADR 0018 — Unblock async/Promise (+ object-handle + callback) FFI: the boundary work to reach 100% web/bun API coverage (v1.0 requirement)

- **Status:** Proposed (design + phased plan; no code lands until Accepted)
- **Date:** 2026-06-05
- **Deciders:** NatesCode
- **Scope update (2026-06-05):** **100% web/bun API coverage is a HARD v1.0 gate** (product decision — it is the migration value-add for Node/Bun users), *not* a v1.0 "target" with closures-dependent phases pushed to v1.x. Consequently **all phases below, including Phase 3 (callbacks/events) and Phase 4 (concurrency), are v1.0-critical**, and the closures/capture work they depend on is pulled forward as a v1.0 keystone — specified in **ADR 0019**. The original §5 candor (don't *silently* couple the FFI deadline to the closures deadline) stands, but the coupling is now an *explicit, accepted* v1.0 commitment.
- **Related:** ADR 0017 (FFI binding generator — its ~10% coverage ceiling is the problem *this* ADR lifts; it emits "what is provably bindable today and records — never silently drops — what is not") · ADR 0016 (`@loop`/iterables — established the FFI-ceiling reasoning and, decisively, the **closures/coroutine deferral** at lines 52–56, 167–170, 302–309, 491–493) · ADR 0012 (effect lattice `pure ⊑ mut ⊑ effectful`, flat, `@extern` = `effectful` — where an `async` tag must attach) · ADR 0011 (rcaps + the *proposed, unbuilt* actor/isolation model — where concurrent tasks eventually land) · ADR 0009 (wasm-GC target — the world externref/funcref slots already live in) · [[silicon-js-string-builtins]] memory (the `JSString` externref machinery this ADR generalizes) · prior art (all web-verified): **JSPI** (WebAssembly/js-promise-integration, Phase 4 Apr 2025; `WebAssembly.Suspending`/`WebAssembly.promising`), **Binaryen Asyncify** (Emscripten `-sASYNCIFY`, whole-program unwind/rewind), **wasm-bindgen-futures** (poll reactor + `Closure` callbacks), the **wasm stack-switching / typed-continuations** proposal (the general primitive under JSPI, shipped nowhere), Go/wasm (`syscall/js.FuncOf` + Asyncify goroutine switch), Blazor (`TaskCompletionSource` + `[JSInvokable]`), AssemblyScript (no native async — confirms the closure floor).

## 1. Context — why today's surface structurally caps coverage at ~10%

Silicon's host FFI is **synchronous, one-shot, and scalar/`JSString`-only**. Three independent properties of the current pipeline combine to make most of every web/bun API *structurally unbindable* — not merely unbound. ADR 0017 already names this ("~10% coverage ceiling … structurally unbindable until ADR 0018 lands", `0017:23`); this ADR is the work that lifts it.

**Blocker A — no async / no event loop. The host runs `_start` and returns.**
`runUnderBun` (`cli/src/host/js-host.ts:100-109`) does `ex._start()`, then `flush(); return 0`. There is no microtask drain, no `await` on a wasm-produced thenable, no reactor. Every wired import is a synchronous function returning a scalar/string (`js-host.ts:41-75`; browser side `playground/playground/web-env.js:136-152`). The **only** existing case of the host driving wasm repeatedly is the `requestAnimationFrame` game loop `startGameLoop` (`web-env.js:171-206`) — and that drives a *host-owned, separately exported* `tick()`; it is a host-side scheduler bolted on, not wasm yielding to a reactor. Consequence: a `Promise` has nowhere to be awaited and nothing to resume into. Any API whose return is `Promise<…>` — `fetch`, `Response.json()`, `crypto.subtle`, `Bun.file().text()`, IndexedDB, the bulk of the modern platform — is unbindable.

**Blocker B — no generic JS-object handle. externref is name-gated to `JSString`.**
The `usesExternref` predicate is literally `moduleName === 'JSString' || siliconResult === 'JSString' || siliconParams.includes('JSString')` (`compiler/src/ir/lower.ts:1426-1427`), and a ref slot is stamped `{nullable:true, extern:true}` *only* when the type string is exactly `'JSString'` (`lower.ts:1468-1479`; template `EXTERN_SLOT` at `lower.ts:532`, injected by `injectJSStringRefSlots` at `lower.ts:457-460,528-539`). `@extern` itself forces scalars: `lowerExternParams`/`lowerExternResult` (`lower.ts:978-993`) run every type through `siliconTypeNameToWasm` (`lower.ts:2263-2270`), which collapses *every* unknown name to `i32`/`i64`/`f32`; `ExternDef_lower` (`compiler/src/strata/defkinds.si:62-76`) even hardcodes the import module string to `'env'`. So a `Promise`, a `Response`, a DOM node, a `Uint8Array`, a `Headers` — any JS object — has **no Silicon type to land in**. A Promise *is* a JS object: Blocker B is a prerequisite of Blocker A.

**Blocker C — no callbacks/events. funcref never crosses the boundary, and there is no capture.**
`@fnref` is purely internal: `lowerFnRef` returns `{kind:'Const', wasmType:'i32', value: idx}` (`lower.ts:1584-1612`) and `@call_indirect` consumes that `i32` to `call_indirect` *within wasm* (`lower.ts:1620-1637`). The signature is hardcoded `{key:'__fn_i_i', params:['i32'], result:'i32'}` (`lower.ts:1608-1609,1625-1626`) — and `@extern`'s scalar collapse means there is no `funcref`/`(ref func)` path at the boundary at all. So `addEventListener`, `setTimeout`, `.then`, `onmessage`, every callback/event surface, cannot be handed a wasm function. ADR 0016 records the deeper half of this as fact #4 (`0016:52-56`): `@fnref` "has no environment capture … A callback cannot close over a surrounding accumulator." **Silicon has no closures.** That single gap is what makes callbacks/events (and the poll-reactor async model) impossible today, and it is why 0016 declined internal iteration outright.

**Net.** Sync + scalar/`JSString` can express roughly the synchronous, value-returning slice of the platform (`console.*`, `performance.now`, `Math`, a `String` in/out). ADR 0017's audit puts that at ~54 hand-bound functions / ~10% of the surface. The remaining ~90% is async-shaped, object-shaped, or callback-shaped — i.e. exactly Blockers A/B/C. **Reaching 100% coverage is therefore not a binding-generator problem; it is a boundary-machinery problem,** and it is the gating dependency for ADR 0017's coverage and for the "100% web/bun API coverage" line item on the v1.0 list.

## 2. Decision

### 2.1 The honest reckoning with Bun/JSC

The mechanism choice hinges on one fact, and we state it plainly:

> **As of June 2026, JSPI does not run on Bun.** Bun embeds **JavaScriptCore**, and JSC has **not shipped** JS Promise Integration. `WebAssembly.Suspending` / `WebAssembly.promising` do not exist in Bun (tracked by `oven-sh/bun#20878`, which notes "WebKit/JSC currently does not support this"). JSPI *is* standardized (Phase 4, Apr 2025) and shipping on V8 (Chrome 137+, Node 24+ default-on, Deno 2.4+) and SpiderMonkey (Firefox 139+); WebKit withdrew its objection in late 2025 and JSPI is an **Interop-2026** focus area, so JSC will get it — but **not in a release we can target today.** Silicon's primary `sgl run` host (`js-host.ts:88`, `{builtins:['js-string']}`) is JSC.

This forces a conditional decision, exactly as the task framed it:

**Because JSC/Bun does NOT support JSPI today, the baseline mechanism is Asyncify-portable, with JSPI as a per-platform fast-path layered on top behind a capability gate.** Concretely:

- **Baseline (ships first, permanent floor): Asyncify-portable.** A whole-program-style unwind/rewind transform that makes a synchronous-looking guest suspendable on **any** engine — V8, **JSC/Bun**, SpiderMonkey, and even native/wasmtime — with **no host async feature** required. It is the only "synchronous-blocking illusion" that runs on Silicon's primary host *now*. It needs **none** of Silicon's missing closures: it serializes live locals into a dedicated "asyncify-stack" region of linear memory and skip-aheads back, so it works on the current value model unchanged.
- **Fast path (layered, capability-gated): JSPI.** On engines that have it (web/V8, Node 24+, Deno; JSC once Interop-2026 lands), the backend emits **no transform at all** — a plain synchronous `call` to a suspending import plus a "promising" entry export — and lets the engine suspend the native wasm stack (~1µs/switch, zero size tax). The loader feature-detects `WebAssembly.promising` and wraps imports/exports; if absent, it falls back to the Asyncify lowering. **Same Silicon source, no user changes** — only the codegen flag and the host glue flip.

This ordering is deliberately the inverse of "JSPI-primary": if we made JSPI primary we would strand the Bun host, which is unacceptable for a v1.0 "web **and** bun" coverage requirement. Asyncify is primary *because* it is the one that runs everywhere today; JSPI is the optimization we turn on per engine.

**Why not the poll-reactor (wasm-bindgen-style) model as baseline?** It is the most capable family — it is the only one that also solves **callbacks/events**, not just Promises — but it is **blocked on closures** (Blocker C): the continuation must close over the awaiting frame's locals, and `@fnref` is fixed `i32→i32` with no environment (`0016:52-56`, `lower.ts:1608-1609`). It cannot ship until closures exist. It is the strategic *destination* (Phase 4 below), not the quick win. Asyncify gets us blocking-`@await` coverage *before* closures land; the reactor gets us events *after*.

### 2.2 Language surface — `@await` / `@async`, identical across both backends

The surface is designed **once** and is backend-agnostic; that is the whole point of choosing two mechanisms with identical synchronous-blocking semantics.

- **`@async`** annotates a function. It (a) marks it as permitting suspension points, (b) if it is an entry, marks its export as the "promising" one the host wraps, and (c) **colors** it in a new `async` effect on ADR 0012's lattice. Registered as a keyword stratum, sibling to those in `compiler/src/strata/control.si` — **no grammar change** (per the compiler's "add a stratum, not a grammar rule" discipline).
- **`@await expr`** is a new expression stratum (registered like `@break`/`@return`/`@fnref` in `control.si`). `expr` must be a call to a *suspending import*. Type rule: `@await f()` where `f : (…) → Future[T]`/suspending yields `T`; using `@await` outside an `@async` body is a typecheck error (an E-series diagnostic, sibling to E0014). This is classic function coloring (`@await` only inside `@async`, the color propagates up the call graph) — and the colored set is *exactly* the set Asyncify instruments.
- **Suspending imports** carry a `@suspending` marker so codegen emits a vanilla import *and* records boundary metadata for the host shim: `@suspending @extern http::get url:JSValue -> JSValue;`. The awaited result type is the new generic object-handle `JSValue` (Blocker B), so a Promise has a type to land in.

User code reads as straight-line synchronous code regardless of backend:

```silicon
@async @fn load_name id:Int -> JSString := {
    @local resp := @await &http::get(&url_for id);   # suspends; looks sync
    @local body := @await &http::text(resp);          # suspends again
    body
};
```

The emitted import is vanilla either way — `(import "http" "get" (func $http__get (param externref) (result externref)))` — plus a sidecar note `{ import: "http.get", suspending: true }` the loader consumes. Under JSPI the loader wraps that import with `WebAssembly.Suspending`; under Asyncify the import is an ordinary Promise-returning function and the *transform* (not the engine) does the unwind/rewind. **No `@await`-specific syntax differs between backends.**

(An effect-typed framing is equivalent and we adopt it for the typechecker: an `@async` function is `async`-effectful; `@await` is the elimination form. See §3.5.)

## 3. Prerequisite work and order

Five pieces of machinery are missing. They are listed in dependency order; the **bold tier** is what each mechanism actually needs.

**P1 — Generic object-handle externref (`JSValue`). [P0; needed by both Asyncify and JSPI]**
Generalize the `JSString`-only externref path. Add a `JSValue`/`Extern[T]` type recognized by `siliconTypeNameToWasm` (`lower.ts:2263-2270`, currently collapses unknown names to `i32`); generalize the `usesExternref` predicate (`lower.ts:1426-1427`) and the slot stamping (`lower.ts:1468-1479`) off the literal `'JSString'` check; add `injectExternRefSlots` modeled on `injectJSStringRefSlots` (`lower.ts:457-460,528-539`). Let `@extern` carry an externref result: `lowerExternParams`/`lowerExternResult` (`lower.ts:978-993`) need an externref branch, and `ExternDef_lower` (`defkinds.si:62-76`) must accept an import-module override (today hardcodes `'env'`). The `JSString` slot machinery is the working template — this is the *smallest, highest-leverage* item: it alone moves ADR 0017's generator from "scalar/string tier" to "object tier."

**P2 — `@suspending` marker + boundary-metadata emission + reactor host model. [P0; needed by both]**
(a) Accept `@suspending` on extern/module fns, emit a vanilla import, and record `{import, suspending:true}` / `{export, promising:true}` into the module sidecar. (b) **Change the host from one-shot to a reactor.** `runUnderBun` (`js-host.ts:100-109`) must call the entry (the promising/Asyncify-rewind export), keep the process alive, drain the microtask queue / re-enter on Promise settle, and wire Promise-returning imports (`fetch`, `setTimeout`). Generalize `startGameLoop` (`web-env.js:171-206`) into a real reactor rather than a bespoke rAF hack. This is host glue, not user-visible — but it is non-trivial and shared by every backend.

**P3 — The Asyncify unwind/rewind transform + async-coloring. [P0 for the Bun/baseline path; NOT needed by JSPI]**
Two routes: **(A)** run Binaryen's Asyncify pass on the emitted binary — `binaryen` v117 is *already* a `compiler/package.json` dependency, so this is the fast bootstrap; or **(B)** an IR-level pass over `IRModule` (`compiler/src/ir/nodes.ts`), sibling to `loopDesugar`/`lowerTry`, that async-colors only functions transitively reachable from an `@await` and instruments *precisely* those — Silicon's IR knows the exact call graph, so it avoids Binaryen's conservative whole-module blowup. Ship A first (correctness, portability), graduate to B for size/perf. Carve the asyncify-stack region out of the bump allocator (**mind the `heap_align` alignment gotcha** noted in memory). v1 is **single-in-flight unwind** (one suspended computation at a time, like `setjmp`'s `jmp_buf`).

**P4 — Closures / capture + funcref-at-boundary. [v1.0-critical for the 100% gate; not needed for blocking `@await`, but required for Phases 3–4 → specified in ADR 0019]**
This is the ADR-0016 gap and the largest single item — and, because 100% is now a hard v1.0 gate, it is **on the v1.0 critical path, not deferred**. ADR 0019 designs it. A real closure is a fat pointer `{funcref, env}` with a captured-environment struct; the `__fn_i_i` hardcoded signature (`lower.ts:1608-1609,1625-1626`) must generalize to arbitrary signatures, and `@fnref` must be able to produce a real `(ref func)`/`funcref` value that crosses an import/export (not the internal `i32` table index it returns today). `@extern` must accept `funcref` params (the same `lowerExternParams`/`ExternDef_lower` surfaces as P1). **Until P4 lands, callbacks/events are impossible and the poll-reactor async model cannot ship** — which is precisely why Asyncify (P3), not the reactor, is the baseline. A new stratum + IR node alongside `control.si`. ADR 0016 (lines 302-309, 491-493) is candid that this is "stack manipulation Silicon lacks"; we are committing to build it, but staging it after blocking-`@await`.

**P5 — `async` effect on ADR 0012's lattice. [P1; design, lands with P2/P3]**
ADR 0012's lattice is flat (`pure ⊑ mut ⊑ effectful`, `@extern` = `effectful`, `0012:25-30`). Add an `async` tag so (a) the `@await`-only-inside-`@async` coloring rule is a principled typecheck rather than a codegen accident, and (b) the optimizer treats a suspension as a barrier. Keep it a flat tag (no effect rows) per 0012's "stay within HM-lite's budget" discipline. Reconcile with ADR 0011's unbuilt actor/isolation model only when true concurrency (P4+, multiple in-flight tasks) arrives.

**Order:** P1 → P2 → P3 (blocking `@await` works, portable, on Bun **today**) → P5 (type-honest coloring) → P4 (closures: unlocks callbacks/events + the poll-reactor + JSPI's nicer rejection-to-`@try` ergonomics). JSPI fast-path slots in opportunistically once P1/P2 exist (it skips P3/P4 entirely).

## 4. Phased roadmap to 100% coverage

| Phase | Lands | Prereqs | Unblocks (coverage) |
|---|---|---|---|
| **0 — Object handles** | `JSValue` generic externref; `@extern` carries externref; import-module override | **P1** | Synchronous object-returning APIs: `Response` fields, `URL`, `Headers`, `Uint8Array` handles, DOM nodes as opaque values. Lifts ADR 0017's generator from scalar/string → **object tier**. ~10% → ~35%. |
| **1 — Blocking `@await` (Asyncify), portable incl. Bun** | `@async`/`@await`/`@suspending` surface; Asyncify transform (Binaryen route); reactor host; `async` effect | **P1·P2·P3·P5** | **All Promise-returning APIs** with a synchronous-looking surface, on **every** engine today (V8, **JSC/Bun**, SpiderMonkey, native): `fetch`, `.json()`/`.text()`, `crypto.subtle`, timers, `Bun.file`, dynamic `import()`. ~35% → ~80%. |
| **2 — JSPI fast path** | Loader feature-detects `WebAssembly.promising`; emits plain `call` + promising export on V8/Node/Deno (and JSC when it ships) | **P1·P2** (skips P3) | No new *API* coverage — same `@await` surface — but **near-zero size/perf** on V8 vs Asyncify's tax, and the long-term landing spot for Bun once JSC ships JSPI. A pure optimization gate. |
| **3 — Closures + funcref-at-boundary** | `{funcref, env}` closures; real funcref across the boundary; `@on`/`addEventListener`/`setTimeout(cb)` binders | **P4** | **Callbacks + events** — the entire listener surface neither Asyncify nor JSPI can reach: `addEventListener`, `onmessage`, `EventTarget`, observers, streaming `.on('data')`. **All five phases are v1.0** (100% is a hard gate, see §5). Phases 0–1 (→80%, every target incl. Bun) are the *foundation*; **Phase 3 (→97%) and Phase 4 (→100%) are equally v1.0-required** and depend on closures (ADR 0019). Phase 2 (JSPI) remains a pure optimization. Each phase is independently shippable behind its capability/feature gate, but none below 100% satisfies the gate.

## 5. Risks, and how "v1.0 requirement" interacts with scope

- **Engine support / Bun-JSC drift (highest external risk).** The fast path (JSPI) depends on a feature JSC lacks today. **Mitigated by construction:** baseline is Asyncify, which needs *no* engine feature, so async works on Bun in Phase 1 regardless. JSPI is a detect-and-upgrade, never a hard dependency. If JSC ships JSPI on the Interop-2026 timeline, Bun upgrades for free; if it slips, nothing breaks. We must **target only the new `Suspending`/`promising` API** (the old `WebAssembly.Suspender` was removed in V8 12.8) and not regress on that.
- **Asyncify's real size/perf tax.** ~+50% typical, up to ~2x on macrobenchmarks, pathological ~5x on one huge function; *requires* running the Binaryen optimizer or size explodes. **Mitigation:** precise IR coloring (P3 route B) instruments only `@async`-colored functions, recovering most of it; and the JSPI fast path (Phase 2) erases the tax entirely on V8/Node where most production traffic is. Honest magnitude: Phase-1 Bun binaries will be meaningfully larger until route B + Phase 2 land.
- **Async coloring (the red/blue function-color burden).** `@await` only inside `@async`, propagating up the call graph — the familiar ergonomic cost. Cheap here (nothing is *generated* behind the color under JSPI; under Asyncify the color just selects the instrumented set), but it is a real surface constraint and a new typecheck rule. We accept it as the standard cost of structured async.
- **GC / leaks of externref handles.** `JSValue` handles cross the boundary; a Promise/Response/DOM node held in linear memory or a handle table can leak if not released. ADR 0017 already flags the object-handle-table tier as "a leak footgun" (`0017:31`). **Mitigation:** ride ADR 0009's wasm-GC externref slots (engine-GC'd) rather than a hand-rolled refcounted handle table where possible; tie handle lifetime to rcap scopes (ADR 0011) when P4's closures introduce captured handles; document handle ownership at the boundary. This needs explicit spec work, not hand-waving.
- **Complexity vs Silicon's "small + bootstrappable" identity (the deepest tension).** This is the most invasive boundary work the language has taken on. We protect the identity three ways: (1) **the surface is three strata** (`@async`/`@await`/`@suspending`) riding existing `Definition`/`FunctionCall` forms — **no grammar change**, consistent with the compiler's strata discipline and ADR 0010's LL(1) target; (2) **the emitted wasm stays vanilla** under both backends (JSPI declares nothing special; Asyncify is a post-pass producing plain MVP wasm) — the *core language* and its bootstrappable subset are untouched, the complexity is quarantined in a codegen pass + host glue; (3) **closures (P4) are deferred to Phase 3**, so the v1.0 path (Phases 0–1) ships *without* the single hardest feature, on the cheapest mechanism that works everywhere.
- **Interaction with "v1.0 requirement" (decided: HARD gate).** 100% web/bun coverage is a **hard v1.0 gate**, not a target — it is the headline migration value-add for Node/Bun users, who will not adopt a runtime that can't `await fetch()` or register an event listener. So **all five phases are v1.0**, and the dependency chain is now explicit and accepted: the gate ⇒ Phases 3–4 ⇒ **P4 closures** ⇒ **ADR 0019** is on the v1.0 critical path. The candor ADR 0016 established is preserved, but inverted in conclusion: we are **not** decoupling the FFI deadline from the closures deadline — we are **deliberately coupling them**, because shipping 80% (Promises + object handles, no events/concurrency) would miss the migration story. The practical consequence is that **closures (ADR 0019) become a gating v1.0 dependency** with the largest single cost in this whole programme; v1.0 planning must budget it as such, and the build order (Phases 0–1 first as the foundation, then closures unlocking 3–4) is about *sequencing within v1.0*, not about what's in or out.

## 6. Options considered

Three mechanism families; all map onto these (everything else is a variant). Decision recap: **Asyncify baseline + JSPI fast path; poll-reactor deferred to post-closures.**

### Option A — JSPI-primary (suspending imports + promising export). **Rejected as *primary*; adopted as the layered fast path.**
The engine suspends the live wasm stack; the `.wasm` is vanilla (no new opcodes/types/sections); `@await` compiles to a plain synchronous `call` with **zero** guest-side runtime — no closures, no Future/Waker, no scheduler, no transform. It is the lowest-effort, lowest-complexity async path that exists and needs *none* of Silicon's three biggest gaps. **Why not primary:** it is **dead on Bun/JSC today** (`bun#20878`) and on Safari, so a JSPI-only story does not run on Silicon's primary `sgl run` host — disqualifying for a "web *and* bun" v1.0 requirement. It cannot be the *only* async story; it must be paired with a portable fallback regardless. **Adopted role:** Phase 2 per-engine optimization, capability-gated by `WebAssembly.promising` feature detection, sharing the identical `@await` surface so flipping it on costs zero source changes. When JSC ships JSPI, Bun graduates onto it for free.

### Option B — Asyncify-portable (whole-program unwind/rewind transform). **Accepted as the baseline / permanent floor.**
A CPS-ish save-locals-to-linear-memory + skip-ahead-rewind transform that makes a sync-looking guest suspendable on **any** engine with no host async feature. It is the **only** synchronous-blocking illusion that runs on **Bun/JSC today**, and uniquely also has a **native/wasmtime** story (the state machine is ordinary wasm a non-JS host can drive). Critically it needs **none** of Silicon's missing closures — it serializes live locals, working on the current value model unchanged — which is the decisive reason it beats the reactor model that Silicon's no-closures constraint blocks. **Accepted because:** it is the one mechanism that delivers blocking-`@await` on every target *now*, it is squarely in scope for a language that emits its own wasm from its IR (the transform is a natural sibling to `loopDesugar`/`lowerTry`, and `binaryen` v117 is already a dep), and it shares the exact `@await` surface with JSPI so the language is designed once. **Accepted costs:** the size/perf tax (§5), single-in-flight-unwind in v1, and re-execution of functions on the rewind path (a side-effect hazard the codegen must respect).

### Option C — Poll-based Futures + callbacks (wasm-bindgen-futures-style reactor). **Deferred to post-closures (Phase 3–4).**
Async modeled as *data*: the guest builds a `Future[T]`, registers a `{funcref, env}` continuation closure as the Promise's wake callback, and returns; the host calls the closure back on a later tick to `poll()`. **Highest ceiling — and the only family that also solves callbacks/events** (a Promise wake, `addEventListener`, `setTimeout`, `onmessage`, and the rAF `tick` are *the identical mechanism*: register a wasm closure with the host), with **zero codegen size tax** and **true concurrency** (many in-flight awaits). **Why deferred, not rejected:** it is **blocked on the one feature Silicon most conspicuously lacks — closures/capture** (Blocker C; ADR 0016 fact #4, `0016:52-56`). The continuation *must* close over the awaiting frame's locals; `@fnref` is fixed `i32→i32` with no environment (`lower.ts:1608-1609`). It also needs funcref-at-boundary, generic externref, a handle/id table, a `Future`/Waker runtime, a reactor host, and a CPS state-machine split of every `@async` function (the heaviest codegen of any option). AssemblyScript — the closest peer — concluded it *cannot* do async without exactly this machinery, which confirms the floor. **Disposition:** it is the **strategic destination** (Phase 4), reached *after* closures (P4) land — at which point it is mostly stdlib + a small reactor, and it is the only path to the final ~97%→100% (events + concurrency). The wasm stack-switching / typed-continuations proposal (the general primitive under JSPI) is the same shape at the engine layer and is **shipped nowhere** (V8/JSC/SM/wasmtime all lack it as of early 2026), so it is not a portability option and is out of scope.

**Why this combination.** No single mechanism is both portable-today and zero-complexity. Asyncify gives portability-today (incl. Bun) at the cost of a size tax; JSPI gives zero-cost at the cost of portability; the reactor gives the highest ceiling at the cost of closures. We take **Asyncify as the floor that guarantees async on every target now, JSPI as the per-engine fast path that erases the tax where available, and the reactor as the post-closures destination that finishes coverage (events + concurrency) to 100%** — all behind **one** `@await`/`@async` surface, so the language is designed once and the backend is a capability decision, not a source-level one.
