# ADR 0019 — First-class closures / capture: the keystone that unblocks FFI callbacks/events and the poll-reactor for 100% web/bun coverage

- **Status:** Proposed (design + phased plan; no code lands until Accepted)
- **Date:** 2026-06-05
- **Deciders:** NatesCode
- **Scope note:** Closures are **on the v1.0 critical path, not deferred.** 100% web/bun API coverage is a HARD v1.0 gate (product decision — it is the migration value-add for Node/Bun users). ## 1. Context — closures are the named gap, and now the gate

A closure is a *code pointer + the values it captured*. Silicon has the code pointer and nothing else.

**What exists.** `@fnref name` (`lower.ts:1584`) accepts only a bare top-level `@fn` name, interns it into a per-module funcref table, and returns `{kind:'Const', wasmType:'i32', value: idx}` — a table index, an empty-env code pointer. `@call_indirect cb, arg` (`lower.ts:1620`) is hardwired to **one** signature `{key:'__fn_i_i', params:['i32'], result:'i32'}` (`lower.ts:1608-1609, 1625-1626`), exactly one i32 argument, i32 result, and **no environment operand**. So the runtime ABI is `i32(table-index) → i32 → i32` with no env. There is no way to write a callback that closes over a surrounding binding, and `@extern` collapses every type to a scalar (`lowerExternParams`/`lowerExternResult`, `lower.ts:978-993`), so no `funcref`/`(ref func)` value crosses the boundary either.

**Where the gap is already named.**
- **ADR 0016, fact #4 (`0016:52-56`):** "`@fnref` has no environment capture. It is fixed i32→i32 … A callback cannot close over a surrounding accumulator — which is the real reason internal iteration is declined (Option C), and why combinators wait on closures." ADR 0016's deferred list (`0016:489-491`) records `map`/`filter`/`fold` and the *dispatched* `IterStep[T,R]` for-each surface as blocked on "closures exist."
- **ADR 0018, Blocker C and P4 (`0018:21-24, 84-85`):** callbacks/events are impossible because funcref never crosses the boundary and there is no capture. P4 explicitly hands the design to this ADR and marks it "v1.0-critical for the 100% gate … the largest single item." Its **Phase 3** (callbacks + events: `addEventListener`, `setTimeout(cb)`, `onmessage`, `EventTarget`, streaming `.on('data')` — ~80%→97%) and **Phase 4** (poll-reactor + tasks: a `Future[T]` whose wake continuation is a closure over the awaiting frame — ~97%→100%) **both depend on closures**, and both are now hard-gate v1.0.

**The motivating case, stated plainly.** The hard case is not the local closure that the optimizer could inline. It is the **escaping FFI-callback closure**: the guest builds a `{code, env}` value and hands it to a host registry (a JS `addEventListener`, a Bun event emitter, a Promise `.then`), which stores it and calls it back at an **unbounded later time**. The env must outlive the lexical scope that built it — and the host outlives every arena. This is the case that decides the representation and the lifetime model, and it is the case ADR 0008's arenas cannot express (an escaping callback has no dominating scope to reset against; the only arena answer is `&move_to_parent_arena`-to-root, i.e. a deliberate leak — ADR 0008 flags return-value escape as unsolved).

**The tension this ADR must reckon with.** Zig deliberately ships **no closures** for exactly Silicon's reasons: no hidden allocation, no hidden control flow, small and bootstrappable. A real closure forces captured values *somewhere* that outlives the frame (the heap), and a no-implicit-allocator language has nowhere to implicitly put them. Silicon's `@fnref` + `@call_indirect` is already ~80% of Zig's "pass a function pointer + an explicit context" pattern. The honest question is whether to stop there (Zig's stance) or build real closures — and the **hard 100% gate forces the answer**: host-callable *captured* callbacks are required, and a bare code pointer with a hand-built context is too lossy and too unsafe for the migration story (untyped `*anyopaque`, manual lifetime, no compiler-checked ctx⇔fn pairing). We build closures, but we build the **smallest representation that satisfies the gate first**, and grow the optimization later.

## 2. Decision

**Ship a single closure surface type `Fn[A -> R]` (HM-lite's existing `Function` kind) with a uniform boxed representation as the v1.0 floor, and grow an unboxed/monomorphized fast path as a later optimization.** Concretely:

1. **Representation (v1.0 floor): one boxed "fat pointer."** A closure value is a heap record `[fnIndex:i32, envPtr:i32]` (8 bytes via the prelude `&alloc`, the exact shape `rc.si` already allocates). `fnIndex` is a slot in the **existing** funcref table (`lowerFnRef`); `envPtr` points at an `rc.si`-headered env block `[refcount:i32 | captured i32s…]` holding the captured bindings in declaration order. The closure is invoked by loading both words and doing a `call_indirect` against a derived signature whose **first param is `env`**. One representation, storable, heterogeneous (a `Vec` of callbacks is a `Vec` of i32 pointers), boundary-crossable. This is the wasm-bindgen `WasmSlice{a=env, b=fn-index}` shape, minus the mandatory heap-`Box`.
2. **Optimization (v1.x): unboxed/monomorphized fast path.** When the consuming call site statically knows the callee and the closure does not escape, specialize the consumer per concrete env type (the **same comptime machinery as generics**, ADR 0001), call the lifted top-level `@fn` directly, and keep the env in the frame/arena — no heap, no indirect call, fully inlinable (Rust `impl Fn` / C++ raw lambda). The non-capturing degenerate case is *already* `@fnref` (empty env ⇒ bare table index, zero new cost).
3. **Decline defunctionalization (whole-program apply-switch) as the general mechanism** — it breaks separate compilation / the CaaS `MetadataReference` story (ADR 0012). It survives only implicitly as the empty-env `@fnref` special case.
4. **Decline Zig's no-closure stance.** The rcap system (ADR 0011) supplies the lifetime proof Zig forces onto the programmer, so Silicon can offer *real, typed* closures without the `*anyopaque ctx` tax.

**Why boxed-first, not unboxed-first (the heart of the decision).** The unboxed/monomorphized form is faster and is the "dream case" for a no-GC language — but it is *exactly the representation that cannot escape to a host*. An unboxed closure has no stored code pointer and no uniform shape; you cannot put it in a host registry, store it in a `Vec`, or hand it across the `@extern` boundary. The v1.0 gate is **precisely the escaping/heterogeneous case**. So the representation we are *required* to ship for the gate is the boxed one; the unboxed one is the optimization we are *allowed* to defer. Building boxed-first means Phase 3/4 (the gate) lands on a representation that actually works for them, and the unboxed fast path is a non-blocking speed win layered on later behind escape analysis — the standard Rust hybrid (`impl Fn` by default, `Box<dyn Fn>` on erasure), but **inverted in build order** because Silicon's binding pressure is the erased case.

This is the candid magnitude: closures are the **largest single item** in the v1.0 programme (ADR 0018 says so). We are committing to capture analysis, a generalized multi-signature funcref ABI, an env-record allocation, an exported invoke trampoline, Rc-as-keep-alive, and a leak-prevention diagnostic — and, later, an escape/erasure trigger plus the monomorphized fast path. The staging below keeps each step shippable and keeps the bootstrappable core untouched.

## 3. Capture semantics and the ADR-0011 rcap interaction

**v1.0 capture is by-value / `&`-immutable only. Mutable (`&mut`/`&uniq`) capture into an escaping closure is unsound and is rejected.** This is forced by ADR 0011's invariants, not a stylistic choice:

- **By-value / `&val` / `&` (immutable shared):** copy the captured i32 into the env. For an `Int`/`Bool` this copies the value; for a heap value (`String`, `Sum` pointer, `Vec`) it copies the *pointer*, which is sound because the pointee is independently lived (arena/Rc), not the stack frame. No aliasing is introduced, nothing outlives its scope — sound for both escaping and non-escaping closures, with no borrow checker required. This is the **only** mode an escaping closure may use in v1.0.
- **`&mut` / `&uniq` (move-in, uniqueness-preserving):** allowed **only for a provably non-escaping** (`local`-scoped, not stored/returned) closure, and **v1.0 conservatively forbids it entirely** until ADR 0011's escape analysis (R4) is implemented. The reason is exact: an escaping closure that captured `&mut x` where `x:@local` would let that `&mut` **outlive `x`'s frame** (ADR 0011 **R1**: a reference may not outlive its scope) **and alias** the caller's other live refs to `x` (ADR 0011 **R2**: at any point 0..n `&` XOR 0..1 `&mut`). Both invariants break. Rejected with a borrow-style E-code.

This recovers Rust's **Fn vs FnMut** split from the **rcap lattice instead of a trait hierarchy**: a by-value/`&`-immutable closure is "Fn"-shaped (re-callable, immutable env, freely shareable, escapable); a `&mut`-capturing closure is "FnMut"-shaped (mutable env, must not escape) and waits on the borrow checker. Mutating captured state in v1.0 is the explicit Swift/Go escape hatch: capture an `Rc[Cell]`/heap-cell **pointer** by value and mutate through it — the pointer is copied, the cell is shared, and the aliasing is visible in the source.

**Capture set.** The free variables the body reads that are bound in an enclosing scope; the env materializes exactly those, in declaration order. Because the body is a **named, lambda-lifted top-level `@fn`** (§6), capture is the only genuinely new analysis — call and dispatch reuse the `@fnref`/`@call_indirect` machinery.

**Effect-lattice interaction (ADR 0012).** A closure inherits its body's effect class. An env holding no `&mut` and a body holding no ocap ⇒ the closure is `pure`, so (once the unboxed fast path exists) its direct call is CSE/LICM-eligible; an env that is `local`/`&uniq` ⇒ guard-free static in-place reuse (ADR 0012 §4), beating Perceus's runtime `rc==1` branch. Conversely a **host-callable** closure re-enters arbitrary host effects on invocation, so it sits **high on the lattice** (re-entrant/impure) and the optimizer must **not** treat the indirect call as pure even though it looks like a plain `CallIndirect`.

## 4. Env lifetime — Rc for escaping, arena for non-escaping, no silent leak

Three tiers, chosen by the env's allocation site per ADR 0012's "allocation site from scope" rule.

**Tier 1 — non-escaping (the common case): frame / arena, no Rc.** A closure that does not outlive its defining scope keeps its env in the frame (unboxed fast path) or in the active arena (boxed). It is freed in bulk at `&with_arena` exit (ADR 0008) like every other heap value — no refcount traffic, no GC. A **synchronous host callback that the host does not retain** (the wasm-bindgen `ScopedClosure` analog — a sort comparator, a synchronous `forEach`) lives here too: capture arena-scoped env, auto-drop at `&with_arena` exit. This is the 80% case and it is fully sound today.

**Tier 2 — escaping within the program (returned/stored, never crosses to the host): arena, with `&move_to_parent_arena` to promote one frame up** (ADR 0008) when the closure outlives its defining block but not the program. Still bulk-freed at the enclosing arena reset; still no GC.

**Tier 3 — escaping to the host (the FFI-callback hard case): Rc, with registration as ownership transfer.** Arenas fail here by construction (no dominating scope). The model is the wasm-bindgen handle made first-class — **never** a silent `.forget()`:
- The env is `rc.si`-headered (`[refcount | captured…]`).
- **Registration is the ownership transfer.** `@on(host_slot, ctx, handler)` does `&rc_clone env` so the host now owns a strong ref via a first-class, **named, droppable handle/token** (the "store the handle" pattern). `@off(token)` does `&rc_drop env`. To make the host's drop actually reclaim bytes, generate a `__closure_release(clo:i32)` **export** whose body is `&rc_drop` on the env; the host calls it from its unregister path.
- **The honest gap, identical to `rc.si` today:** `rc_drop` decrements but does **not** physically free — Silicon's v1.0 bump allocator has no free list (`rc.si` header: "the bytes leak until the enclosing allocator scope resets"; ADR 0008 `prelude-ir.ts:211-212` "permanently lost — acceptable for 1.0"). So an escaping-then-released callback **leaks until ADR 0008's `AllocatorABI.free`/free-list lands**. v1.0 ships this as a *documented, honest* leak with the same status as `rc.si` — the gate is satisfied (the callback *works*, repeatedly, for the program's lifetime), and reclaim is the free-list follow-up.
- **Cycles** (host ↔ closure) inherit the Swift/Rc retain-cycle weakness; mitigate with `weak` registry handles later.

**On `--target=wasm-gc` (ADR 0009), Tier 3 collapses to engine GC.** The closure is `(struct (field i32 fnIndex) (field (ref $env)))`, the host holds it as an `externref`, V8/SpiderMonkey traces and collects it — cycles handled, `__closure_release` is a no-op. **Caveat (whole-program, ADR 0009):** the GC scanner does not visit linear memory, so a gc-ref env cannot be smuggled through a linear-memory registry and vice-versa. The escaping-callback lifetime answer is therefore **target-split** (Rc on wasm-mvp, engine-GC on wasm-gc) — two stories to document and test, not one uniform one.

**Leak-prevention (turn wasm-bindgen's runtime footgun into a compile-time E-code, mirroring how E0014 gates `@global` misuse — `diagnostic.ts:75`).**
1. Registration returns a **named, droppable token**, never an implicit leak.
2. The borrow-rcaps (ADR 0011) **warn/error** if an escaping-callback token is dropped without an explicit `@off`/`leak` — a new E-series diagnostic, sibling to E0014.
3. The `ScopedClosure` (Tier 1 synchronous) path is the safe default for non-retaining callbacks: no Rc, no token, auto-drop.

## 5. funcref-at-boundary — generalize `@fnref`/`@call_indirect` beyond `__fn_i_i`

The single hardcode `__fn_i_i` (`lower.ts:1608-1609, 1625-1626`) is the one place that fights everything (HM-lite, the env operand, the boundary). Generalizing the two lowering functions is the core codegen change.

1. **`lowerFnRef` (`lower.ts:1584`):** stop forcing `__fn_i_i`. **Derive** the slot's signature from the referenced `@fn`'s actual param/result wasm types (the same function-signature source the existing ref-slot injection reads), intern a per-signature key `__fn_<paramcodes>_<resultcode>` (mirroring the emitter's `sigKey`), deduped into `funcrefTable.signatures` (already an **array** — `lower.ts:1608` merely seeds the first entry). For a *closure*, `@fnref` of a capturing body returns the **fat pointer** (idx + env), not the bare i32 index.
2. **`lowerCallIndirect` (`lower.ts:1620`):** relax the fixed-arity-2 guard; accept `(cb, env?, …args)`; choose `sigKey` from the operand types instead of the constant. `IRCallIndirect.args` becomes the full list with **env first by convention**; the emitter already pushes args-in-order then the index, so no emitter change once `args` is the generalized list. The closure-call form loads `fnIndex` + `envPtr` from the fat record and calls with env as the first arg; the lifted `@fn` has env as its first param.
3. **Export an invoke trampoline so the *host* can call back (the bootstrappable, no-ref-types-at-the-boundary route — wasm-bindgen `makeMutClosure` / Go `_makeFuncWrapper`).** A keyword (e.g. `@export_callback name`) synthesizes an **exported** wasm function `__invoke_<sig>(idx:i32, env:i32, …args) -> ret` whose body is exactly an `IRCallIndirect{ tableIndex=idx, args=[env, …args], sigKey }`. Register it in `IRModule.exports`; it flows through the existing export-section/emit path unchanged. The host stores `{idx, env}` opaquely and, in its JS callback, calls `instance.exports.__invoke_<sig>(idx, env, …eventArgs)`. **The closure crosses the boundary as two plain i32s** — no reference-types support required. Reentrancy (host→guest→host) is normal nested wasm calls.
4. **(Optional, wasm-gc / ref-types) `@extern` accept/return `funcref` directly.** The `IRRefSlot`/externref precedent already exists (the `JSString` externref machinery: a `{nullable, extern}` slot + a `0x6F` encoding path). Add a `funcref` discriminant beside `extern`, teach `lowerExternParams`/`lowerExternResult` (`lower.ts:978-993`, today no ref path) to recognize an `Fn[…]` Silicon type as a funcref slot, add the `0x70` encoding next to the `0x6F` branch, and extend the export section to emit table-export descriptor `0x01` so the host can `instance.exports.__funcs.get(idx)` a callable funcref. **Preferred only for env-less callbacks** — a bare funcref drops the env, so the trampoline (step 3) is the route for closures-with-env. Reserve this for the wasm-gc graduation.

The `__closure_release` export (§4) registers the same way as the invoke trampoline — both are ordinary extra `IRExport` func entries.

## 6. Surface syntax — a stratum, not a grammar rule

**No grammar change. No `\x. e` lambda literal.** This is a hard constraint (CLAUDE.md: "add a stratum, NOT a grammar rule"; ADR 0010's LL(1) target). The `\\` sigil is the **attached-signature** token, consumed only as a signature (`lexer.ts:135`: `attachedSig`), never as an expression — so an anonymous-function form `\x. e` would require a **new expression production**, which is forbidden. The bar is set by the `..` range operator (ADR 0016): added as a 2-char lexer token riding the binary-op chain (`lexer.ts:162-170`), **no grammar production**. Lexer/strata may add surface; grammar may not grow.

Closures therefore ride the existing `FunctionCall` form as keyword calls, exactly as `@fnref`/`@call_indirect` do today:
- `&@closure body_fn, cap0, cap1, …` — build a closure: `body_fn` is a **named top-level `@fn`** whose first param is the env; `cap0…capN` are captured by value into the env in argument order. Returns the fat-record pointer.
- `&@call_closure clo, args…` — invoke (unpack `fnIndex`+`env`, `call_indirect` with env first).
- `&@export_callback body_fn` — emit the host-callable `__invoke_<sig>` + `__closure_release` exports.
- `&@on slot, ctx, body_fn` / `&@off token` — the FFI event binder (Phase 3); returns/consumes the named keep-alive token.

**Implementation = AST→AST lambda-lifting in the elaborator**, the exact pattern `loopDesugar.ts` used to land all of ADR 0016 with **zero new IR** (and `lowerTry`/`lowerWithArena` before it): lift each capturing body to a synthesized top-level `@fn(env, …args)`, prepend env-unpack `@local` binds (`&WASM::i32_load (env + 4*i)`), register it via the `@fnref` table machinery, and emit the env+record `&alloc` at the capture site. The grammar stays frozen; only the **capture analysis** is new.

**The body stays named — deliberately.** A true `\\`-attached anonymous lambda would want **let-generalization**, which HM-lite forbids (schemes come only from syntactic `[T]` decls). A named lifted body gets its scheme from its `\\` signature, so lambda-lifting is both *grammar-cheaper* and *inference-cheaper* than an anonymous form. A sugary anonymous lambda is an explicit, separate, later grammar discussion — out of scope here.

## 7. HM-lite + monomorphization integration

**HM-lite needs no new machinery.** `SiliconType` already has a `Function` kind (params/result); a closure's surface type `Fn[A -> R]` is that kind and unifies structurally (the same-kind compound rule) like any other compound. Because the body is a named lifted `@fn` whose scheme comes from its `\\` signature, the closure gets a **monomorphic** `Function` type and needs no inference of a polymorphic env and no let-generalization. Closures fit HM-lite **without touching its restrictions** — this is the second reason to lift to a named body rather than an anonymous lambda.

**Monomorphization is the enabler of the v1.x unboxed fast path and reuses the existing comptime/generics pipeline (ADR 0001).** A generic combinator `map[T,U]` taking a closure monomorphizes per `(T,U)` — and, for the unboxed path, per concrete env type — via the existing polymorphic-call checking. Each comptime instantiation produces a distinct lifted `@fn` ⇒ a distinct funcref signature ⇒ its own derived `sigKey` and its own generated invoke trampoline — exactly wasm-bindgen's per-signature *described shims*, but resolved at **comptime** instead of via a runtime describe protocol. No dictionaries, no HKT, no Fn-trait lattice (traits are deferred per the generics strategy); the Fn/FnMut distinction comes from the rcap lattice (§3), not a trait lattice. Generalizing `lowerFnRef`/`lowerCallIndirect` off `__fn_i_i` (§5) is precisely what makes funcref signatures monomorphic-per-instantiation.

## 9. Risks and the small/bootstrappable tension

- **This is the largest single v1.0 item, and it is on the critical path (no slack).** ADR 0018 already states this. The dependency chain is explicit and accepted: gate ⇒ Phases 3–4 ⇒ C1–C3 closures. There is no "ship 80% and defer events" fallback — that would miss the migration story. v1.0 planning must budget closures as the dominant cost.
- **The Zig tension is real and we are deliberately spending it.** Zig omits closures to keep "no hidden allocation / no hidden control flow / small + bootstrappable." A boxed closure *does* allocate (the env), and a host callback *does* re-enter control. We protect the identity four ways: (1) **no grammar change** — closures are strata on the existing `FunctionCall` form (§6), the bootstrappable core is untouched; (2) **no new IR node** — lambda-lifting is AST→AST (loopDesugar precedent), the call reuses `IRCallIndirect` with a second `sigKey`; (3) **allocation is visible** — the env is an ordinary `&alloc`, freed by the arena it lives in, and Tier-3's keep-alive is a *named, droppable* handle, not a hidden leak; (4) **the unboxed fast path (C5) is deferred**, so v1.0 ships *one* representation, not Rust's two-world boxed/unboxed decision procedure. The complexity that remains (capture analysis, the multi-sig ABI, the trampoline, Rc keep-alive) is the irreducible cost of host-callable captured callbacks — the thing the gate requires.
- **The escaping-FFI case leaks until the free-list lands (ADR 0008 v1.1).** `rc_drop` does not physically free. v1.0 ships C2 as a documented leak (parity with `rc.si`'s current status). Honest, not silent — the callback works for the program's lifetime; reclaim is the free-list follow-up. If this is unacceptable for a given deployment, that program must use `--target=wasm-gc` (C4), where the engine reclaims.
- **wasm-gc is whole-program and cannot mix (ADR 0009).** The Tier-3 lifetime answer is target-split (Rc on mvp, engine-GC on gc); a gc-ref env cannot pass through a linear-memory registry. Two stories to document and test.
- **Mutable capture is gated on the unimplemented borrow checker (ADR 0011).** v1.0 closures are by-value/`&`-immutable ("Fn"-shaped); `&mut` capture ("FnMut") waits on R4 escape analysis. Early closures cannot mutate captured locals directly — the documented workaround is capture an `Rc[Cell]` pointer.
- **Cycles (host ↔ closure) are not collected by Rc** (Swift's retain-cycle / Go's mandatory `Release` lesson); needs `weak` registry handles, deferred.
- **Generalizing the `__fn_i_i` hardcode touches hot lowering paths.** C0 must preserve byte-equal codegen for every existing non-funcref/single-sig program — keep `funcrefTable` `undefined` for those and gate the new multi-sig path on its presence.

## 10. Options considered

Three design families; this ADR picks **A's representation, staged boxed-first**, with **C scoped down as the FFI binder surface, not the general closure**.

### Option A — Hybrid: unboxed-monomorphized default + boxed fat-pointer on escape (the Rust `impl Fn`/`Box<dyn Fn>` split). **Chosen as the representation; build order inverted to boxed-first.**

Two representations behind one `Fn[A->R]`: unboxed (env *is* the comptime env struct, no stored code pointer, direct/inlined call) by default; boxed `[fnIndex, envPtr]` over an Rc env on escape/erasure. **Pros:** zero-cost default (the no-GC dream case; non-capturing degenerates to today's `@fnref`); the boxed form is the only storable/heterogeneous/FFI-crossable representation; reuses three existing subsystems (`@fnref` table, `rc.si` alloc, HM-lite `Function`); dual-backend clean (linear-memory + call_indirect on mvp, `(struct …)` + call_ref on gc). **Cons:** two representations = a monomorphize-vs-box decision procedure (Silicon must implement the escape/erasure trigger); code bloat from monomorphization; the heaviest option overall. **Why chosen, and why boxed-first:** the gate is exactly the escaping/heterogeneous case, which *only the boxed form* expresses — so we ship boxed first (C1–C3) to meet the gate, and add the unboxed fast path (C5) later behind escape analysis as a non-blocking optimization. This keeps the *required* representation small and defers the *optional* second representation.

### Option B — Uniform boxed closures everywhere (one fat-pointer-over-Rc, indirection on every call; the C++ `std::function`/Swift/Go funcval model). **Adopted as the v1.0 floor; not the permanent ceiling.**

One shape, no second case: every closure is `[fnIndex, envPtr]` over an Rc env, always heap, always `call_indirect`, even for non-capturing or tiny closures. **Pros:** truly uniform — one layout, one call lowering, no "is this boxed?" branch, simplest to specify/implement/reason about; solves the escaping FFI callback directly; storable heterogeneously by construction (unblocks ADR 0016's combinators); clean wasm-gc upgrade. **Cons:** an indirection cost on *every* call (load + table bounds/type-check), no inlining even when the callee is static (the `std::function` 3–5× penalty); always heap-allocates the env, even for the empty-env case that `@fnref` already handles for free, with no SBO; refcount traffic on every share; effects hidden from the optimizer behind an opaque indirect call. **Disposition:** this *is* the C1–C3 floor of Option A — we adopt B's uniform representation as v1.0 and then *grow out of it* with A's unboxed fast path (C5). The empty-env case keeps degenerating to `@fnref` so non-capturing callbacks pay nothing. Choosing B as a permanent ceiling would leave the no-GC peak-performance win on the table; choosing it as the floor is exactly right.

### Option C — Defunctionalized "explicit context" binders (Zig's `*anyopaque ctx`, comptime-synthesized; `@on`/`@export_callback` over a user-built ctx). **Rejected as the general closure; adopted in scoped form as the FFI binder surface.**

A boundary callback is not a closure value but three first-order things — a funcref index, a user-built-and-owned ctx pointer, and a host token — that travel together by convention. No compiler-synthesized env, no capture analysis, no `{code,env}` first-class value. **Pros:** avoids the three hardest parts wholesale (no lambda literal, no anonymous-env let-generalization, no escaping-env soundness proof — the ctx is an ordinary user pointer under existing rules); minimal/bootstrappable (generalize the two functions + one export + a stratum); honest visible lifetime; beats Zig on safety because the named fn lets the compiler check ctx⇔fn pairing. **Cons (decisive):** **no capture sugar** — every callback is a named `@fn` over `(ctx, args)`, so it cannot close over a surrounding accumulator, which is *strictly less expressive than ADR 0016's combinators require* (it does **not** unblock `map`/`filter`/`fold`); **manual, untyped context threading** (`i32_store ctx, …` / `i32_load (ctx+off)` by hand) is the full Zig ergonomic tax; no heterogeneous storage. **Why rejected as the general mechanism but kept as a binder:** C's honest ceiling is "FFI/event callbacks only, not general first-class functions." But its `@on`/`@off`/`@export_callback`/exported-invoke-trampoline machinery is *exactly* the boundary surface Option A's boxed form needs in C2 — so we adopt C's **binder shape and trampoline route** as the host-facing API while letting the compiler (not the user) synthesize and own the env. This gives the gate's callbacks the visible-lifetime, typed-token discipline of C **without** C's no-capture, manual-ctx tax.

**Why this combination.** The gate forces a storable, escapable, host-callable *captured* callback. B's uniform boxed representation is the smallest thing that delivers it; A's unboxed fast path is the speed we add later without moving the gate; C's binder/trampoline is the right *boundary surface*, scoped to FFI and backed by compiler-synthesized envs so it stays a real closure, not a hand-rolled context struct. We decline Zig's no-closure stance (rcaps supply the lifetime proof) and decline whole-program defunctionalization (it breaks separate compilation / CaaS) — keeping it only as the empty-env `@fnref` special case Silicon already ships.

## Consequences

- **Positive:** unblocks ADR 0018 Phases 3–4 (the 100% gate) and ADR 0016's deferred combinators on a representation that actually escapes; reuses the `@fnref` table, `rc.si`, HM-lite `Function`, and the loopDesugar AST→AST pattern (minimal new identity); no grammar change, no new IR node; the rcap system gives typed, lifetime-checked capture without Zig's `*anyopaque` tax; a clean wasm-gc upgrade and a non-blocking unboxed fast path are both available later.
- **Negative:** the largest single v1.0 item, on the critical path; escaping-to-host closures leak until ADR 0008's free-list lands (honest, documented); mutable capture is gated on the unimplemented borrow checker; the Tier-3 lifetime story is target-split (Rc vs engine-GC); generalizing `__fn_i_i` touches hot lowering paths and must preserve byte-equal codegen for existing programs.
- **Follow-up work:** C0–C5 in §8 (C0–C3 v1.0-critical, C4–C5 deferred); the free-list/`AllocatorABI.free` under Rc (ADR 0008 v1.1) so Tier-3 reclaims; `weak` registry handles for host↔closure cycles; the drop-without-`@off` E-code (sibling to E0014); the optional `funcref` `IRRefSlot` + table-export for the wasm-gc boundary; eventually Perceus-style compiler-inserted dup/drop so users never hand-write `rc_clone`/`rc_drop` on registration.
