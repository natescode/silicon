# ADR 0019 — First-class closures / capture: the keystone that unblocks FFI callbacks/events and the poll-reactor for 100% web/bun coverage

- **Status:** Accepted — **revised 2026-06-05 to a multi-mode, mode-tiered model** (see §2 and the change-log at the end of §2). **Implemented through C1 + the C2 escape gate (2026-06-09):** C0 (the multi-signature funcref ABI) and C1 (non-escaping closures with by-value capture — `@closure`/`@call_closure`, the loopDesugar-style `closureDesugar.ts`, running on every mode) are shipped and tested; the C2 conservative escape/host-reachability classifier + the mode-gate that rejects a closure crossing an `@extern` boundary are shipped. The C2 wasm-gc **Tier-B codegen** (the `(struct $Clo)` record, the `__invoke_<sig>(externref,…)` trampoline, the `ref.cast`/`extern.convert` ops, the host shims) and C3 (poll-reactor continuations) remain — the ADR's "largest single v1.0 item" (§9).
- **Date:** 2026-06-05
- **Deciders:** NatesCode
- **Scope note:** Closures are **on the v1.0 critical path, not deferred.** 100% web/bun API coverage is a HARD v1.0 gate (product decision — it is the migration value-add for Node/Bun users). Per ADR 0018, that gate ⇒ Phases 3–4 (callbacks/events + poll-reactor) ⇒ closures with **host-callable captured environments**. This ADR designs that feature and owns its cost. The motivating, load-bearing case is the **escaping FFI-callback closure** (a wasm guest closure a JS/Bun host stores in a registry and invokes at an unbounded later time).
- **Mode-tiering decision (the keystone refinement).** Silicon is **multi-mode**: a systems/native substrate (`--target=wasm-mvp` default + `--native` QBE — bump allocator + arenas, no GC, ADR 0008) and a managed substrate (`--target=wasm-gc` — engine GC, externref/structref, ADR 0009). Closures **tier by mode**, because the one case that forces a heap/refcount/traced env — the **escaping host-callable** closure — *only exists where a JS/Bun host owns a registry*, and that is exactly where wasm-gc lives. A `--native` binary and a non-JS wasm embedding have **no host registry to outlive a scope against**, so they never need that tier. Therefore: **non-escaping closures are zero-cost and available on every mode; escaping/host-callable closures are `--target=wasm-gc`-only** (engine-GC'd, no leak, cycles collected). **This dissolves the original uniform-`rc.si`-boxed floor and its "leaks until the free-list lands" caveat** — that caveat existed only because the escaping env was placed in linear memory with no reclaimer; under mode-tiering the escaping env is an engine-traced GC struct and the Rc-on-linear-memory tier **ceases to exist**.
- **Verified foundation (reproduced).** The pivot's single load-bearing fact — *Bun/JSC runs wasm-gc today* — was confirmed **two independent ways on the installed runtime (Bun 1.3.14)**, not merely recalled: (1) a hand-assembled wasm-gc module using `struct.new` + `struct.get` compiled and ran under Bun's `WebAssembly` with **no opt-in flag** (`f() = 42`); (2) the repo's own `CharCodeArray` — a wasm-gc `(array (mut i16))` — compiles and runs through Silicon's pipeline under `--platform=bun` (`examples/charcode_array.si` → `Hi! / 3 / 72`). WasmGC is **one** proposal: an engine that runs its arrays and structs runs the `(struct $Clo)` env representation. Crucially, the **v1.0 escaping-closure *invocation* reuses the existing `call_indirect`** (the `(ref $Clo)` crosses to the host as an `externref` — the *shipped* JSString externref path, `lower.ts:1426-1479`, already crosses under Bun), so the v1.0 shape needs **no engine feature Bun lacks**; `call_ref` is only the deferred v1.x optimization (§5). This matches ADR 0009's engine table (JSC ships WasmGC ≥ Safari 18.4, 2025; WebKit blog 16301 dates it to Safari 18.2 / Dec 2024). **Surgically separate from JSPI:** JSPI is still absent in JSC/Bun (`oven-sh/bun#20878`), but the closure pivot needs wasm-gc (present), not JSPI (absent), and ADR 0018 already routes async via Asyncify on Bun. The two are independent; the pivot is unaffected.

## 1. Context — closures are the named gap, and now the gate

A closure is a *code pointer + the values it captured*. Silicon has the code pointer and nothing else.

**What exists.** `@fnref name` (`lower.ts:1584`) accepts only a bare top-level `@fn` name, interns it into a per-module funcref table, and returns `{kind:'Const', wasmType:'i32', value: idx}` — a table index, an empty-env code pointer. `@call_indirect cb, arg` (`lower.ts:1620`) is hardwired to **one** signature `{key:'__fn_i_i', params:['i32'], result:'i32'}` (`lower.ts:1608-1609, 1625-1626`), exactly one i32 argument, i32 result, and **no environment operand**. So the runtime ABI is `i32(table-index) → i32 → i32` with no env. There is no way to write a callback that closes over a surrounding binding, and `@extern` collapses every type to a scalar (`lowerExternParams`/`lowerExternResult`, `lower.ts:978-993`), so no `funcref`/`(ref func)` value crosses the boundary either.

**Where the gap is already named.**
- **ADR 0016, fact #4 (`0016:52-56`):** "`@fnref` has no environment capture. It is fixed i32→i32 … A callback cannot close over a surrounding accumulator — which is the real reason internal iteration is declined (Option C), and why combinators wait on closures." ADR 0016's deferred list (`0016:489-491`) records `map`/`filter`/`fold` and the *dispatched* `IterStep[T,R]` for-each surface as blocked on "closures exist."
- **ADR 0018, Blocker C and P4 (`0018:21-24, 84-85`):** callbacks/events are impossible because funcref never crosses the boundary and there is no capture. P4 explicitly hands the design to this ADR and marks it "v1.0-critical for the 100% gate … the largest single item." Its **Phase 3** (callbacks + events: `addEventListener`, `setTimeout(cb)`, `onmessage`, `EventTarget`, streaming `.on('data')` — ~80%→97%) and **Phase 4** (poll-reactor + tasks: a `Future[T]` whose wake continuation is a closure over the awaiting frame — ~97%→100%) **both depend on closures**, and both are now hard-gate v1.0.

**The motivating case, stated plainly.** The hard case is not the local closure that the optimizer could inline. It is the **escaping FFI-callback closure**: the guest builds a `{code, env}` value and hands it to a host registry (a JS `addEventListener`, a Bun event emitter, a Promise `.then`), which stores it and calls it back at an **unbounded later time**. The env must outlive the lexical scope that built it — and the host outlives every arena. This is the case that decides the representation and the lifetime model, and it is the case ADR 0008's arenas cannot express (an escaping callback has no dominating scope to reset against; the only arena answer is `&move_to_parent_arena`-to-root, i.e. a deliberate leak — ADR 0008 flags return-value escape as unsolved).

**The tension this ADR must reckon with.** Zig deliberately ships **no closures** for exactly Silicon's reasons: no hidden allocation, no hidden control flow, small and bootstrappable. A real closure forces captured values *somewhere* that outlives the frame (the heap), and a no-implicit-allocator language has nowhere to implicitly put them. Silicon's `@fnref` + `@call_indirect` is already ~80% of Zig's "pass a function pointer + an explicit context" pattern. The honest question is whether to stop there (Zig's stance) or build real closures — and the **hard 100% gate forces the answer**: host-callable *captured* callbacks are required, and a bare code pointer with a hand-built context is too lossy and too unsafe for the migration story (untyped `*anyopaque`, manual lifetime, no compiler-checked ctx⇔fn pairing). We build closures, but we build the **smallest representation that satisfies the gate first**, and grow the optimization later.

## 2. Decision

**Ship a single closure surface type `Fn[A -> R]` (HM-lite's existing `Function` kind), and tier its *representation* by two axes — does the env ESCAPE its defining scope, and is it HOST-REACHABLE — granting each mode exactly the closure power its runtime substrate honestly supports.** There is **no uniform boxed floor**: a closure that does not need a heap/refcount/traced env never gets one, and the only tier that does (escaping-to-host) lives **on `--target=wasm-gc` only**, where the engine reclaims it. Concretely:

1. **NON-ESCAPING closures — every mode (`wasm-mvp`, `--native`, `wasm-gc`), zero-cost.** A closure that does not outlive its defining scope (combinators, comparators, synchronous non-retained host callbacks, intra-arena escapes) is **comptime-monomorphized and unboxed**. The lifted body is a top-level `@fn(env, …args)`; the env is an **anonymous struct** of the captured i32s, materialized in the **frame** or the active **arena** (ADR 0008) — *never* a heap-with-refcount block. The consumer is specialized per concrete env type (ADR 0001 comptime), so the call is a **direct call** to the lifted `@fn` (inlinable, CSE/LICM-eligible per ADR 0012), not a `call_indirect`; no `fnIndex` word is even stored. **The empty-env degenerate case is byte-identical to today's `@fnref`** — a bare i32 table index (`lower.ts:1584`), zero new cost, zero regression (the C0 byte-equality guard: `funcrefTable` stays `undefined` for these programs). On `wasm-gc` the same source compiles; the env is just plain GC-traced locals (arenas elide to `{body}` per ADR 0009's lifecycle-elision table), and the engine never has to trace a frame/arena env that is gone at scope exit.

2. **ESCAPING / HOST-CALLABLE closures — `--target=wasm-gc` ONLY (rejected on `wasm-mvp`/`--native`).** The load-bearing FFI-callback case (a JS/Bun registry stores the closure and invokes it at an unbounded later time) is represented as a **managed fat record**: `(type $Clo (struct (field i32 fnIndex) (field (ref $Env))))`, where `$Env = (struct (field …))` holds the captured bindings in declaration order. Invocation = `struct.get` the two fields, then `call_indirect` (v1.0 table route) or `call_ref` (v1.x) against the derived per-signature key. **Crossing to the host:** the closure is handed across `@extern` as an **`externref`** (the engine boxes `(ref $Clo)` as `externref`); the host stores it opaquely in its registry (`addEventListener`, `.then`, `EventEmitter`) and, to call back, invokes an exported `__invoke_<sig>(clo: externref, …args)` trampoline that `ref.cast`s back to `(ref $Clo)` and dispatches. **Lifetime = engine GC:** V8/SpiderMonkey/JSC trace the `externref` the host holds, keep `$Clo` and `$Env` alive while registered, and collect **both — including host↔closure cycles — when the host drops it**. `__closure_release` is a **no-op**. **On `wasm-mvp`/`--native` this use-case is rejected at typecheck** with a new E-code (sibling to E0012/E0013/E0014): *"host-callable escaping closure requires `--target=wasm-gc`."* On native this rejection can *never legitimately fire* — there is no JS host to call back; it only ever fires on a web/bun program compiled with the wrong target, which is a **fix-the-flag diagnostic, not a lost capability**.

3. **The `rc.si`-over-linear-memory boxed tier does NOT exist in any mode.** The original v1.0 floor placed the escaping env in linear memory with `rc_drop`-that-doesn't-free, accepting an honest leak until ADR 0008's free-list. Under mode-tiering that tier is **deleted, not documented**: the escaping case is served only by engine GC (no leak, cycles collected), and the non-escaping case by the allocator that already reclaims it (frame/arena). The leak caveat is **dissolved** — see §4.

4. **Decline defunctionalization (whole-program apply-switch) as the general mechanism** — it breaks separate compilation / the CaaS `MetadataReference` story (ADR 0012). It survives only implicitly as the empty-env `@fnref` special case.
5. **Decline Zig's no-closure stance.** The rcap system (ADR 0011) supplies the lifetime proof Zig forces onto the programmer, so Silicon can offer *real, typed* closures without the `*anyopaque ctx` tax — and on every mode native gets *more* than Zig (non-escaping **and** arena-escaping captured closures), declining only the GC-requiring tier.

**Why mode-tiered, not uniform-boxed (the heart of the revised decision).** The original ADR shipped one uniform `[fnIndex,envPtr]`-over-`rc.si` closure on *every* target because the gate needs an escaping/heterogeneous representation and that form is the smallest thing that escapes. But that taxed **native** — which has no host to escape *to* — with an indirection + heap-alloc + refcount + an *uncollectable* leak, for a capability it can never exercise. The refined insight: **the escaping-to-host case is definitionally a JS-host case** (ADR 0018's load-bearing examples — `addEventListener`, `setTimeout`, `.then`, `onmessage` — are *all* JS APIs), and the JS host lives **only on web/bun**, which is **exactly where wasm-gc lives**. So we put the escaping tier where it belongs (managed mode, engine-GC'd) and give native only the zero-cost non-escaping tier it can actually use. Native does not just "not lose" — it gains a **strictly faster representation** than the uniform-boxed floor it sheds.

**Why escaping=wasm-gc-only is *sound and complete* for the gate.** The verified foundation (scope note) confirms Bun/JSC runs the `(struct …)` env + the `externref` boundary today, default-on — and the v1.0 invocation reuses the **existing** `call_indirect` (so `call_ref` is the deferred v1.x optimization, not a gate dependency). So the gate's escaping-callback case (ADR 0018 Phase 3/4) reaches **100% on both web AND bun** under `--target=wasm-gc`, with **no JSPI dependency** (the closure machinery is independent of the async mechanism; Asyncify remains the portable `@await` floor). The honest cost is only that a **bun program registering a host-retained callback must compile as `--target=wasm-gc`** — and that is the same posture ADR 0009 already establishes for the managed target. The gate gets *cleaner* (no documented leak), not harder.

**The candid magnitude (unchanged at the top line, redistributed below it).** Closures are still the **largest single item** in the v1.0 programme (ADR 0018 says so). We commit to capture analysis, a generalized multi-signature funcref ABI, an anonymous-struct env, the comptime-monomorphized non-escaping path, the wasm-gc `(struct $Clo)` escaping representation + `externref` boundary + the `__invoke_<sig>` trampoline, and **a conservative escape/host-reachability classifier** (the one cost moved *forward* — see §9 residual risks). We **drop** the `rc.si` keep-alive token, the real `__closure_release` body, the drop-without-`@off` leak E-code, and the target-split-Rc-vs-GC double lifetime story. The escaping lifetime story is now **single (engine GC)**, not two.

---

**Change-log vs the original (uniform-boxed) decision, for reviewers who signed off on "100% on the boxed mvp floor."**
- **Was:** one uniform `[fnIndex,envPtr]`-over-`rc.si` closure on every target; unboxed/monomorphized fast path deferred to C5; wasm-gc closures "C4 — No (opt-in), not v1.0."
- **Now:** non-escaping = unboxed/monomorphized on **every** mode (the old C5 fast path, pulled *forward* for the non-escaping case); escaping/host-callable = **wasm-gc-only managed struct (the old C4, now the ONLY escaping path and v1.0-critical)**; the `rc.si`-on-linear-memory escaping tier (old §2.1 / Tier-3-on-mvp) is **removed**.
- **Net for each mode:** native/`wasm-mvp` ships *faster* closures (zero-cost, no dead-weight leak) and *loses only a tier it cannot use*. web/bun reach the 100% gate via `--target=wasm-gc`, leak-free, cycles collected.

## 3. Capture semantics and the ADR-0011 rcap interaction

**v1.0 capture is by-value / `&`-immutable only. Mutable (`&mut`/`&uniq`) capture into an escaping closure is unsound and is rejected.** This is forced by ADR 0011's invariants, not a stylistic choice:

- **By-value / `&val` / `&` (immutable shared):** copy the captured i32 into the env. For an `Int`/`Bool` this copies the value; for a heap value (`String`, `Sum` pointer, `Vec`) it copies the *pointer*, which is sound because the pointee is independently lived (arena/Rc), not the stack frame. No aliasing is introduced, nothing outlives its scope — sound for both escaping and non-escaping closures, with no borrow checker required. This is the **only** mode an escaping closure may use in v1.0.
- **`&mut` / `&uniq` (move-in, uniqueness-preserving):** allowed **only for a provably non-escaping** (`local`-scoped, not stored/returned) closure, and **v1.0 conservatively forbids it entirely** until ADR 0011's escape analysis (R4) is implemented. The reason is exact: an escaping closure that captured `&mut x` where `x:@local` would let that `&mut` **outlive `x`'s frame** (ADR 0011 **R1**: a reference may not outlive its scope) **and alias** the caller's other live refs to `x` (ADR 0011 **R2**: at any point 0..n `&` XOR 0..1 `&mut`). Both invariants break. Rejected with a borrow-style E-code.

This recovers Rust's **Fn vs FnMut** split from the **rcap lattice instead of a trait hierarchy**: a by-value/`&`-immutable closure is "Fn"-shaped (re-callable, immutable env, freely shareable, escapable); a `&mut`-capturing closure is "FnMut"-shaped (mutable env, must not escape) and waits on the borrow checker. Mutating captured state in v1.0 is the explicit Swift/Go escape hatch: capture an `Rc[Cell]`/heap-cell **pointer** by value and mutate through it — the pointer is copied, the cell is shared, and the aliasing is visible in the source.

**Capture set.** The free variables the body reads that are bound in an enclosing scope; the env materializes exactly those, in declaration order. Because the body is a **named, lambda-lifted top-level `@fn`** (§6), capture is the only genuinely new analysis — call and dispatch reuse the `@fnref`/`@call_indirect` machinery.

**Effect-lattice interaction (ADR 0012).** A closure inherits its body's effect class. An env holding no `&mut` and a body holding no ocap ⇒ the closure is `pure`; because the **non-escaping path is unboxed/monomorphized in v1.0 on every mode** (§2.1), its **direct call is CSE/LICM-eligible immediately** (not gated on a deferred fast path). An env that is `local`/`&uniq` ⇒ guard-free static in-place reuse (ADR 0012 §4), beating Perceus's runtime `rc==1` branch. Conversely a **host-callable escaping** closure (Tier B, wasm-gc only) re-enters arbitrary host effects on invocation, so it sits **high on the lattice** (re-entrant/impure) and the optimizer must **not** treat its `call_indirect`/`call_ref` dispatch as pure.

## 4. Env lifetime — the leak caveat is DISSOLVED by mode-tiering

The original ADR carried three tiers culminating in a **Tier 3 Rc-on-linear-memory escaping env that leaked until ADR 0008's free-list lands**. That tier existed *only* because the escaping env was forced into linear memory (the uniform-boxed floor) on a target with no reclaimer. **Mode-tiering removes the cause, so the leak is dissolved, not documented.** Two tiers remain — one served by the allocator that already reclaims it, one served by the engine that reclaims it — and **no tier leaks**.

**Tier A — non-escaping (the common case, ~80%): frame / arena, on EVERY mode, zero-cost.** A closure that does not outlive its defining scope keeps its env as an **anonymous struct in the frame** (the monomorphized fast path, §2.1) or in the active **arena** (when it must outlive the frame but is bulk-freed at `&with_arena` exit, ADR 0008). No refcount, no GC, no token; freed in bulk at scope/arena exit like every other value. A **synchronous host callback the host does not retain** (the wasm-bindgen `ScopedClosure` analog — a sort comparator, a synchronous `forEach`) lives here: arena-scoped env, auto-drop at `&with_arena` exit. The **empty-env degenerate case is byte-identical to today's `@fnref`** (bare i32 table index). On `wasm-gc` the env is plain GC-traced locals (arenas elide per ADR 0009). This tier is fully sound today and carries cases 1–3 of the technique map (§"technique assignment") unboxed on all modes.

**Tier B — escaping / host-callable (the FFI-callback hard case): `--target=wasm-gc` ONLY, engine-GC'd.** Arenas fail here by construction — an escaping host callback has **no dominating scope to reset against** (ADR 0008 flags this as the unsolved return-escape case; the only arena answer is promote-to-root, i.e. a deliberate leak). So this tier does **not** use linear memory at all. The closure is the managed `(type $Clo (struct (field i32 fnIndex) (field (ref $Env))))`; the host holds it as an `externref`; **V8/SpiderMonkey/JSC trace it, keep `$Clo`+`$Env` alive while the host holds it, and collect both — including host↔closure cycles — when the host drops it.** `__closure_release` is a **no-op** (registration/unregistration are just the host acquiring/dropping its `externref`; the engine does the reclaim). The Swift/Rc/Go retain-cycle weakness the original ADR had to defer to "`weak` handles later" **vanishes** — the engine traces cycles.

> **The dissolved caveat, stated plainly.** The original §4 Tier 3 said: *"`rc_drop` decrements but does not physically free … an escaping-then-released callback leaks until ADR 0008's free-list lands."* Under mode-tiering **there is no `rc_drop` and no linear-memory env on the escaping path** — the only escaping path is wasm-gc, where the engine reclaims. The free-list dependency on the closure-lifetime story is **gone**. (ADR 0008's free-list is still wanted for *other* linear-memory data, but the escaping-closure gate no longer waits on it.)

**On `wasm-mvp`/`--native`, Tier B does not exist — and that is correct, not a gap.** A native binary has no JS host registry; the escaping-host case has no referent. A non-JS wasm embedding (e.g. an edge runtime *without* wasm-gc) that genuinely wants a host-retained callback under linear memory is the one residual niche where the deleted tier would have applied — it is shrinking but not empty in 2026 (see §9 residual risks); such a program must move to `--target=wasm-gc` or hand-roll the Zig-style `fnref` + explicit `ctx` pointer (which `@fnref`/`@call_indirect` already are, `lower.ts:1584/1620`) and own the lifetime manually.

**Whole-program constraint (ADR 0009), now SINGLE-story not split.** The escaping representation lives entirely in managed refs, so a gc-ref `$Env` cannot be smuggled through a linear-memory registry; a program mixing a linear-memory data structure with host-callable closures inherits ADR 0009's no-mixing sharp edge ("I want arenas for my hot loop AND host callbacks in one module" is not expressible). But this is the *same* whole-program flag ADR 0009 already imposes, and — crucially — there is **no longer a second (Rc-on-mvp) escaping story to document and test in parallel**. The escaping lifetime answer is **one** thing (engine GC), gated by **one** flag.

**Leak-prevention is now structural, not a hand-written discipline.**
1. The escaping closure is engine-traced; there is no token to forget, no `.forget()` footgun, no drop-without-`@off` leak.
2. The non-escaping (`ScopedClosure`) path on all modes is the safe default for non-retaining callbacks: no Rc, no token, auto-drop at scope/arena exit.
3. The only compile-time E-code needed is the **mode gate**: an escaping/host-callable closure on `wasm-mvp`/`--native` is rejected with *"requires `--target=wasm-gc`"* (sibling to E0012/E0013, mirroring the JSString-without-web/bun gate the repo already ships). **Divergence is in CAPABILITY (what compiles), never in SEMANTICS** (a closure that *does* compile behaves identically on every target).

## 5. funcref-at-boundary — generalize `@fnref`/`@call_indirect` beyond `__fn_i_i`

The single hardcode `__fn_i_i` (`lower.ts:1608-1609, 1625-1626`) is the one place that fights everything (HM-lite, the env operand, the boundary). Generalizing the two lowering functions is the core codegen change.

1. **`lowerFnRef` (`lower.ts:1584`):** stop forcing `__fn_i_i`. **Derive** the slot's signature from the referenced `@fn`'s actual param/result wasm types (the same function-signature source the existing ref-slot injection reads), intern a per-signature key `__fn_<paramcodes>_<resultcode>` (mirroring the emitter's `sigKey`), deduped into `funcrefTable.signatures` (already an **array** — `lower.ts:1608` merely seeds the first entry). For a **non-escaping** capturing body, `@fnref` does not produce a stored code pointer at all (the consumer is monomorphized to a direct call, §2.1); for an **escaping (wasm-gc) closure**, it produces the `(ref $Clo)` managed fat record (`fnIndex` field + `(ref $Env)` field).
2. **`lowerCallIndirect` (`lower.ts:1620`):** relax the fixed-arity-2 guard; accept `(cb, env?, …args)`; choose `sigKey` from the operand types instead of the constant. `IRCallIndirect.args` becomes the full list with **env first by convention**; the emitter already pushes args-in-order then the index, so no emitter change once `args` is the generalized list. The non-escaping path lowers to a **direct `call`** to the lifted `@fn` (env in the frame); the escaping (wasm-gc) path does `struct.get` for `fnIndex`/`$Env` then `call_indirect` (v1.0 table route) or `call_ref` (v1.x).
3. **Export an `externref` invoke trampoline so the JS/Bun *host* can call back (wasm-gc route — the verified shape).** A keyword (e.g. `@export_callback name`) synthesizes an **exported** wasm function `__invoke_<sig>(clo: externref, …args) -> ret` whose body `ref.cast`s the `externref` back to `(ref $Clo)`, `struct.get`s `fnIndex`+`$Env`, and dispatches via `IRCallIndirect{ args=[env, …args], sigKey }`. Register it in `IRModule.exports`; it flows through the existing export-section/emit path unchanged. The host stores the **`externref` opaquely** (`addEventListener`, `.then`, `EventEmitter`) and, in its JS callback, calls `instance.exports.__invoke_<sig>(clo, …eventArgs)`. The engine keeps `$Clo`+`$Env` alive while the host holds the `externref` and collects both (incl. cycles) when it drops. **The escaping closure crosses the boundary as one `externref`** — the verified Bun/JSC shape (scope note: a hand-assembled `struct` module and the shipped `CharCodeArray`/`externref` path both run under Bun 1.3.14; the v1.0 dispatch is the existing `call_indirect`). Reentrancy (host→guest→host) is normal nested wasm calls. **`__closure_release` is a no-op** (the engine reclaims; there is no Rc to drop). *(On `wasm-mvp`/`--native` this keyword is rejected — Tier B does not exist there, §4.)*
4. **`@extern` accept/return `externref`/`funcref` directly (wasm-gc).** The `IRRefSlot`/externref precedent already exists (the `JSString` externref machinery, `lower.ts ~1426-1479`: a `{nullable, extern}` slot + the `0x6F`/`(ref extern)` encoding path). Recognize an `Fn[…]` Silicon type as a `(ref $Clo)`→`externref` slot at the boundary; the `funcref` `0x70` elemtype path is the env-less degenerate case (a bare funcref drops the env, so the `externref`-`$Clo` route of step 3 is the closures-with-env path). This is the wasm-gc representation, default on web and **verified default-on under Bun/JSC** (scope note).

The `__invoke_<sig>` export is the only synthesized boundary entry that carries a body; `__closure_release` is a **no-op stub** (§4 — the engine reclaims), kept only if a host's unregister path wants a symbol to call.

## 6. Surface syntax — a stratum, not a grammar rule

**No grammar change. No `\x. e` lambda literal.** This is a hard constraint (CLAUDE.md: "add a stratum, NOT a grammar rule"; ADR 0010's LL(1) target). The `\\` sigil is the **attached-signature** token, consumed only as a signature (`lexer.ts:135`: `attachedSig`), never as an expression — so an anonymous-function form `\x. e` would require a **new expression production**, which is forbidden. The bar is set by the `..` range operator (ADR 0016): added as a 2-char lexer token riding the binary-op chain (`lexer.ts:162-170`), **no grammar production**. Lexer/strata may add surface; grammar may not grow.

Closures therefore ride the existing `FunctionCall` form as keyword calls, exactly as `@fnref`/`@call_indirect` do today (the surface is identical on every mode — only the *representation* the elaborator selects differs, per §2/§4):
- `&@closure body_fn, cap0, cap1, …` — build a closure: `body_fn` is a **named top-level `@fn`** whose first param is the env; `cap0…capN` are captured by value into the env in argument order. **Non-escaping ⇒ no record (monomorphized, env in frame/arena); escaping (wasm-gc) ⇒ a `(ref $Clo)` managed fat record.**
- `&@call_closure clo, args…` — invoke (direct `call` to the lifted `@fn` for non-escaping; `struct.get`+`call_indirect`/`call_ref` for the escaping wasm-gc record).
- `&@export_callback body_fn` — emit the host-callable `__invoke_<sig>(externref, …)` export (wasm-gc only; **rejected on `wasm-mvp`/`--native`** with the mode-gate E-code, §4).
- `&@on slot, body_fn` / `&@off handle` — the FFI event binder (Phase 3); on wasm-gc the host simply holds/drops the `externref` (no keep-alive token to thread — the engine traces it). The `ctx` operand of the original Rc design is **gone**: the env *is* the engine-traced `$Env`.

**Implementation = AST→AST lambda-lifting in the elaborator**, the exact pattern `loopDesugar.ts` used to land all of ADR 0016 with **zero new IR** (and `lowerTry`/`lowerWithArena` before it): lift each capturing body to a synthesized top-level `@fn(env, …args)`, prepend env-unpack `@local` binds, register it via the `@fnref` table machinery. For the **non-escaping** case the env-unpack reads frame/arena locals and the consumer is monomorphized; for the **escaping wasm-gc** case the env is a `struct.new $Env` and the unpacks are `struct.get`. The grammar stays frozen; the new analysis is **capture + the escape/host-reachability classifier** (§9) that picks which representation.

**The body stays named — deliberately.** A true `\\`-attached anonymous lambda would want **let-generalization**, which HM-lite forbids (schemes come only from syntactic `[T]` decls). A named lifted body gets its scheme from its `\\` signature, so lambda-lifting is both *grammar-cheaper* and *inference-cheaper* than an anonymous form. A sugary anonymous lambda is an explicit, separate, later grammar discussion — out of scope here.

## 7. HM-lite + monomorphization integration

**HM-lite needs no new machinery.** `SiliconType` already has a `Function` kind (params/result); a closure's surface type `Fn[A -> R]` is that kind and unifies structurally (the same-kind compound rule) like any other compound. Because the body is a named lifted `@fn` whose scheme comes from its `\\` signature, the closure gets a **monomorphic** `Function` type and needs no inference of a polymorphic env and no let-generalization. Closures fit HM-lite **without touching its restrictions** — this is the second reason to lift to a named body rather than an anonymous lambda.

**Monomorphization is the enabler of the v1.x unboxed fast path and reuses the existing comptime/generics pipeline (ADR 0001).** A generic combinator `map[T,U]` taking a closure monomorphizes per `(T,U)` — and, for the unboxed path, per concrete env type — via the existing polymorphic-call checking. Each comptime instantiation produces a distinct lifted `@fn` ⇒ a distinct funcref signature ⇒ its own derived `sigKey` and its own generated invoke trampoline — exactly wasm-bindgen's per-signature *described shims*, but resolved at **comptime** instead of via a runtime describe protocol. No dictionaries, no HKT, no Fn-trait lattice (traits are deferred per the generics strategy); the Fn/FnMut distinction comes from the rcap lattice (§3), not a trait lattice. Generalizing `lowerFnRef`/`lowerCallIndirect` off `__fn_i_i` (§5) is precisely what makes funcref signatures monomorphic-per-instantiation.

## 8. Phased plan (tied to ADR 0018 Phases 3–4 and ADR 0016's deferred combinators)

The ABI generalization is the prerequisite; the **non-escaping monomorphized form** unblocks the combinators on every mode; the **wasm-gc escaping form** is the gate. The old "C4 wasm-gc — not v1.0" and "C5 unboxed fast path — deferred" are **re-staged**: the non-escaping unboxed path is pulled *forward* (it is now the v1.0 non-escaping representation), and the wasm-gc escaping path is pulled *forward* (it is now the *only* escaping path and therefore v1.0-critical for the gate).

## 9. Risks and the small/bootstrappable tension

- **This is the largest single v1.0 item, and it is on the critical path (no slack).** ADR 0018 already states this. The dependency chain is explicit and accepted: gate ⇒ Phases 3–4 ⇒ C1–C3 closures. There is no "ship 80% and defer events" fallback — that would miss the migration story. v1.0 planning must budget closures as the dominant cost.
- **THE NEW LOAD-BEARING COST: escape analysis moves from "speed" to "correctness," forward into v1.0.** The original ADR shipped one uniform representation and deferred the escape/erasure decision to C5, so a misclassification was only a missed optimization. Under mode-tiering the **escape/host-reachability classifier is on the C2 critical path**, and misclassifying an escaping-to-host closure as non-escaping is a **soundness bug** (a frame/arena env handed to a host that outlives it). ADR 0011's R4 escape analysis is unbuilt, so **v1.0 must ship a conservative over-approximation**: treat *any* closure that syntactically reaches `@extern`/`@export_callback` as escaping-to-host ⇒ requires `--target=wasm-gc`. This is sound by construction but **may reject some safe `wasm-mvp` programs** (a closure that crosses `@extern` but the host provably does not retain). That false-rejection is the accepted v1.0 cost of not having R4; it loosens as R4 lands.
- **Two representations land in v1.0, earlier than the original plan.** The original deferred the unboxed/monomorphized decision to C5. Mode-tiering puts *both* the non-escaping unboxed form (C1) and the escaping wasm-gc managed-struct form (C2) in v1.0. This is more codegen + test surface than "one uniform boxed closure" — mitigated by the clean split (the classifier picks exactly one; there is no hybrid per call site) and by the empty-env degenerate staying byte-identical to `@fnref`.
- **wasm-gc is whole-program and cannot mix (ADR 0009) — but the lifetime story is now SINGLE, not split.** A gc-ref `$Env` cannot pass through a linear-memory registry, so "arenas for my hot loop AND host callbacks in one module" is not expressible; that is the existing ADR 0009 whole-program flag, unchanged. The improvement over the original: there is **no longer a parallel Rc-on-mvp escaping story** — the escaping lifetime answer is one thing (engine GC), one flag, one test matrix. The original's "two stories to document and test" is **halved**.
- **Residual niche: a JS-ish host WITHOUT wasm-gc.** Dropping the Rc-on-linear-memory tier means a `wasm-mvp` embedder that is **not** pure-native yet **lacks** wasm-gc (an edge runtime that holds callbacks but has no engine GC) has no host-callback answer except "the tier you deleted." Adversarial check: this niche is **shrinking but not empty in 2026** — but it is also not the gate's target (web+V8 and bun/JSC both ship wasm-gc default-on, verified). For such an embedder the answer is the Zig-style hand-rolled `fnref` + explicit `ctx` (`lower.ts:1584/1620`), owning the lifetime manually. We accept this as a deliberate, documented cut, not a silent one.
- **Mutable capture is gated on the unimplemented borrow checker (ADR 0011).** v1.0 closures are by-value/`&`-immutable ("Fn"-shaped) on every mode; `&mut` capture ("FnMut") waits on R4 escape analysis. Workaround: capture an `Rc[Cell]`/heap-cell **pointer** by value and mutate through it (the same on every mode).
- **Cycles are collected on the only mode that has them.** Host↔closure cycles exist *only* on the escaping (wasm-gc) path, where the **engine traces them** — so the Swift/Go retain-cycle / `weak`-handle burden the original ADR deferred is **gone**. Native has no host, so it has no host↔closure cycles to collect.
- **Generalizing the `__fn_i_i` hardcode touches hot lowering paths.** C0 must preserve byte-equal codegen for every existing non-funcref/single-sig program — keep `funcrefTable` `undefined` for those and gate the new multi-sig path on its presence (the C0 byte-equality guard).

## 10. Options considered

Four design families. This ADR picks **A's two representations, but assigned BY MODE rather than by a per-call escape/erasure trigger**: the non-escaping unboxed form on every mode, the managed `(struct $Clo)` escaping form on wasm-gc only. **Option B (uniform boxed-over-Rc everywhere) is now REJECTED as the floor** — the original ADR adopted it; the multi-mode steer overturns that. **Option C** survives as the FFI binder surface; **Option D (mode-tiered)** is the new, chosen frame.

### Option A — Hybrid: unboxed-monomorphized non-escaping + a fat record on escape (the Rust `impl Fn`/`Box<dyn Fn>` split). **Both representations chosen; assignment is BY MODE, not by a per-site box-vs-unbox trigger.**

Two representations behind one `Fn[A->R]`: unboxed (env *is* the comptime env struct, no stored code pointer, direct/inlined call) for non-escaping; a fat record for escape. **Pros:** zero-cost non-escaping (the no-GC dream case; non-capturing degenerates to today's `@fnref`); the fat record is the only storable/heterogeneous/FFI-crossable form; reuses `@fnref` table + HM-lite `Function`; dual-backend clean. **Cons:** two representations + an escape/erasure decision; monomorphization code bloat. **Why chosen, and how the mode-tiering simplifies it:** instead of a *per-call-site* monomorphize-vs-box trigger (the "heaviest option"), the **mode + a single escape-to-host predicate** picks the representation — non-escaping ⇒ unboxed (all modes), escaping-to-host ⇒ managed `(struct $Clo)` (wasm-gc only). The escape fat record is **not the original `[fnIndex,envPtr]`-over-`rc.si` linear-memory record** — it is an engine-traced GC struct, which is why the leak is dissolved.

### Option B — Uniform boxed closures everywhere (one `[fnIndex,envPtr]`-over-Rc, indirection + heap-alloc + refcount on every call; the `std::function`/Swift/Go funcval model). **REJECTED as the floor (was adopted in the original ADR).**

One shape, no second case, every target. **Pros (why the original ADR adopted it):** truly uniform; solves the escaping FFI callback directly; storable heterogeneously; simplest to specify. **Cons (why the multi-mode steer rejects it):** it taxes **native** — which has *no host to escape to* — with a `std::function` 3–5× indirection, a heap-allocated env even for the empty-env case `@fnref` already handles for free, refcount traffic on every share, **and an uncollectable leak** (the `rc_drop`-doesn't-free caveat) for a capability native can never exercise. It also forces a *second* escaping-lifetime story (Rc on mvp vs engine-GC on gc) to document and test. **Disposition:** the empirical finding that Bun/JSC runs wasm-gc default-on removes the only reason to keep B's Rc tier alive on the managed target — so B is dropped entirely. Non-capturing still degenerates to `@fnref`, but capturing closures are **unboxed (non-escaping, all modes)** or **engine-GC'd managed structs (escaping, wasm-gc)**, never B's uniform Rc box.

### Option C — Defunctionalized "explicit context" binders (Zig's `*anyopaque ctx`). **Rejected as the general closure; its trampoline/binder SHAPE survives, re-pointed at `externref`.**

A boundary callback as three first-order things (funcref index, user-built ctx, host token) traveling by convention. **Cons (decisive):** no capture sugar (does not unblock `map`/`filter`/`fold`); manual untyped ctx threading (the Zig tax); no heterogeneous storage. **Why its shape is still kept:** the `@on`/`@off`/`@export_callback`/exported-invoke-trampoline *machinery* is exactly the wasm-gc boundary surface C2 needs — but the trampoline now passes the closure as **one `externref`** (the engine-traced `$Clo`), and the compiler (not the user) synthesizes and owns `$Env`. C's visible-lifetime, typed discipline **without** C's no-capture / manual-ctx tax, and without C's user-owned ctx pointer.

### Option D — Mode-tiered closures: zero-cost non-escaping on every mode; engine-GC'd escaping ONLY where a host exists (wasm-gc). **CHOSEN — the frame this revision adopts.**

The cut is driven by one fact: the escaping-to-host case only exists where a JS/Bun host owns a registry, and that is web/bun = exactly where wasm-gc lives. So native/`wasm-mvp` get only the non-escaping tier (Option A unboxed, the cheapest thing, no dead weight), and the escaping tier is the wasm-gc managed struct (Option A's escape form, engine-traced). **Pros:** native pays strictly *less* than B's floor and never carries a leak; the escaping case is reclaimed by the only allocator that can (engine GC), so the leak caveat and the retain-cycle weakness are both **deleted**; capability-coherent with ADR 0009's existing target split (same E-code gate mechanism as E0012/E0013); reuses shipped machinery (struct/externref/funcref emit paths, the JSString externref boundary precedent); reaches the 100% web **and** bun gate with no JSPI dependency (verified). **Cons:** two representations land in v1.0 (not deferred); the escape/host-reachability classifier is now load-bearing for *soundness* and must ship conservative until ADR 0011 R4; `wasm-mvp`/`--native` cannot host-register callbacks at all (real for the no-GC-JS-ish-host niche, illusory for true native); the wasm-gc whole-program constraint means "arenas + host callbacks in one module" is not expressible. **Verdict (incorporating the adversarial review):** the `REGIONS-MIDDLE` Tier-2 (leak-free intra-program escape on native via `&move_to_parent_arena`) does **not** move the gate and depends on ADR 0008 v1.1 deep-promotion — correctly scoped **post-v1.0 (C5)**. The two-tier model — non-escaping (all modes) + escaping host-callable (wasm-gc) — **is sufficient to reach 100% on both web and bun.**

**Why this combination.** The gate forces a storable, escapable, host-callable *captured* callback — and that callback only exists on a JS host, which only exists on wasm-gc-capable web/bun. So the escaping representation goes where the host (and the GC that reclaims it) is; native gets the zero-cost non-escaping closures it can actually use. We decline Zig's no-closure stance (rcaps supply the lifetime proof), decline B's uniform-box-everywhere floor (it taxes native for nothing and resurrects a leak), and decline whole-program defunctionalization (it breaks separate compilation / CaaS) — keeping defunctionalization only as the empty-env `@fnref` special case Silicon already ships.

## Consequences

- **Positive:** unblocks ADR 0018 Phases 3–4 (the 100% gate, leak-free on web+bun via engine GC) and ADR 0016's deferred combinators on **every mode** (the non-escaping unboxed path is not gated on wasm-gc); **dissolves** the original ADR's "escaping-to-host leaks until the free-list lands" caveat and the host↔closure retain-cycle weakness (the engine reclaims, incl. cycles); native ships **faster** closures than the old uniform-boxed floor and carries **no** dead-weight leak; reuses the `@fnref` table, HM-lite `Function`, the loopDesugar AST→AST pattern, and the shipped struct/externref/funcref emit paths; no grammar change, no new IR node; the escaping lifetime story is now **single** (engine GC), not target-split.
- **Negative:** the largest single v1.0 item, on the critical path; **two representations and the escape/host-reachability classifier are now in v1.0** (the classifier is load-bearing for soundness and ships conservative — may false-reject some safe `wasm-mvp`-host programs — until ADR 0011 R4); `wasm-mvp`/`--native` cannot host-register a captured callback (real only for the shrinking no-GC-JS-ish-host niche; illusory for true native); the wasm-gc whole-program constraint forbids mixing a linear-memory registry with host-callable closures in one module; mutable capture still waits on the borrow checker on every mode; generalizing `__fn_i_i` touches hot lowering paths and must preserve byte-equal codegen.
- **Follow-up work:** C0–C3 (v1.0-critical) and C5 (post-v1.0 regions tier for intra-program native escape, gated on ADR 0008 v1.1 deep-promotion) in §8; ADR 0011 R4 escape analysis to *loosen* the conservative classifier (let provably-non-retained `@extern` callbacks stay on `wasm-mvp`); the mode-gate E-code (sibling to E0012/E0013); `call_ref` graduation off the v1.0 `call_indirect` table route on wasm-gc. **No longer needed:** the `rc.si` keep-alive token, the real `__closure_release` body, the drop-without-`@off` leak E-code, `weak` registry handles, and the free-list dependency *on the closure path*.
