Sigil Bootstrap Plan

Self-hosting the Silicon compiler · Target: WebAssembly (WASIX) executed by Wasmer · Stage 0: the current TypeScript/Bun compiler · Stage N: Silicon compiling Silicon

1. Executive Summary

Goal. Rewrite the Sigil compiler in Silicon itself, compile it with the existing TypeScript-on-Bun compiler, and have the resulting sigil.wasm module — running under Wasmer with the WASIX runtime — compile any Silicon program (including its own source) to a byte-identical main.wat.

Strategy. Bring the compiler up one phase at a time, starting with the parser. Each phase ships as a stand-alone Silicon module whose output the TypeScript compiler can still cross-check. A phase is "done" only when its Silicon implementation produces the same AST/IR/WAT as the TypeScript reference for every fixture in the test corpus.

Acceptance. The bootstrap is complete when sigil-tsN(src) == sigil-siN(src) for the full fixture corpus and sigil-siN.wasm reproduces itself bit-for-bit (the classic stage-2 / stage-3 fixed-point check).

Why this is achievable for Silicon. Silicon's grammar is fixed and small, the elaboration model is data-driven (strata), and the compiler's "interesting" logic already lives in .si files. The bootstrap is therefore mostly about porting six relatively independent passes — not redesigning the language.

2.2 The Triple Test

For each commit on the bootstrap branch the CI runs:

  1. sigil-ts0 boot/*.si → sigil-si1.wasm
  2. wasmer run sigil-si1.wasm -- boot/*.si → sigil-si2.wasm
  3. wasmer run sigil-si2.wasm -- boot/*.si → sigil-si3.wasm
  4. sha256(sigil-si2.wasm) == sha256(sigil-si3.wasm) — fixed point reached
  5. For every fixture f.si in tests/corpus/: sigil-ts0 f.si == wasmer run sigil-si2.wasm -- f.si

2.3 What "Phase" Means in This Plan

A phase is one compiler subsystem ported to Silicon. The TypeScript reference for that subsystem stays in place until the Silicon port passes a structural-equivalence test, after which the TypeScript copy becomes a debugging fallback (--ts-fallback <phase>) and is eventually deleted.

3. The Four Grammatical Forms

Silicon has four grammar rules. The Ohm file (src/grammar/silicon-official.ohm) lists ~30 productions, but they are nearly all interior structure or wrapper layers introduced for Ohm's pattern-match conveniences. Semantically, every Silicon program is built from exactly four top-level forms, each picked by a single leading sigil. The bootstrap parser is organised around these four forms, not the 30 Ohm productions.

3.1 The Four Forms

#FormLead sigilSurface syntaxExample
1Definition@@kw name [params] [: type] [:= body];@global add a, b := a + b;
2Annotation@@@@name [args] — decorates the next form@@inline @fn fast := ...;
3Function call& · && · \&name [args] (&& = comptime, \ = lambda invocation)&print 'hello'
4Binary expressionnoneexpr op expr (op expr)* — flat, no precedencea + b * c == 30

3.2 What Is Not a New Form (Interior Structure)

Everything else in the Ohm grammar is interior structure used within the four forms. None of it appears at the top of a parse:

  • Literals: 'string', 42, 3.14, $[1,2,3], ${k=v}, $(a,b), @true/@false
  • Namespaces: a.b, a::b
  • Blocks: { item; item; trailing } — recursively a sequence of the four forms
  • Type annotations: : TypeName
  • Generic params: [T, U]
  • Doc comments: ## line

The Ohm productions Element, Item, Statement, ExpressionStart, ExpressionEnd are wrapper layers that exist for Ohm's semantic-action API. They carry no meaning of their own — every parse tree node of one of those kinds has exactly one child of one of the four forms. The bootstrap parser does not reproduce them.

3.3 Why This Matters for the Bootstrap

  • Five-way top-level dispatch, not 30. When the parser peeks the first token of an element, it sees one of five lead sigils (@@, @, &&, &, \) or falls through to a binary-expression parse. That's the entire top-level grammar.
  • The Ohm wrappers are dead weight. Re-implementing Element → Item → Statement → Definition in Silicon "for fidelity" inflates the parser, the AST, and every later phase that has to walk those nodes. The bootstrap emits AST nodes for the four forms directly.
  • Guardrail against language growth. A proposed feature that requires a fifth top-level form is changing Silicon's surface — not just adding sugar. New features fit inside one of the four; usually as a new @kw definition or a new @@name annotation.
  • Strata fit naturally inside the four. @stratum_keyword and @stratum_operator are definitions (form 1); their bodies are sequences of function calls (form 3) and binary expressions (form 4). The mod architecture in §7 needs no new grammatical category.
  • Stage 1 only needs forms 1, 3 (just &), and 4. Annotations (@@), comptime calls (&&), and lambdas (\) are post-Stage-3 — the bootstrap compiler doesn't even tokenise them.

3.4 Bootstrap Status of Each Form

FormStage 0 TodayBootstrap Phase

3.5 Sketch: the Parser's Top-Level Dispatch

@fn parseElement -> Int := {
  @local t := peek;
  # Five lead sigils — one of four forms; default is binary expression.
  @if t == TOK_AT_AT      { @return parseAnnotation;     };  # form 2
  @if t == TOK_AT         { @return parseDefinition;     };  # form 1
  @if t == TOK_AMP_AMP    { @return parseComptimeCall;   };  # form 3 (comptime)
  @if t == TOK_AMP        { @return parseCall;           };  # form 3
  @if t == TOK_BACKSLASH  { @return parseLambda;         };  # form 3 (lambda)
  parseBinaryExpression;                                                # form 4 (default)
};

Six lines cover every top-level position in Silicon. Recursion under each of those parsers is bounded by a small fixed set of terminators (;, ), }, ,, end-of-input), which is what keeps the parser at ~600 LoC.

Drift warning for Phase 1. When the parser implementer starts work, the temptation will be to mirror Ohm's Element → Item → Statement → Definition → ExpressionStart → ExpressionEnd hierarchy in Silicon "for fidelity." Do not. Those wrappers exist for Ohm's pattern-match API, not for Silicon. The bootstrap parser produces exactly four AST root kinds from parseElement, matching the four forms above.

6. The Silicon-Core Feature Budget

Silicon is large. The bootstrap compiler must not use every feature — only a stable subset (Silicon-Core) that the existing Stage 0 compiler already handles correctly. This freezes the bootstrap source against churn in the rest of the language.

6.1 Already in Stage 0 (no new work)

An audit of src/ir/, src/intrinsics/, and src/codegen/std.wat confirms the following are already supported and the bootstrap can rely on them as-is:

  • Memory intrinsics — WASM::i32_load, i32_store, i32_load8_u, i32_load8_s, i32_store8, f32_load, f32_store, memory.grow, memory.size all callable from Silicon (intrinsics.ts:94–106).
  • String literal interning — string literals lower to length-prefixed UTF-16 LE data segments via allocString (lower.ts:79); IRDataSegment emission works (emit.ts:67).
  • Heap runtime — $alloc bump allocator and $str_concat already live in std.wat and are inlined into every module.
  • Definition kinds — @global, @fn, @local, @local, @extern, @export all lower correctly via existing strata.
  • Expressions — arithmetic, comparison, bitwise (signed i32 + f32), @if, @match on values, @loop, @break, @continue, @return, blocks, function calls, intrinsics.

6.2 Stage 0 Gaps That Block the Bootstrap

These features are required by the self-hosted compiler but Stage 0 does not yet provide them. Each must land before Phase −1 completes.

6.3 Survivable, Don't-Fix-Yet

  • No function tables / call_indirect — strata dispatch in the bootstrap is a giant @match on integer tag. Slower than a table lookup; fine for Stage 1.
  • No generics — every container is monomorphic vec<i32>; we cast through i32 handles.
  • No ARC/GC — arena allocator + free-at-end-of-pass.
  • @type_alias, @type_distinct declared but no-op today — readability loss only.
  • Coroutines, effects, @try — post-Stage 3.
  • Annotations (@@) — post-Stage-3. Bootstrap compiler does not tokenise @@; once self-hosted, annotations land as a normal Silicon-side feature addition (parser, elaborator, Metadata-Strata registration). Keeping them out of the bootstrap also keeps the fixed-point invariant safe from macro-expansion non-determinism.

7. Strata as Mods — The Open Extension Surface

Strata are not just a dispatch table for operators and keywords. The Silicon spec (docs/strata.md) positions strata as the language's mod system — third-party packages add new types, new type systems, new IR passes, custom codegen backends, embedded DSLs, and quote/unquote macros. Think of them as the way Doom WADs extend the engine without recompiling it.

This shapes the bootstrap from day one. Even though Stage 1 ships with only built-in strata filling every extension point, the architecture must be registry-driven everywhere so that user strata can plug in later without a compiler rewrite.

7.2 Architectural Requirements the Bootstrap Must Honour

Three concrete requirements fall out of the "strata as mods" framing. Skipping any of them means the post-Stage-3 capabilities require a compiler rewrite.

R1 — The IR is open-tagged, not a closed @type_sum

Stage 0 today encodes IR node kinds as a TypeScript discriminated union ('Const' | 'BinOp' | 'If' | … in ir/nodes.ts). The bootstrap cannot follow that pattern: a Codegen-Stratum installed by a user must be able to register a new IR node kind (e.g. 'VectorAdd', 'AtomicCmpXchg') without forking the compiler.

  • IR node kinds are strings registered into the strata registry at load time, each carrying a printer hook and a lowering hook.
  • The Silicon-side representation is still (tag:i32, …fields) records in the arena, but tag is a registry-allocated integer, not a closed enum.
  • Emit-phase dispatch is "look up the printer registered for this tag," not a hard-coded switch.

R2 — Every phase is a registered middleware list

Phases 4–7 stop being monolithic functions; they become ordered pipelines whose stages are contributed by strata.

PhasePipeline shapeStage 1 contents
1 — ParserCore parser + zero or more parse_dsl_region hooks keyed on (open-token, close-token)Core only
4 — ElaborateWalker + operator_expanders[], defkind_expanders[], metadata_expanders[]Built-in strata only
5 — Type CheckGeneric constraint engine fed by type_strata[] and constraint_strata[]One built-in type stratum supplying Int, Float, Bool, sum types
6 — IR LowerWalker + codegen_strata[] per AST kindBuilt-ins only
6.5 — Optimise(new) Ordered list of ir_pass[] middlewareEmpty in Stage 1; const-fold lands post-Stage-3
7 — EmitBackend selected from codegen_backends[] by target nameOne backend: wat

Concretely: the Phase 6 lowerer is not a giant @match on AST tag with a default arm. It is a loop over codegen_strata, each stratum claiming the AST tags it handles. The default "no stratum claimed this" case is a compile error, surfaced with the same diagnostic regardless of whether the missing tag is built-in or user-defined.

R3 — The strata registry is the spine of the toolchain

Phase 3 stops being "load defkinds.si into a dict." It becomes the central registry consulted by every later phase. Two implications for the bootstrap:

  • Two-pass loading. The bootstrap parses just enough of the source to find @use 'stratum.si' declarations and built-in strata bundles; then it loads them; then it re-enters the main parse with any DSL/macro hooks now registered. Stage 1 skips the re-entry (no DSL strata) but the entry point exists.
  • Capability stub. Each registration carries a required_caps: i32 bitmask checked against the installing stratum's granted caps. In Stage 1 every cap is granted; the check is a no-op. Post-Stage-3 it becomes load-bearing.

7.3 Macros, Quote / Unquote, and Compile-Time Evaluation

The Silicon spec (v1-features.md) lists quote/unquote macros ('expr') as a v1 feature. The bootstrap acquires this almost for free:

  • The strata body interpreter built in Phase 4 is a compile-time Silicon evaluator.
  • Generalising it from "evaluate a strata body" to "evaluate any Silicon expression with the @comp / quote marker" is a small extension, deferred until the Sulfur (dynamic superset) work begins.
  • Bootstrap requirement: the body interpreter must be a standalone module, not coupled to strata loading.

7.4 What This Doesn't Change About the Plan

The phase ordering, gates, and corpus structure stay the same. What changes is the internal shape of Phases 3–7: registry-driven dispatch instead of hard-coded switches; open IR tags instead of closed unions; capability stubs at every registration site. The phase deliverables (§8 below) are updated to call out the registry-driven shape.

1.1 Why a single-pass predictive parser is enough

Silicon was designed to be trivially parseable. Three properties make a hand-written predictive parser the right tool:

  • No operator precedence. Binary expressions are a flat left-to-right chain (ExpressionStart = ExpressionEnd (BinOp ExpressionEnd)*). No Pratt parser, no climb tables.
  • Sigil-led tokens. Every interesting construct starts with a unique sigil: @ for keywords and definitions, & for calls, $ for literals, ' for strings, digits for numbers. One token of lookahead picks the right rule every time.
  • Fixed, small grammar. ~25 non-terminals; no left recursion in any rule that can't be re-expressed as a loop.

So the bootstrap uses a straight-line LL(1) parser — no backtracking, no packrat memo, no codegen. Each rule is one Silicon function that reads tokens and emits one AST node. Total: ~600 LoC of Silicon, no library dependencies.

Ohm.js is scaffolding, not spec. The Stage 0 CST shape is an implementation detail of how Ohm happens to encode parses. The bootstrap is free to skip the CST layer entirely and produce AST nodes directly from the token stream. The normative output of Phase 1 is "the AST that Phase 2 expects" — nothing more.

1.2 Lexer

Whitespace-skipping with #-line comments. Tokens are (kind, source_offset, source_length) triples in a flat vec<i32> — no token-string allocation. Kinds:

NUM_DEC, NUM_HEX, NUM_BIN, NUM_OCT, NUM_FLOAT,
STRING, IDENT, NAMESPACE_SEP,                # '::' or '.'
KEYWORD,                                     # '@xxx'
SIGIL_AMP, SIGIL_DOLLAR, SIGIL_AT, SIGIL_BACKSLASH,

1.3 Parser

One function per grammar production. Each function consumes tokens from a shared cursor and returns the index of an AST node (or -1 on miss). Iteration is an explicit @loop; "choice" is a one-token peek followed by a dispatch.

@fn parseProgram -> Int := {
  @local elements:Int := vec_new;
  @loop {
    @if peek == TOK_EOF { @break; };
    @local elt:Int := parseElement;
    vec_push elements, elt;
  };
  ast_make AST_PROGRAM, elements;
};

@fn parseExpressionStart -> Int := {
  # No precedence — left-fold a flat chain of BinOps.
  @local lhs:Int := parseExpressionEnd;
  @loop {
    @if peek != TOK_OPERATOR { @break; };
    @local op  := take;
    @local rhs := parseExpressionEnd;
    lhs := ast_make_binop op, lhs, rhs;
  };
  lhs;
};
Gate   Every fixture in tests/corpus/parser/ parses without error and the Phase 2 gate (AST JSON equivalence with Stage 0) passes when fed the Silicon parser's output. There is no separate CST-equivalence check.
Risk: Empty/lone/trailing-separator cases in ListOf<X, sep>-style rules. Mitigation: add targeted fixtures for empty arg lists, single-arg lists, trailing commas, and empty blocks before tackling full programs.

Inputs

9. Cross-Cutting Concerns

9.1 Determinism

Every byte of sigil.wasm's output must be a pure function of the input. Anything non-deterministic — iteration order, pointer values, timestamps, RNG, file-system order — breaks the stage-N=stage-N+1 invariant.

  • All hash maps iterate in insertion order.
  • All glob-style directory reads sort their results alphabetically.
  • No clock_time_get inside any compile phase (only the CLI's "wall time" log line at the end).
  • The arena allocator must reset to the same base address on every run — never seed from argv.

9.2 Embedding Data (strata, std.wat, grammar)

Stage 0 reads src/strata/*.si, std.wat, and silicon-official.ohm from disk. Stage 1 cannot do that without leaking the install path. Instead, the build embeds these files as WAT (data ...) segments at known offsets:

# Generated at Stage 0 build time (one-shot script: scripts/embed-stdlib.ts):
(data (i32.const 0x0010_0000) "# operators.si\n@stratum_operator Plus (...)\n...")
(data (i32.const 0x0014_0000) "# defkinds.si\n...")
(data (i32.const 0x0020_0000) "(module ...)")  # std.wat

A tiny embedded::open(filename) function returns the right (ptr,len) for the requested logical name. Once Phase 1 is self-hosted the grammar file is no longer needed (the parser is handwritten); only the strata files and std.wat stay embedded.

9.3 Error Reporting Parity

Stage 0's error format must be preserved character-for-character so that test fixtures don't have to be rewritten. Concretely: line/column numbers, the formatTypeError templates, and the ANSI color sequences in sigil_cli.ts.

9.6 Memory Model

Layout

  • 0x0000_0000..0x0010_0000 — runtime globals + stack
  • 0x0010_0000..0x0030_0000 — embedded read-only data (strata, std.wat)
  • 0x0030_0000..0x0F00_0000 — arena heap (grows by memory.grow)
  • 0x0F00_0000.. — output buffer (WAT being built)

Lifecycle

  1. argv parsed; source read into arena.
  2. Lex → parse → AST in arena.
  3. Elab/typecheck/lower in arena.
  4. Emit WAT to output buffer.
  5. WASIX fd_write output buffer to main.wat.
  6. arena_reset & proc_exit.

12. Definition of Done

  • Self-host: wasmer run sigil-si2.wasm -- boot/main.si emits a sigil-si3.wasm identical to itself.
  • Coverage: the full tests/corpus/ tree produces identical artifacts under Stage 0 and Stage 2.
  • End user: wasmer run natescode/sigil -- compile main.si compiles a fresh Silicon project end-to-end with no Bun, no Node, no Ohm.js, no TypeScript on the user's machine.
  • Docs: README.md install instructions reduced to "install Wasmer, run wasmer install natescode/sigil".
  • Stage 0 retained in legacy/ts-stage0/ for one release cycle as the reference implementation; CI keeps building it.
  • Open-extension fixtures green — the three "test stratum" fixtures from the Phase 3, 5, and 6 gates pass:
    • Registering a new IR kind 'TestKind' round-trips through emit (Phase 3).
    • A test Type-Stratum adding BigInt type-checks an @global x:BigInt := 5 program (Phase 5).
    • A test Codegen-Stratum overriding 'BinOp' changes lowering output (Phase 6).
    Together these prove that none of the nine StrataTypes is hard-coded into the bootstrap — the architecture is mod-ready even if only built-ins are loaded.

13. Open Questions

  1. silicon-official.ohm was Stage 0 scaffolding only — the Silicon parser is now normative. Should the .ohm file be deleted, kept as informal documentation, or rewritten as a reference grammar in a separate doc?
  2. Where do compiled main.wasm binaries get produced — embed binaryen in WAT (huge), shell out to wat2wasm from a launcher script (current plan), or finish a pure-Silicon WAT→binary encoder post-bootstrap?
  3. Is the WASIX dependency acceptable for end users on Windows? (Wasmer ships first-class Windows binaries — yes — but verify on CI.)
  4. Do we ship a separate sigil-debug.wasm with the TS-stage0 cross-check baked in, or keep that strictly a dev-time tool?
  5. Will Stage 1 support the full --strata CLI flag, or is it baked-strata-only until Stage 3?
