Silicon — External Tooling Adoption

Roadmap for getting Silicon (.si) recognized by external developer tooling — line counters, syntax highlighters, editor grammars, language servers, and finally GitHub Linguist. Each phase is roughly ordered by effort-to-reach ratio. Date: 2026-05-24.

Status note (post-boot/ removal): This document was authored when the Silicon-in-Silicon bootstrap lived under boot/. That tree has been removed and the self-hosted compiler is slated for a future rewrite. References to boot/* paths describe the historical layout. The TypeScript compiler in src/ is the current production compiler.
Strategy. Two grammars do most of the heavy lifting: a TextMate grammar (Phase 2) fans out to VS Code, Sublime, shiki, bat, and eventually Linguist; a tree-sitter grammar (Phase 3) fans out to Neovim, Helix, Zed, Emacs 29+, and GitHub code search. Both should land before the long tail of one-off highlighter PRs (Phase 4), because several of those tools (shiki, chroma) can consume one of the two grammars directly.

Phase 5 (LSP) is already on the roadmap as docs/language-server-plan.html; this doc only tracks the editor-integration side of it.
Sizing. S = 1–2h focused session · M = 3–5h session · L = 1 day · XL = 2+ days.

Tracking. To mark a task done, add complete to the <details> class list and append <span class="done">✅</span> to the summary. Use partial + 🟡 for in-progress, blocked + ⛔ for blocked.

Phase 1 — Line counters

Easiest wins. Each tool needs a single JSON/Go entry describing Silicon's extension (.si), comment markers (# line comments, no block comments based on boot/), and string delimiters. Target: all four done in one afternoon.
LC-1 tokei — submit language entry S, ~1h

Get Silicon recognized by tokei, the Rust-based code counter used by many CI pipelines and the cargo-license ecosystem.

Acceptance:
  • PR opened against XAMPPRocky/tokei adding a Silicon entry to languages.json.
  • Entry fields: name: "Silicon", line_comment: ["#"], extensions: ["si"], quotes: [["\"", "\""]].
  • Sample .si file committed to tests/data/ with expected counts in tests/data/silicon.toml.
  • PR linked from docs/language-tooling.html.
Subtasks:
  1. Fork tokei, add the Silicon entry alphabetically in languages.json.
  2. Copy a small boot/tests/*_test.si as the fixture.
  3. Run cargo test locally to verify the new fixture.
  4. Open PR with title Add Silicon language support.
LC-2 scc — submit language entry S, ~1h

scc (Go-based, parallelizes across files) is the fastest counter and used in many enterprise CI setups.

Acceptance:
  • PR opened against boyter/scc adding Silicon to languages.json.
  • Same shape as the tokei entry: # line comments, no block comments, double-quote strings.
  • go generate ./... regenerates the language database without errors.
Depends on: none. Can be done in parallel with LC-1.
LC-3 gocloc — submit language entry S, ~1h

gocloc is a popular Go port of cloc, used by gitstats and several reporting tools.

Acceptance:
  • PR opened against hhatto/gocloc adding Silicon to the languages.go map.
  • Test added to gocloc_test.go with a small Silicon fixture.
LC-4 cloc — ship --read-lang-def file in-repo, optional upstream PR S, ~1h

cloc is Perl-based with a single maintainer (AlDanial) and a slow review cycle. Ship a definition file in the Sigil repo first so users can opt in immediately; upstream PR is a stretch goal.

Acceptance:
  • scripts/cloc-silicon.txt committed to the repo. Format: 
    Silicon
    filter remove_matches ^\s*#
    extension si
    3rd_gen_scale 1.00
  • README snippet showing cloc --read-lang-def=scripts/cloc-silicon.txt ..
  • Stretch: PR to AlDanial/cloc with the same definition baked in.

Phase 2 — TextMate grammar + immediate consumers

Writing one silicon.tmLanguage.json unlocks a long tail of downstream tools. This is the highest-leverage single artifact in the whole roadmap. Output: silicon.tmLanguage.json + a minimal VS Code extension wrapping it.
TM-1 Author silicon.tmLanguage.json L, ~1 day

Hand-write a TextMate grammar covering Silicon's lexical surface. Use boot/parser/lex.si as the source of truth for token categories.

Acceptance:
  • Grammar covers: comments (#), strings ("…" with escapes), numeric literals (int + float), keywords (@fn, @if, @loop, @match, @type, @struct, @enum, @local, @defer, …), built-in types (Int, Int32, Int64, Float, Bool, String, u8…u64), operators, the $Variant sigil, the &keyword call prefix, and the :: path separator.
  • Tested visually in VS Code via "Developer: Inspect Editor Tokens and Scopes" against every file in boot/tests/.
  • Grammar committed to tooling/textmate/silicon.tmLanguage.json in the Sigil repo (new directory).
  • Accompanying language-configuration.json for VS Code (bracket pairs, comment toggle, auto-closing pairs).
Subtasks:
  1. Enumerate keyword set from boot/strata/builtin/*.si and the elaborator.
  2. Write the JSON grammar with named scopes following TextMate conventions (keyword.control.silicon, storage.type.silicon, etc.).
  3. Manual visual sweep of boot/tests/ in VS Code.
  4. Commit grammar + language-configuration.
TM-2 VS Code extension — publish to marketplace M, ~4h

Package TM-1 as a standalone VS Code extension so users can install Silicon support with one click. This becomes the primary install path until the LSP ships.

Acceptance:
  • Extension scaffold under tooling/vscode-silicon/ with package.json declaring the language contribution.
  • Includes TM-1's grammar + language config; no LSP yet.
  • Published to the VS Code Marketplace under publisher natescode (or org).
  • README links to the Sigil repo and shows a screenshot of highlighted code.
  • CI workflow that runs vsce package on tag pushes.
Depends on: TM-1.
TM-3 shiki — submit grammar S, ~2h

shiki uses TextMate grammars directly and ships with VS Code, Astro Starlight, Docusaurus v3, and most modern documentation sites. One PR, very wide reach.

Acceptance:
  • PR against shikijs/textmate-grammars-themes or shikijs/shiki (check current location) adding silicon to the language list.
  • Grammar file copied from TM-1, scope name source.silicon.
  • Sample snippet in the shiki test fixtures.
Depends on: TM-1.
TM-4 bat — submit sublime-syntax conversion M, ~3h

bat (Rust cat replacement) ships sublime-syntax grammars, which are a close cousin of TextMate. Convert TM-1 with sublime-syntax-from-tmlanguage or hand-port.

Acceptance:
  • PR against sharkdp/bat adding Silicon.sublime-syntax to its assets.
  • bat --list-languages includes Silicon after rebuild.
  • Manual smoke test on a Silicon source file.
Depends on: TM-1.
TM-5 Sublime Text — Package Control submission M, ~3h

Sublime Text users find packages via Package Control. Submitting registers Silicon as an installable package.

Acceptance:
  • Public repo sigil-lang/sublime-silicon with TM-1's grammar repackaged as a .sublime-package.
  • PR against wbond/package_control_channel adding the package.
Depends on: TM-1.

Phase 3 — Tree-sitter

Tree-sitter is the modern path for editor grammars: Neovim, Helix, Zed, Emacs 29+, GitHub code search, and Linguist's symbol indexer all consume tree-sitter parsers. Bigger lift than TM-1, but Silicon's grammar shape is already settled in boot/parser/parse.si, so this is mostly translation.
TS-1 Scaffold tree-sitter-silicon repo S, ~2h

Create a public repo with the standard tree-sitter layout: grammar.js, queries/, test/corpus/, package.json, GitHub Actions CI.

Acceptance:
  • Repo at github.com/sigil-lang/tree-sitter-silicon (or under natescode/).
  • tree-sitter generate && tree-sitter test runs (even if grammar is a stub).
  • CI green on Linux + macOS.
TS-2 Author grammar.js mirroring boot/parser/parse.si XL, 2–3 days

Translate the recursive-descent parser into a tree-sitter grammar. The structure is known — risk is in tree-sitter's LR(1) constraints around the keyword/operator forms.

Acceptance:
  • Grammar parses every file in boot/tests/ without errors.
  • Grammar parses boot/ itself end-to-end.
  • Conflicts resolved with explicit precedence rather than conflicts: arrays where possible.
Subtasks:
  1. Lexer-level rules: identifiers, numbers, strings, keywords, sigils.
  2. Expression precedence (binary ops, calls, member access).
  3. Definition forms: @fn, @type, @struct, @enum, @stratum.
  4. @match arms (both flat and arrow forms).
  5. Block expressions, @if/@loop/@defer.
Depends on: TS-1.
TS-3 Corpus tests covering boot/tests/ M, ~4h

Tree-sitter's corpus format lets us pin the parse tree of representative inputs. Use this as a regression net.

Acceptance:
  • Corpus file per major feature: expressions, definitions, types, match, strata.
  • Every boot/tests/*_test.si represented in at least one corpus case.
  • tree-sitter test green.
Depends on: TS-2.
TS-4 queries/highlights.scm M, ~3h

Syntax highlighting query. This is what Neovim/Helix/Zed render.

Acceptance:
  • queries/highlights.scm covers all token categories used in TM-1.
  • Visual check in Neovim with nvim-treesitter pointed at the local repo.
Depends on: TS-2.
TS-5 locals.scm, indents.scm, injections.scm M, ~3h

Editor-feature queries. locals drives rename + go-to-def in editors without LSP; indents drives auto-indent; injections lets us embed e.g. WAT inside @asm blocks if those ever land.

Acceptance:
  • All three query files present and validated by tree-sitter query.
  • Auto-indent works correctly in Neovim for definitions and block expressions.
Depends on: TS-4.
TS-6 Publish to npm + submit to nvim-treesitter S, ~2h

npm publish is the canonical distribution channel for tree-sitter grammars. nvim-treesitter is the largest single downstream consumer.

Acceptance:
  • tree-sitter-silicon published to npm.
  • PR against nvim-treesitter/nvim-treesitter adding Silicon to the parser list.
  • :TSInstall silicon works after the PR merges.
Depends on: TS-5.
TS-7 Helix — submit language config S, ~1h

Helix bundles tree-sitter grammars directly in its language config.

Acceptance:
  • PR against helix-editor/helix adding a [[language]] entry for Silicon pointing at TS-6's grammar.
  • Language config covers file types, comment token, indent settings.
Depends on: TS-6.
TS-8 Zed extension M, ~3h

Zed extensions are themselves small Rust crates that wrap a tree-sitter grammar.

Acceptance:
  • Extension repo created following the Zed extension template.
  • Submitted to zed-industries/extensions.
  • Installable from Zed's extension picker.
Depends on: TS-6.

Phase 4 — Standalone syntax highlighters

These tools don't consume TextMate or tree-sitter grammars directly — each needs its own bespoke lexer. All are small, independent PRs. Prioritize Pygments since the longest tail of doc tooling depends on it (Sphinx, MkDocs material, Rouge fallback paths).
SH-1 Pygments lexer M, ~4h

Write SiliconLexer in Python, submit to pygments/pygments. Used by Sphinx, MkDocs, Jupyter, and as a fallback by many other highlighters.

Acceptance:
  • pygments/lexers/silicon.py with a SiliconLexer class using RegexLexer.
  • Lexer registered in pygments/lexers/_mapping.py.
  • Example file added to tests/examplefiles/silicon/.
  • PR opened against pygments/pygments.
SH-2 highlight.js language S, ~2h

highlight.js is the default highlighter for Slack, Discord, many blogs, and Markdown renderers.

Acceptance:
  • src/languages/silicon.js contributed to highlightjs/highlight.js.
  • Sample in test/markup/silicon/.
  • PR merged or open.
SH-3 Prism.js component S, ~2h

Prism is the default highlighter for Docusaurus, Next.js docs, and many MDX setups.

Acceptance:
  • components/prism-silicon.js contributed to PrismJS/prism.

Phase 6 — GitHub Linguist

The capstone. Linguist drives GitHub's language breakdown bar, syntax highlighting in the web UI, and code-search symbol indexing. Linguist requires evidence of real-world usage before accepting new languages — historically ~200 repos using the extension, now closer to "sustained adoption signal." Park this phase until the rest of the ecosystem has produced enough adopters.
GL-1 Track public .si usage S, recurring

Linguist looks for breadth of usage. Maintain a running list of public repos using Silicon as evidence for GL-2.

Acceptance:
  • Living section in this doc (or a separate silicon-usage.md) listing public repos.
  • Quarterly GitHub code-search check: extension:si path:*.si.
GL-2 Linguist PR — language entry + grammar reference M, ~4h

Open the Linguist PR once usage signal is strong. Linguist will reference the tree-sitter grammar (TS-6) for syntax highlighting and indexing.

Acceptance:
  • PR against github-linguist/linguist adding Silicon to languages.yml.
  • Submodule pointer to TS-6's grammar repo at a tagged release.
  • samples/Silicon/ populated from representative boot/tests/ fixtures.
  • Evidence of adoption from GL-1 cited in the PR description.
  • Acknowledge in PR that the language is in active development with a stable extension.
Depends on: TS-6 (tree-sitter grammar published); GL-1 (usage signal).
GL-3 Post-merge: verify GitHub web UI rendering S, ~30min

Once Linguist ships in a GitHub deploy, verify Silicon files render with highlighting on github.com and that the repo language bar attributes correctly.

Acceptance:
  • Visual confirmation on a public Sigil repo.
  • Repo language bar shows Silicon at the expected percentage.
