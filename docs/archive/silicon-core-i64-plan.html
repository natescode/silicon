Silicon-Core i64 Support

Adding 64-bit integers as a first-class type in Silicon — just enough to declare and call WASI surfaces that use them (notably path_open's rights flags), without trying to model every conceivable i64 operation in one go.

Status note (post-boot/ removal): This document was authored when the Silicon-in-Silicon bootstrap lived under boot/. That tree has been removed; the work described below shipped against the historical bootstrap layout. The Int/Int32/Int64 surface it defines is current; the file paths under boot/* referenced below describe the prior structure.

scope unblocks Phase 4b four phases (A–D) no grammar changes byte-equal self-host preserved at every phase

1. Motivation

Today Silicon-Core has exactly two value types: Int (i32) and Float (f32). This works for every Silicon program written so far, but it fences off a real chunk of the WASI surface where the world speaks i64:

  • path_open — both rights-flag parameters (fs_rights_base, fs_rights_inheriting) are u64. Without i64, stage1 cannot open files by path. This is the gating blocker for Phase 4b of docs/sigil-wasm-bootstrap-plan.html.
  • fd_filestat_get, fd_seek, fd_tell, clock_time_get — sizes, offsets, and timestamps all use i64.
  • Process-id / signal numbers on richer Unix-style runtimes (Wasmer-WASIX) often expose i64 fields.

Over the last few sessions we've twice resisted the temptation to add a “WAT-prelude wrapper” hack that hardcodes i64.const -1 for the rights bits and exposes only i32 args to Silicon. It would have unblocked Phase 4b in a day, but it would have buried a permanent vestigial wart in emit_std_prelude — every program the compiler ever emits would carry an unused (import "wasi_snapshot_preview1" "path_open" ...) with i64 in its signature. And it wouldn't help the next i64 caller, or the one after that. The right time to do this properly is now.

An honest assessment of what we already have wrong

src/strata/modules/wasi_snapshot_preview1.si currently declares path_open with seven Int parameters. The two rights-flag positions are typed as i32 in Silicon but are actually i64 in the underlying WASI ABI. Stage 0 calls today either avoid this function entirely or pass 0 and rely on lucky alignment of i32-vs-i64 arg encoding (which only happens to work because we don't actually call it). Phase A of this plan is what makes those declarations honest.

2. Goals & Non-Goals

In Scope

  • Int64 recognised as a type by parser, typechecker, lowerer, emitter.
  • Integer literal suffix i64 — 0i64, 1234i64, 0xFFi64.
  • @extern foo:Int64 a:Int64; emits correct WAT i64 import.
  • Call-site argument passing: &foo 0i64 emits (i64.const 0).
  • i64 binops: +, -, *, /, %, ==, !=, <, >, <=, >= for two i64 operands.
  • Explicit casts: @toInt64 x (Int → Int64, signed extend), @toInt x (Int64 → Int, truncate).
  • Strata-driven dispatch for arithmetic operators by operand type.
  • Migration of existing bare @extern declarations in src/strata/modules/*.si to use Int64 where the WASI ABI does.
  • TS Stage 0 and bootstrap stay in sync at every phase.

Out of Scope

  • Unsigned types (no UInt64). WASI uses u64 but Silicon treats it as signed i64 at the type level — the bit pattern is what matters at the WASM call boundary, and i64 vs u64 only differ in how comparisons interpret the high bit.
  • i64 bitwise ops (|, &, ^, <<, >>) — deferrable; add when a caller needs them.
  • i64 ↔ Float casts — deferrable.
  • Implicit Int ↔ Int64 conversion — explicit casts only, no surprises.
  • Function overloading by operand type — strata-driven dispatch handles operators; user functions stay strict-signature.
  • i128 or arbitrary-precision — out, indefinitely.
  • WebAssembly extended v128 SIMD types — out.

3. Design Decisions

3.1 Type name — Int64

Matches the existing Int / Float naming pattern. Explicit about width. Reads cleanly in declarations: @global x:Int64, @extern fd_seek a:Int, b:Int64, c:Int. Alternatives considered and rejected:

NameVerdict
Int64Pick. Consistent with Int; explicit width.
LongReject — C-style ambiguity; size varies across platforms in everyday usage.
I64Reject — breaks PascalCase type convention (Int, Float, String, Bool).
i64Reject — lowercase type names already exist as low-level escape hatch (see parseTypeName's case 'i32': / case 'f32':); reserving i64 for that escape hatch is OK, but using it as the user-facing type would mix conventions.

The low-level escape hatch i64 in parseTypeName also gets added in Phase A — symmetrical with the existing i32 and f32 aliases.

3.2 Literal suffix — i64

# Decimal
@global a:Int64 := 0i64;
@global b:Int64 := 1234i64;

# Hex / binary / octal
@global mask:Int64 := 0xFFFFFFFFFFFFFFFFi64;
@global flags:Int64 := 0b1010i64;
@global perm:Int64 := 0c755i64;

# Negative
@global neg:Int64 := 0i64 - 1i64;     # reliable
@global neg2:Int64 := -1i64;          # IF unary-minus syntax lands

Suffix sits where the alphabet would otherwise be expected to start, which is exactly where the existing number lexer stops — tokenisation is a small surgical change, not a rewrite. No conflict with the e-style float exponent.

Plain integer literals without suffix keep their existing meaning (Int = i32). This means @global x := 5; is still i32 even if assigned to an Int64 variable — the typechecker raises a mismatch error. Explicit suffix required: @global x:Int64 := 5i64;.

3.3 Casts — @toInt64 and @toInt

@global a:Int   := 42;
@global b:Int64 := @toInt64 a;       # signed extend i32 -> i64
@global c:Int   := @toInt   b;       # truncate i64 -> i32 (wrap)

Implemented as strata keywords (per docs/strata.md — the same machine that powers @toInt / @toFloat today). WAT mapping: i64.extend_i32_s and i32.wrap_i64.

3.4 Mixed-type rules — strict, no implicit promotion

ExpressionResult
5 + 3Int (existing)
5i64 + 3i64Int64 (new in Phase C)
5 + 3i64Type error. Must @toInt64 5.
&f 5 where f declares :Int64Type error. Must pass 5i64.
@global x:Int64 := 5;Type error. Initializer is Int.

The rule: Int64 doesn't fall out of any arithmetic / call / initialisation unless every contributing operand is explicitly Int64. Mirrors the existing Int ↔ Float strictness in the typechecker — the language doesn't auto-coerce, and that's a feature.

Every line above is a real, small, surgical change. The biggest single change is the lexer surgery to accept the i64 suffix; everything else is “duplicate the existing i32 path with an i64 variant.”

5. Phased Rollout

Four phases, each independently shippable. Phases A and B (TS & bootstrap parity for declarations + literals) carry the bulk of the lift; Phase C lets users do actual arithmetic; Phase D is the payoff — Phase 4b of the bootstrap plan finally lands.

Big-int constant emission

The bootstrap's current ew_write_int formats an i32 as a decimal string. For an i64 literal that fits in i32 (e.g. 0i64, 1i64, 42i64), the same routine works. For the rights-flag constants we actually need in Phase D (-1i64 = 0xFFFFFFFFFFFFFFFF), we need either: (a) a Silicon-side big-int print helper, or (b) emit a multi-instruction sequence like (i64.const 0) (i64.const 1) (i64.sub) that the WAT parser folds. Option (b) avoids needing 64-bit arithmetic in ew_write_int at the cost of slightly uglier emitted WAT. Decision deferred to Phase B kickoff.

6. Bootstrap Capability Check

Two questions to settle before committing: can Stage 0 host this design today? Can the self-hosted compiler express the same semantics?

6.1 Stage 0 (TS host) — yes, with one grammar concession

  • have it Type system parameterised by SiliconType.kind — new kind 'Int64' drops in.
  • have it WasmValType union ('i32' | 'f32') — widen to include 'i64'.
  • have it Strata-driven operator dispatch — the same machine that picks +Float over + picks +Int64.
  • need it Grammar change for the literal suffix. Touches src/grammar/silicon-official.ohm's intLiteral rule. This is the one place CLAUDE.md says “ask first” — the change itself is one optional terminal.
  • need it parseTypeName case for 'Int64' + 'i64'.
  • need it Lower / emit small additions (handful of switch arms).

6.2 Bootstrap (Stage 1+) — yes, with the existing primitives

  • have it Hand-written lexer in boot/parser/lex.si — no parser-generator dependency, so the suffix change is a few extra peek-bytes after the digit run.
  • have it AST tag space in boot/parser/ast.si — next free integer constant slot.
  • have it IR record machinery (boot/ir/nodes.si) — ir_i64_const is a near-copy of ir_i32_const.
  • have it Strata loader / registry — PlusInt64 registers via the same path that PlusFloat does today.
  • need it Big-int decimal printer (ew_write_i64) — or use the multi-instruction emission workaround in the design note above.

6.3 The bootstrap chicken-and-egg

Phase B modifies boot/* to support Int64. Stage 1 (stage1.wasm) is compiled FROM that modified boot/*. But the OLD stage1.wasm — the one doing the compiling — doesn't yet know about Int64. Can the old compiler handle the new source? Yes — as long as the modified boot/* source itself doesn't use Int64. All of Phase B's changes are additive (new lexer branch, new AST tag, new IR constructor) that the existing bootstrap can parse + lower without invoking the new behaviour. The first source file that actually uses Int64 in boot/* lands in Phase D (the open_file_for_read wrapper). By then, stage1 itself understands Int64.

8. Risks & Open Questions

Risk: lexer surgery on a hot path

The number lexer in boot/parser/lex.si is exercised by every integer literal in every source file ever parsed. A bug in the suffix detection (e.g. accidentally consuming i from an identifier that immediately follows a number) would break a vast range of programs in subtle ways. Mitigation: tight test for “number followed by identifier” cases (5i, 5i6, 5i64x, 5if, 5 + foo); never advance past i unless the next two bytes are exactly 6 and 4 and the byte after is non-alphanumeric.

Risk: byte-equal self-host at each phase

Phases A and B are pure additions to existing code paths; no source in boot/* uses Int64 until Phase D. This means stage1 compiling itself should produce byte-identical WAT before and after the additions — the new code is reachable but never reached. Phase C adds strata for new operators; same property — new strata are registered but unused. Phase D is the first phase that actually exercises the new paths in boot/*, and that's where the byte-equal gate moves to a new (larger) size.

Risk: mixed-type semantics surprise

The decision “no implicit Int ↔ Int64 conversion” is the right one but means every existing call site that touches a now-Int64 extern parameter will error. The migration in §7 catches the WASI declarations themselves, but any third-party user program that was relying on Stage 0's lax i32 acceptance of i64 args will start failing. Mitigation: the existing test corpus exercises every WASI surface used in the bootstrap; no third-party programs use these yet.

Risk: grammar change touches the “ask first” surface

The intLiteral suffix is a real grammar addition. CLAUDE.md explicitly flags grammar changes as last-resort. The argument for: integer literal suffixes are a tiny, additive, non-ambiguous extension that other Silicon-shaped languages (Rust, Zig) also use; there's no alternative that doesn't introduce a different language surface (a magic identifier or call syntax). Bring this to the user before landing Phase A.

Open question: integer printing for i64 in the emitter

Decision needed in Phase B kickoff: real big-int printer in ew_write_i64, or the multi-instruction (i64.const 0) (i64.const 1) (i64.sub) workaround for values that don't fit in i32. The workaround is simpler but uglier; the big-int printer is a self-contained 30–50 lines of Silicon. Both work.

Open question: unary-minus syntax

Today there is no unary minus — -1 is “subtract 1 from missing operand,” a parse error. The convention is 0 - 1. Phase A could either inherit this constraint (0i64 - 1i64) or add unary minus. Unary minus is its own design slice — recommended: defer it; the 0i64 - 1i64 form works for everything Phase D needs.

9. What Lands First

Phase A is the right next move. Smallest slice that meaningfully changes what the language can express — once it lands, Silicon code can declare and pass i64 values, even if it can't yet add them or read them from disk. Phase A is also the only phase that touches the grammar; getting that out of the way early means Phases B–D are pure code surgery.

Recommended kickoff for the new session:

  1. Open this doc plus docs/sigil-wasm-bootstrap-plan.html (Phase 4).
  2. Ask the user about the grammar change in §6.1 — explicit go-ahead before touching silicon-official.ohm.
  3. Branch silicon-core/i64-support off main.
  4. Decide the open questions in §8 (big-int printer vs workaround; unary minus deferral).
  5. Execute Phase A. Self-host must stay byte-equal at 49,134 bytes (the post-Phase-4a size on main as of this writing).

10. What Phase 4b Looks Like After i64 Lands

Once Phase D of this plan completes, Phase 4b of the bootstrap plan is no longer blocked. The shape: 

# boot/std/io.si
@fn open_file_for_read:Int dir_fd:Int, path_ptr:Int, path_len:Int := {
  @local fd_out := &scratch_alloc 4;
  @local rc := &wasi_snapshot_preview1::path_open
    dir_fd, 0, path_ptr, path_len, 0,
    0i64, 0i64,                        # rights flags; 0 = inherit preopen rights
    0, fd_out;
  &@if (rc == 0),
    { &WASM::i32_load fd_out },
    { 0 - 1 }
};

# boot/cli.si — extended dispatch
@fn cli_dispatch := {
  ... existing argv walk ...
  # When a positional arg is seen, treat it as a source file path.
  &@if (positional_path != 0), {
    @local dir_fd := &find_preopen_dir;
    @local fd      := &open_file_for_read dir_fd, positional_path, positional_path_len;
    &@if (fd < 0), { &panic_stderr 'sigil: cannot open source file', 3; };
    SOURCE_FD = fd;                # global picked up by _start's read loop
  };
};

No exotic mechanism, no WAT prelude hack, no special-case bootstrap rule. Just normal Silicon code calling normal Silicon-typed externs. The language did the work.
