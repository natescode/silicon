Robust Strata Implementation Plan

Sigil Compiler · Architectural evolution toward a compiler-as-API model where Silicon Strata drive all lowering

1. Executive Summary

Problem: Several definition kinds — def_local, meta_export, meta_platform — are declared in Silicon strata (defkinds.si, metadata.si) but their actual lowering logic lives in hardcoded TypeScript functions inside lower.ts (lowerLocalDef, lowerExportDecl). The strata are therefore hollow: they name an IR intrinsic but cannot actually implement the lowering behavior themselves.

Goal: Expose compiler internals as a minimal, stable API callable from Silicon code inside strata. Every definition-kind lowerer, operator expander, and metadata handler should be expressible as pure Silicon — with zero TypeScript switch-case fallback for domain logic.

Principle: Sigil native code handles only: parse Silicon syntax, understand stratum syntax, expose IR intrinsics, expose compiler library APIs. Everything else is Silicon.

2.2 Definition-Kind Lowering Flow (Current)

How @local x: Int = 5 travels through the compiler today:

1 defkinds.si declares @stratum_keyword LocalDef ('@local', Node) = { &IR::def_local; }
↓
2 strataLoader.ts reads the Silicon file, calls codegenKindFromIntrinsic('IR::def_local') → 'local', registers DefKindEntry{ keyword:'@local', codegenKind:'local' }
↓
3 elaborator.ts stamps Definition.hook = 'local' on the AST node
↓
4 lower.ts — lowerDefinition() checks defExpanders.get('local') → not registered → falls through to switch(hook) { case 'local': return lowerLocalDef(...) }
↓
5 lowerLocalDef() (hardcoded TS): calls siliconTypeNameToWasm(), pushes to ctx.pendingLocals, sets ctx.locals → returns null
The gap: Step 1 looks like Silicon controls the lowering. It does not — the strata entry only registers the name of the hook. Steps 4–5 are hardcoded TypeScript. Silicon cannot express the logic of "resolve my type annotation, push a local declaration, record me in the locals map."

3. Target Architecture

3.1 Core Principle: Compiler as API

The compiler's native layer shrinks to four responsibilities:

  1. Parse Silicon syntax — grammar, tokenizer, AST construction
  2. Understand strata syntax and semantics — load, register, dispatch
  3. Expose IR intrinsics — the raw WebAssembly instruction set as named calls
  4. Expose compiler library APIs — type resolution, context mutation, IR node constructors

All definition lowering, operator expansion, metadata handling, and code generation strategy moves to Silicon strata.

3.3 How def_local Moves Into Silicon

After the change, lowerLocalDef in TypeScript is replaced by a Silicon strata definition that calls compiler APIs:

// src/strata/defkinds.si  (target state)

@stratum_keyword LocalDef ('@local', Node) = {
  // Resolve the Silicon type annotation to a WASM value type
  @local wasmType: WasmValType = $compiler.resolveType(Node.name.typeAnnotation);

  // Register the local in the function context (hoisted)
  $compiler.ctx.pendingLocals.push({ name: Node.name.name, wasmType });
  $compiler.ctx.locals.set(Node.name.name, wasmType);

  // Return null — the block lowerer emits the initialiser as a LocalSet
  &IR::null;
};

The strataLoader evaluates this body at registration time, building a closure over the compiler API surface. When lowerDefinition encounters hook='local' it calls the registered stratum body — the TypeScript switch case is gone.

3.4 Communication Flow in the Target Architecture

1 strataLoader.ts parses a .si file. It sees @stratum_keyword LocalDef with a body referencing $compiler.* API calls and &IR::* intrinsics
↓
2 strataLoader compiles the stratum body into an IRDefExpander — a TypeScript closure that captures the compiler API object and the IR node constructors
↓
3 The expander is registered: registry.defExpanders.set('local', compiledExpander)
↓
4 lower.ts — lowerDefinition() checks defExpanders.get('local') → found → calls expander.expand(node, name, ctx)
↓
5 The expander body runs — it calls CompilerAPI.resolveType(), mutates ctx.pendingLocals, mutates ctx.locals, returns null
↓
6 lower.ts switch has no case for 'local' — the expander handled everything. The switch becomes a last-resort error.

4. Compiler API Surface

These are all the native APIs Sigil must expose to Silicon strata. They are callable as $compiler.* in stratum bodies. The goal is a small, stable API that covers every use case currently hardcoded in TypeScript.

5. Silicon Strata in Target State

What the key strata look like after the architecture change:

5.1 def_local (LocalDef)

@stratum_keyword LocalDef ('@local', Node) = {
  @local wasmType = $compiler.resolveType(Node.name.typeAnnotation);
  $compiler.ctx.pendingLocals.push({ name: $compiler.watId(Node.name.name), wasmType });
  $compiler.ctx.locals.set($compiler.watId(Node.name.name), wasmType);
  &IR::null;
};

5.2 meta_export (ExportDecl)

@stratum_keyword ExportDecl ('@export', Node) = {
  @local name = $compiler.watId(Node.name.name);
  @local what = $compiler.isVarName(name) ? 'global' : 'func';
  $compiler.ir.makeExport(name, name, what);
};

5.3 meta_platform (PlatformDecl)

@stratum_keyword PlatformDecl ('@platform', Node) = {
  // Platform declarations are metadata only — no WAT emitted
  &IR::null;
};

5.4 def_function (LetDef / FnDef) — target

@stratum_keyword FnDef ('@fn', Node) = {
  @local name       = $compiler.watId(Node.name.name);
  @local params     = Node.params.map($compiler.lowerParam);
  @local returnType = $compiler.resolveType(Node.name.typeAnnotation) ?? 'void';
  @local body       = $compiler.lowerExpr(Node.binding.expression);
  $compiler.ir.makeFunction(name, params, returnType, [], body);
};

5.5 def_global (VarDef) — target

@stratum_keyword VarDef ('@local', Node) = {
  @local name     = $compiler.watId(Node.name.name);
  @local wasmType = $compiler.resolveType(Node.name.typeAnnotation) ?? 'i32';
  @local initExpr = $compiler.lowerExpr(Node.binding.expression);
  $compiler.ctx.globals.set(name, wasmType);
  $compiler.ctx.varNames.add(name);
  $compiler.ir.makeGlobal(name, wasmType, true, initExpr);
};

7. Architectural Trade-offs and Decisions

How are stratum bodies executed?

Option A: Interpret Silicon AST at registration time — strataLoader walks the stratum body AST and builds a JS closure. Simple, no extra compile step. Silicon in strata is a subset (no recursion, no heap allocation).

Option B: Compile strata to JavaScript — heavier machinery, enables full Silicon semantics in strata. Better long-term.

Recommendation: Option A for phases 1–4, Option B as a future phase.

How do strata reference each other?

Strata bodies may need to call helpers defined in other strata (e.g., a shared type-resolution helper). The registry should support $strata.call('HelperName', args) as a cross-stratum dispatch mechanism.

This defers cross-stratum calls until registration is complete, avoiding ordering issues.

What stays in the compiler forever?

  • Silicon grammar parsing (.ohm file and Ohm.js)
  • Stratum syntax parsing and registration
  • IR node types (nodes.ts) — the data model
  • WAT emission (emit.ts) — the output layer
  • The CompilerAPI object and its implementations
  • Type-checking pass (structural, not semantic)

What Silicon features are needed in strata?

Strata bodies use a constrained subset of Silicon:

  • @local variable declarations
  • Field access (Node.x.y)
  • .map(), .push() calls on collections
  • Ternary / @if for conditional logic
  • $compiler.* API calls
  • &IR::* intrinsic returns

No heap allocation, no recursion, no closures in strata bodies (at Phase A).

9. What the Compiler Reduces To

After all phases complete, the Sigil native compiler is responsible for exactly four things — everything else is Silicon:

1. Parse Silicon syntax
grammar/silicon-official.ohm + Ohm.js — fixed, never changes
2. Understand strata syntax and dispatch
elaborator/strataLoader.ts — loads .si files, compiles bodies to closures, registers in ElaboratorRegistry
3. Expose IR intrinsics
intrinsics/intrinsics.ts — the WASM instruction table; Silicon calls these as &IR::name or &WASM::name
4. Expose compiler library APIs
compiler-api/index.ts — CompilerAPI: type resolution, context mutation, IR node constructors, AST traversal

The IR node types (nodes.ts) and WAT emitter (emit.ts) remain native as they are the data format and output format, not semantic logic.
