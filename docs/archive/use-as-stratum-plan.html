@use as a Proper Language Construct

Promoting @use from a textual preprocessor pass into a first-class stratum-driven module system. Foundation for cross-file typechecking, real @export visibility, and a clean on-ramp for dynamic imports later.

Status note (post-boot/ removal): This document was authored when the Silicon-in-Silicon bootstrap lived under boot/. That tree has been removed. References to boot/* paths describe the historical layout; the design is still applicable to the TypeScript compiler in src/ and to the future self-hosted rewrite.

scope stratum-driven stage-0 → stage-1 requires one new CompilerAPI primitive grammar unchanged

1. Motivation

Today @use 'path.si'; is resolved by src/modules/useResolver.ts — a pre-parse string preprocessor that concatenates included files into one big source text. By the time the parser, elaborator, or typechecker runs, every @use directive has been erased. This was the right call for getting the bootstrap unstuck (docs/bootstrap-plan.html Phase −1.C), but it leaks in several places:

  • The LSP has to fake it. lsp-silicon/src/workspace.ts contains a hand-written stripUseDirectives regex plus two diagnostic-suppression filters (one for E0004, one for E0002 <unknown> mismatches). These suppressions hide false positives at the cost of also hiding real bugs.
  • @export is a no-op. Because everything ends up in one global namespace post-inline, marking something exported or not has no effect. The metadata stratum exists (src/strata/metadata.si) but the typechecker never consults it for visibility.
  • Diagnostics point at the wrong file. Errors raised against inlined content of std/vec.si appear with offsets in the concatenated buffer, not the source file the user is editing.
  • Incremental compilation has no hook. Each compile re-reads and re-inlines every transitive dependency from disk; there is no per-module cache the LSP or future build tooling can target.
  • The grammar already pretends @use exists. The directive looks like a Definition form, but the parser never actually sees one — it sees whitespace where the preprocessor stripped it. That mismatch costs us every time we reach for the AST.

The fix is to make @use a real construct, processed via the same data-driven stratum machinery that already handles @global, @fn, @if, @loop, and the rest.

2. Goals & Non-Goals

In Scope

  • @use parses to a real AST node visible to elaborator / typechecker / LSP.
  • One new CompilerAPI primitive: &Compiler::import.
  • A @stratum_keyword Use body that is sugar over that primitive.
  • Per-file module signatures (what each file exports).
  • Per-file typecheck that consults imported signatures — the LSP suppressions go away.
  • Cycle detection and dedup, preserved from the existing resolver.
  • @export finally gating visibility.
  • Migration path that keeps boot/ and existing tests green throughout.

Out of Scope (this doc)

  • Dynamic imports — tracked separately. The static module signatures we build here become the type of dynamic handles later, but the runtime side is its own design.
  • Selective imports (@use 'std/vec.si' { vec_new, vec_push };).
  • Renamed imports / namespaces (@use 'std/vec.si' as V;).
  • Package-manager-style indirection (@use 'natescode/foo';).
  • Conditional / platform-gated imports.
  • Grammar changes — the existing @use 'path'; shape stays exactly as-is.

3. Design Overview

3.1 What the user writes — unchanged

# lex.si
@use '../std/vec.si';
@use 'tokens.si';

@fn lex:Int ptr:Int, len:Int := { ... };

Syntax does not change. The grammar already accepts @use 'path'; as a Definition-like form — we just stop deleting it before the parser sees it.

3.2 What the stratum looks like

# src/strata/use.si  (new)
@stratum_keyword Use ('@use', Node) = {
  &IR::def_import;                                  # dispatch marker
  @local path := &Compiler::arg Node, 0;
  &Compiler::import path;                          # sugar → the primitive
};

The body is one line of real work. Everything else — locating the file, parsing, building its signature, registering the symbols in scope — lives behind the new &Compiler::import primitive. Same pattern as every other definition kind: the stratum is a tiny adapter; the primitive does the heavy lifting.

3.3 The new CompilerAPI primitive

Add one method to the CompilerAPI interface in src/compiler-api/index.ts: 

/**
 * Resolve and load a module by path. The path is relative to the current
 * file, absolute, or a module-registry key (e.g. 'std/vec.si').
 *
 * Side effects:
 *   - Reads + parses the target file if not already cached.
 *   - Recursively resolves its @use graph.
 *   - Adds an entry to the current file's import list.
 *   - Publishes the target's exported signatures into the current
 *     elaboration scope so the typechecker can see them.
 *
 * Returns the module handle so future surface (selective imports, renames)
 * can chain off it.
 */
import(path: string): ModuleHandle;

The TS-side field name is $compiler.import (consistent with $compiler being the internal handle the body interpreter dispatches into). The Silicon-side call site is &Compiler::import — the existing naming convention used by every stratum in src/strata/*.si.

3.4 The pipeline reshape

This is the structurally interesting part. Today's pipeline: 

source → useResolver → parser → AST → elaborate → typecheck → IR lower → emit

The reshape introduces an explicit module-resolution pass instead of the textual one:

source → parser → AST → resolveImports → elaborate → typecheck → IR lower → emit
                              ↑
                              runs the @use stratum bodies via the body interpreter;
                              primes the per-file module-symbol table the typechecker reads from

The new pass is data-driven by the registry: it walks top-level Definitions, finds those whose keyword is registered as an import kind in the strata registry, evaluates each body via the body interpreter, and collects the resulting ModuleHandles into the current file's import list.

Why a new pass and not "during IR lower"

Today's rich-body interpreter (compileBodyToExpanderFn / compileBodyToDefExpander) runs during IR lowering. That is far too late for @use — by then the typechecker has already raised every cross-file diagnostic. The body interpreter itself is reusable; we just invoke it earlier in the pipeline for the subset of stratum keywords flagged as import-kind. No new evaluator, no new language for the body — same machine, called earlier.

3.5 Module signatures

What each file exports, gathered after parse + a minimal "declaration pass" on the imported file:

Definition kindGoes into module signature?Notes
@fn name:T params := bodyYes — as FunctionSig { name, params: T[], result: T }Already in src/types/typechecker.ts.
@global name := exprYes — as { name, type, mutable: false }Type from initialiser inference.
@local name:T := exprYes — as { name, type, mutable: true } 
@externYesExternal signatures are inherently exported.
@enum / @type / @type_alias / @type_distinctYes — as type declarationsTypes and constructors both go into scope.
@stratum_keyword / @stratum_operatorYes — as registry entriesImported strata extend the importer's elaborator registry.
@localNoFunction-scoped — not even visible at file scope.
Top-level expressions / @use directivesNoSide-effect content, not a signature.

3.6 Visibility — @export stops being a lie

Once module signatures are real, two simple rules:

  • If a file contains at least one @export name declaration, its module signature contains only the explicitly-exported names. Implicit visibility off.
  • If a file contains no @export declarations, everything top-level is exported (back-compat — matches today's accidental behaviour).

This is the same rule TypeScript modules use ("file has at least one export ⇒ ES module") and avoids forcing a one-shot migration where every existing boot/ file suddenly needs @export stamped on everything.

3.7 Cycles, dedup, error mapping

  • Cycles: same rule as today's resolver — track the resolution stack, raise E0XXX ImportCycle with the cycle path.
  • Dedup: first-sighting wins; subsequent @uses of the same absolute path are a no-op (the signature is already in scope).
  • Error mapping: errors raised against imported file content carry that file's URI in the SourceSpan — no inlined-buffer offsets. Diagnostics show up on the correct file in the editor.

4. Is the Bootstrapped Compiler Capable of This?

Two distinct questions: can Stage 0 (the TS compiler) host this design today? Can Stage 1+ (the Silicon self-hosted compiler in boot/) express this in its own source code?

4.1 Stage 0 — Yes, with a small, surgical change

Capability check: the architecture already supports every piece this needs.

  • have it Data-driven stratum registry (buildStrataRegistry) — new keyword slots in.
  • have it Body interpreter that dispatches &Compiler::* calls (strataBody.ts) — reuse as-is.
  • have it Per-file module loader (src/modules/loader.ts) — we already read + cache files by absolute path.
  • have it Typechecker that consumes a pre-populated symbol map (typecheck(program, registry, modules)) — the seam already exists; today's ModuleRegistry is just used for a different purpose.
  • need it One new CompilerAPI method (import) and its LowerFns backing.
  • need it A new pre-elaborate pass (resolveImports) that runs the body interpreter on import-kind strata. ~150–250 LOC.
  • need it An "is this stratum keyword an import?" flag on registry entries — one boolean.
  • retire src/modules/useResolver.ts (kept as a back-compat shim during migration; deleted at the end of Phase C).

4.2 Stage 1+ self-host — Yes, with the existing primitive surface

The Silicon self-hosted compiler in boot/ needs to be able to:

  1. Read a source file from disk — already there, via WASIX fd_read in boot/std/io.si.
  2. Walk a parsed AST looking for @use definitions — already there, the AST traversal in boot/parser/parse.si handles Definition forms generically.
  3. Maintain a cache of "already-resolved modules" keyed by absolute path — already there, the bootstrap boot/std/map.si works for this.
  4. Re-invoke the parser on a new buffer — already there, the parser is just a Silicon function.
  5. Mutate the current elaboration context to add symbols — this is the only place the self-host needs to follow the same architecture as Stage 0: the symbol table in the bootstrap elaborator needs an import_symbols entry point. Adding it is mechanical (~30 LOC of Silicon).

Verdict: the bootstrapped compiler is capable today. The Silicon-side work is "implement &Compiler::import in Silicon" — one function that uses the same module-loader primitives the rest of the runtime already exposes. No new bootstrap-blocking dependency, no new WASIX surface needed, no grammar churn.

4.3 The bootstrap chicken-and-egg, addressed

Can the @use stratum itself be imported via @use?

No — and it doesn't need to be. Built-in strata are loaded by the elaborator eagerly from src/strata/*.si via strataLoader.ts (Stage 0) and from a hardcoded path in the bootstrap (boot/strata/) at Stage 1+. They are infrastructure — they predate the first user file and are available before @use resolution starts. Same as how today's operators.si isn't pulled in via @use.

The bootstrap order is: load built-in strata → parse entry file → resolve imports (using the now-loaded @use stratum) → elaborate → typecheck → .... No cycle.

7. Risks & Open Questions

Open: signature inference for @global initialisers

Building a module signature requires knowing the type of every exported @global. Today inference happens during typecheck, after elaborate. We need a lighter "signature inference" pass during resolveImports — or, simpler, allow forward references and only resolve @global types lazily on lookup. The simpler path adds latency to LSP hover/definition but avoids a duplicate pass. Decision deferred to Phase B kickoff.

Open: import-time vs. lower-time strata

Today every stratum body runs at IR lower time. @use needs to run earlier. The cleanest distinction is a metadata field on the @stratum_keyword registration: phase: 'import' | 'lower' (default 'lower'). User code can't currently set that — it's only meaningful for built-ins. Decision: leave it un-surfaced to user strata in Phase B; revisit when (if) a third-party stratum legitimately needs to run pre-elaborate.

Risk: byte-identical self-host

The Stage 1 == Stage 2 byte-equality (38,262 bytes today) is the load-bearing invariant of the whole bootstrap. Switching from textual inline to AST-level module resolution must not change the emitted WAT. Two ways this could break: top-level expression ordering (today's resolver concatenates in dependency order; the new pass must preserve that for the entry file's $__start) and synthetic name collisions (today's concat puts everything in one namespace; the new pass must still produce the same WAT identifiers). Phase B gate is byte-equal output on every boot/ file — any drift fails the phase.

Risk: strata-imports-strata ordering

If a @use'd file contains @stratum_keyword definitions, the importer's elaborator registry needs to be extended before the importer's other code is elaborated. That's the same ordering problem strata loading already handles; the resolution pass needs to do registry extension as it walks. Tested via a Phase B fixture: file A imports file B which defines an operator stratum; file A uses that operator.

Risk: WASIX in-memory loader

The bootstrap WASIX tests use an in-memory file loader (see UseResolverOptions.readFile). The new resolveImports pass needs the same injection seam — cleanly typed, ideally on the ModuleRegistry rather than on individual primitives.

8. What Lands First

Phase A — the thin slice — is the right next move. It removes the LSP's stripUseDirectives hack and gives go-to-definition a real AST node to hang off, without touching any of the typechecker / module-resolution semantics. The user sees an immediate quality bump in editor experience; the compiler internals stay unchanged. Risk is low because useResolver is still doing the actual work — the AST node is informational.

Phase B is where the design pays off: the LSP suppressions go away, errors point at the right files, and the foundation for @export and selective imports is in place. Worth doing in a dedicated session with its own design pre-read because the byte-identical-self-host gate is unforgiving.

Phase C is mostly clean-up — once Phase B is in, the preprocessor is dead weight.
