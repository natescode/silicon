Testing the Bootstrapped Sigil Compiler

A step-by-step walkthrough for verifying a fresh clone of Sigil, running the Silicon-side test suite, exercising the opt-in typechecker, and confirming the byte-equal self-host fixed point. Zero TypeScript, zero Bun — only wasmtime, wat2wasm, and a shell.

1. Prerequisites

The bootstrapped compiler runs as a single WebAssembly module (wasm-bin/stage1.wasm). To execute that module and assemble its WAT output you need two tools on PATH (or installed project-locally for wat2wasm):

ToolWhyInstall
wasmtime ≥ 14 WASI reference runtime; runs stage1.wasm and any user wasm you build. macOS / Linux 
curl https://wasmtime.dev/install.sh -sSf | bash
Windows

Download from github.com/bytecodealliance/wasmtime/releases.
wat2wasm (WABT) Assembles the WAT text Sigil emits into a WebAssembly binary. project-local (no admin) 
./scripts/install-wat2wasm.sh           # bash, writes ./bin/wat2wasm
.\scripts\install-wat2wasm.ps1          # PowerShell
system-wide 
brew install wabt                       # macOS
sudo apt install wabt                   # Debian / Ubuntu

Why wasmtime? Wasmer's WASI compat layer has known bugs at both 2.x (mapped-dir rights) and 7.x (post-path_open fd corruption + Windows absolute-path stdout) that broke the bootstrap during the port. The build is verified against wasmtime; other runtimes may or may not work.

2. Get the code

git clone https://github.com/natescode/sigil.git
cd sigil

There is nothing to install. No bun install, no npm ci, no cargo build. The repository ships with wasm-bin/stage1.wasm checked in — that's the bootstrap seed, and from it the rest is rebuilt.

3. Smoke test — build.sh check

The fastest way to confirm everything is wired up correctly is to ask the build to recompile itself and verify the bytes match: 

./build.sh check

Expected output:

build.sh: bundling 30 source files…
build.sh: compiling via stage1.wasm under wasmtime…
build.sh: assembling WAT → WASM via /…/bin/wat2wasm…
build.sh: ✓ fixed point — rebuilt stage1.wasm byte-equal to seed
PASS CRITERION   Final line contains ✓ fixed point — rebuilt stage1.wasm byte-equal to seed.

If you see this, every link in the chain works: wasmtime can run stage1.wasm; stage1.wasm can parse its own source; wat2wasm can assemble the emitted WAT; and the result is the same bytes as the seed — proving the compiler can produce itself.

5. Run one test by name

Useful while iterating:

./test.sh boot/tests/types_test.si
test.sh: running 1 test(s)…
  ✓ PASS boot/tests/types_test.si — types OK

test.sh: 1 passed, 0 failed

Glob expansion works too:

./test.sh boot/tests/check_*_test.si

Adding a new file: drop it into boot/tests/ as <feature>_test.si, make it write "<feature> OK" to stdout on success and call &wasi_snapshot_preview1::proc_exit 1 on failure, then append the path to DEFAULT_TESTS in test.sh.

6. Compile a Silicon program

Single-file program via stdin:

cat > hello.si <<'EOF'
@extern wasi_snapshot_preview1::fd_write:Int
  fd:Int, iovs_ptr:Int, iovs_len:Int, nwritten_out:Int;
@extern wasi_snapshot_preview1::proc_exit
  code:Int;

@fn _start:Void := {
  &wasi_snapshot_preview1::proc_exit 0;
};
EOF

wasmtime wasm-bin/stage1.wasm < hello.si > hello.wat
wat2wasm hello.wat -o hello.wasm
wasmtime hello.wasm
echo "exit=$?"
exit=0

Multi-file program with @use dependencies — give stage1 a positional path so its in-Silicon @use resolver walks the graph:

wasmtime --dir ./src::src wasm-bin/stage1.wasm src/main.si > main.wat
wat2wasm main.wat -o main.wasm
wasmtime main.wasm

The --dir ./src::src form is wasmtime's preopen syntax: “mount the host's ./src directory and let the guest see it as src”. stage1's @use resolver opens files relative to that preopen. Without it, stage1.wasm has no filesystem access and can only read stdin.

7. Use the opt-in typechecker

Add --typecheck to ask stage1 to run its Silicon-side checker between parse and lower: 

echo '@global x:Int := 5; @global y:Float := 3.14; (x + y);' \
  | wasmtime wasm-bin/stage1.wasm --typecheck > bad.wat

Standard output is the WAT (the compile continues regardless), and stderr carries the diagnostic:

[InvalidOperator] operator '+' cannot be applied to (Int, Float)

A clean program emits no stderr:

echo '@global x:Int := 5; (x + 1);' \
  | wasmtime wasm-bin/stage1.wasm --typecheck > ok.wat
# stderr is empty; exit 0

Error kinds the checker can report

KindTriggers whenExample diagnostic
MismatchExpected one type, got another (arg vs param, if-branches, etc.)[Mismatch] expected Int, got Float
InvalidOperatorOperator doesn't apply to the operand pair.[InvalidOperator] operator '+' cannot be applied to (String, Int)
UnboundIdentifierReference to a name not in scope.[UnboundIdentifier] unbound identifier 'foo'
UnknownTypeType annotation names a type the checker doesn't recognise.[UnknownType] unknown type 'Widget'
HeterogeneousArrayArray literal mixes element types.[HeterogeneousArray] array literal must be homogeneous: first element is Int, found Float
AnnotationBinding initialiser doesn't match the declared annotation.[Annotation] 'x' declared as Int but initialiser has type Bool
ImmutableAssignmentWriting to a @global / @fn / @extern name.[ImmutableAssignment] 'foo' is immutable and cannot be reassigned

Warn-only today. The typechecker prints diagnostics and the build proceeds regardless. This keeps the self-host fixed point intact while the checker matures. To make it fatal in your own project, wrap the stage1.wasm invocation in a shell test that fails the build when stderr is non-empty.

8. Verify the byte-equal self-host

This is the most important test in the repo — it proves that the compiler in boot/*.si produces, when run on its own source, the exact bytes of wasm-bin/stage1.wasm. Any change that breaks this gate is rejected by CI. 

./build.sh check

Internally the script:

  1. Reads the bootstrap seed wasm-bin/stage1.wasm from disk.
  2. Bundles the 30 files from STAGE1_FILES + the WASI extern stub.
  3. Pipes that bundle through the seed under wasmtime; captures stdout as new WAT.
  4. Runs wat2wasm on the new WAT to produce candidate WASM.
  5. Compares the candidate to the seed with cmp.
PASS   The two files are byte-identical. Print “✓ fixed point — rebuilt stage1.wasm byte-equal to seed”. Exit 0.

If you intentionally changed boot/*.si, the bytes will differ and check will fail. To advance the seed:

./build.sh                # overwrites wasm-bin/stage1.wasm with the new build
./build.sh check          # now verifies the NEW seed is self-consistent
./test.sh                 # re-run the suite under the new seed
git add wasm-bin/stage1.wasm    # commit the seed alongside your source change

10. Troubleshooting

SymptomLikely causeFix
build.sh: seed compiler …/stage1.wasm not found The checked-in seed was deleted or moved. git checkout HEAD -- wasm-bin/stage1.wasm to restore from the most recent commit.
build.sh: missing required tool 'wasmtime' wasmtime not on PATH. Install per step 1 or add it to PATH.
build.sh: missing required tool 'wat2wasm' WABT not installed and no project-local copy. Run ./scripts/install-wat2wasm.sh (or the .ps1).
✗ fixed-point FAILED Source under boot/ changed since the seed was built. If intentional, run ./build.sh to advance the seed; commit the new wasm-bin/stage1.wasm alongside your source change. If unintentional, git diff boot/ to find the drift.
A test prints "<name> FAIL <code>" The named test asserted an invariant and bailed. The numeric code maps to the assertion site in the test file — grep for e_die <code> (or the test's matching *_die helper) to find which check tripped.
parseWat failed: with garbage text in the error wasm-bin/boot.wasm from an unrelated run was reused by another tool. Not relevant on the Silicon-only path — boot.wasm is no longer produced. If you encounter it after porting custom scripts, delete wasm-bin/boot.{wat,wasm} and rebuild.
Error: failed to open wasm module "wasm-bin/stage1.wasm" You're invoking wasmtime from a different working directory. Use an absolute path, or cd to the repo root first. The build scripts always cd first themselves.

11. Adding your own tests

Convention: every test file lives at boot/tests/<feature>_test.si and:

  • Prints "<feature> OK\n" to stdout (fd 1) on success.
  • Calls &wasi_snapshot_preview1::proc_exit 1 with a non-zero status on any failed assertion.
  • Uses @use to pull in boot/std/io.si + any module under test. The shell runner resolves the @use graph depth-first; you don't need to manually order dependencies.

Boilerplate template:

# boot/tests/<feature>_test.si
@use '../std/io.si';
@use '../path/to/module_under_test.si';

@fn die code:Int := {
  &write_str 1, '<feature> FAIL ';
  # inline integer printer — copy from one of the existing tests
  &write_byte 1, 10;
  &wasi_snapshot_preview1::proc_exit 1;
};

@fn run := {
  &module_init;

  &@if ((&some_check) != EXPECTED), { &die 1; };
  &@if ((&another_check) != 42),    { &die 2; };

  &write_str 1, '<feature> OK';
  &write_byte 1, 10;
  &wasi_snapshot_preview1::proc_exit 0;
};

&run;

Then add the path to DEFAULT_TESTS in test.sh and re-run ./test.sh. If your test compiles via stage1 but the runtime needs special wasmtime args (e.g. preopen directories), invoke it manually:

cat boot/std/io.si boot/your_module.si boot/tests/your_test.si \
  | wasmtime wasm-bin/stage1.wasm > your_test.wat
wat2wasm your_test.wat -o your_test.wasm
wasmtime --dir ./fixtures::fx your_test.wasm
