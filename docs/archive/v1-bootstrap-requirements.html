Sigil 1.0 — Bootstrap Requirements

Audit of src/ TypeScript compiler gaps that must be resolved before porting to boot/ Silicon and permanently deleting src/. Date: 2026-05-26. Companion to v1-user-stories.html (Phase 7 boot/ port) and api-boundaries.md.

Verdict. src/ is roughly 80% ready to be the spec for a permanent boot/ port, but eight unresolved items — if not closed first — will force expensive re-litigation in Silicon syntax later. Three are hard blockers (architectural decisions baked into surface contracts), three are surface stability issues (decisions you'll regret leaving open), and two are completion debt (things known-broken in src/ that boot/ would inherit). None individually are large; collectively they're ~2–3 weeks of focused work.

The single biggest hidden risk is @generic monomorphization: it has 4/4 + 5/6 skipped tests in src/. It is not working end-to-end. If you port the current shape to boot/, you bake a non-functional surface into the self-host gate.

1. Audit methodology & scope

The audit covered five orthogonal angles against the current src/ tree (63 TypeScript files, ~16,300 LOC; 21 skipped tests across the suite):
  • CaaS API surface — is src/caas/ + src/api.ts stable enough to port?
  • Strata 2.0 surface — is the unified @stratum := { ... } + phase-hooks model finalized?
  • Type system & HM-lite — is generics, inference, and monomorphization shipping?
  • Codegen backends — are WAT, WASM-binary, and QBE paths uniformly abstracted?
  • JS/Bun/Node coupling — what TypeScript-specific patterns won't translate to Silicon?
The output below distinguishes what must be fixed in src/ before porting (this doc) from what's tracked as Phase 7 work in boot/ (see v1-user-stories.html § Phase 7). These are complementary lists, not duplicates.

2. Hard blockers — architectural decisions

These are design questions not yet locked in src/. Each one bakes itself into the surface contract — porting them to Silicon without an answer means committing to a shape you may want to revise later, in the harder language.

2a. @generic monomorphization is aspirational, not shipping

Severity: HIGH · Type: Architectural decision · Est: decide ~1d; ship if chosen ~3d

src/elaborator/generic-monomorph.test.ts — 4/4 tests are test.skip. src/elaborator/generic-e2e.test.ts — 5/6 tests are test.skip.

HM-lite inference itself works (33 e2e + 38 unit tests pass). Option[T] / Result[T,E] ship cleanly because they're polymorphic sum types, not polymorphic functions. But for a user-written @generic id[T] x:T := x; followed by &id 42 and &id 3.14, the monomorphization stratum (capture-template → infer at call site → mangle name → push synthesised @fn → rewrite call) does not actually run.

Evidence: the skipped tests document the intended behaviour but never assert on real output. The stratum exists in src/strata/modules/ but isn't wired into the lowering pass beyond a no-op on::lower hook.

Why this blocks the port: the entire Strata 2.0 surface advertises monomorphization as the canonical pattern (docs/strata-authoring-guide.md §8). Porting it to boot/ in its current skipped state means the boot/ tests for it would also need to be skipped — and you can't validate the boot/ implementation works without a working src/ reference.

Action: answer the question "does 1.0 ship with user-written @generic functions, or does that move to Phase 11 (interpreter-only)?" Either complete the @generic stratum in src/ (the aspirational test bodies sketch the design well), or formally move monomorphization to v1.1 and stop documenting it in 1.0 strata-authoring materials. This single answer reshapes 30–40% of remaining Phase 7 work.

2b. legacyBlockTranslator.ts: keep permanently or rip out?

Severity: HIGH · Type: Architectural decision · Est: decide ~2h; rewrite path ~4–6h

src/elaborator/legacyBlockTranslator.ts rewrites inline-handler &Compiler::* bodies into the post-dissolution &compiler::* form. 17 strata2 tests skip pending "new-form rewrites." The translator is currently load-bearing — the Phase C engine has no interpreter fallback, so when WASM compilation of a handler fails the compiler errors out instead of falling back.

If you port the translator to Silicon, you're committing to maintain two surface forms forever in the self-hosted compiler. If you don't, every built-in stratum in boot/strata/builtin/*.si and src/strata/modules/*.si that uses &Compiler::* must be rewritten to &compiler::* first.

Evidence: ~17 test.skip entries in src/elaborator/strata2.test.ts with notes like "interpreter-specific patterns (template-handle field access, scope-variable method calls) skipped pending new-form rewrites."

Action: the cleanest path is "rewrite all in-tree strata to new &compiler::* form, delete the translator, then port". Roughly half a day of mechanical rewrites; saves ~400 LOC of Silicon you'd otherwise need to author and maintain. Pair with unskipping the 17 strata2 tests as the acceptance gate.

2c. Comptime engine production path (story 3-22 / 8-9)

Severity: HIGH · Type: Architectural decision · Est: decide ~1d; complete import shim ~1d

src/ ships an AST-interpreter shim today, not the WASM-in-WASM engine. The plan moved from "embed a tiny wasm interpreter" to "link wasmtime as a C library from the Phase 8 native binary." This is a scoping decision that affects what boot/ needs to do at all — if wasmtime-as-library is the answer, boot/ never needs a WASM-in-WASM engine, and a chunk of story 3-22 evaporates.

Today's src/ bridge pattern has a hardcoded handler-import surface (src/comptime/imports.ts) that is incomplete — it lacks ast::clone, ast::patch_types, the type::* primitives, and complex AST field accessors. Handlers that need these silently fall back to interpretation — but the interpreter was deleted in Phase 1 dissolution, so they crash today.

Evidence: src/comptime/engine.ts lines 93, 137 document the fallback; src/comptime/imports.ts top-of-file comment lists what's NOT yet wired through. Story 3-22 in v1-user-stories.html confirms "production deferred to 8-9."

Action: lock the comptime engine architecture. Either (a) complete src/comptime/imports.ts so the bridge surface is total in src/ (and boot/ has a faithful target), or (b) commit to "comptime engine = wasmtime FFI" and stop growing the JS-side import shim. Either way, document the final import surface in docs/compiler-api.md so the boot/ port has a fixed target.

3. Architectural decisions still pending

Below the hard blockers are three medium-severity items. Each is a cross-cutting refactor that is much cheaper to do in TypeScript now than in Silicon later.

3a. No uniform Backend interface

Severity: MEDIUM · Type: Refactor · Est: ~4h

Today compileToWat(p, ...), compileToWasm(p, ...), and lowerToQbe(p, ...) are three independent entry points with different signatures. Boot/ inherits the same ad-hoc shape unless you finalize a Backend<T> abstraction now.

This isn't a bug today — but it's the kind of cross-cutting refactor that's painful to do in Silicon in boot/, much cheaper to do in TypeScript in src/. The boot/ port will be 3× faster if there's a single contract to mirror instead of three ad-hoc ones.

Action: add a Backend<T> interface (~30 lines TS) with three implementations. Define the contract as compile(program, registry, fns, opts): { artefact: T, diagnostics: Diagnostic[] }. Wire src/sigil_cli.ts through the interface so the public CLI proves the abstraction holds.

3b. Map / WeakMap in public CaaS types

Severity: MEDIUM · Type: Surface refactor · Est: ~6h

src/modules/registry.ts:16 exposes Map<string, ModuleEntry> in LowerOptions2. src/caas/workspace.ts returns ReadonlyMap<string, Document>. src/ast/semanticModel.ts uses WeakMap<object, SiliconType> for type annotations.

Silicon has no Map; boot/ will need a HashMap stdlib type (which exists per 5a-6 ✅) or arrays-of-pairs. The current public types declare a porting hazard — if you ship 1.0 with these signatures, the boot/ compiler must either (a) expose JS-Map-shaped data through a HashMap-backed wrapper that breaks the stability contract, or (b) you break it earlier by changing the surface in 1.x.

Action: redesign the affected public signatures to use arrays-of-pairs or named opaque types. Internal Maps can stay; the surface must be Silicon-portable. Approximately 3 type definitions and ~10 call sites to update.

3c. Direct WASM binary emitter has a WAT-fallback escape hatch

Severity: MEDIUM · Type: Backend completion or formal retirement · Est: ~1d either way

src/codegen/wasm-emitter.ts doesn't handle IRCallIndirect — funcref-using modules currently route through compileToWat → watToWasmSync (binaryen.js). If boot/ ports this as-is, you replicate the same crutch — and boot/ doesn't have binaryen.js to fall back to without shelling out to wabt/wasm-tools.

Mixed mode (sometimes direct binary, sometimes WAT-fallback) is the worst case to inherit because callers can't predict which path a given program takes.

Action: two acceptable paths. Either (a) complete the direct binary emitter (funcref type/table/elem/call_indirect sections, ~1d), or (b) formally retire the direct emitter for 1.0 and route everything through WAT → binaryen until v1.1. Document the decision in docs/wasm-binary-emitter-plan.md. Mixed mode is not an acceptable port target.

4. Surface cleanup before port

These are low-severity hygiene items, but each represents a "thing that doesn't make sense to translate to Silicon." Cheaper to remove or finalize in src/ than to litigate in boot/.

4a. QBE backend missing critical Silicon features

Severity: MEDIUM (blocks native self-host) · Type: Completion · Est: ~4–5d

QBE lowerer is ~65% complete. Missing:

  • String literals & data segments — selfhost excludes string_literal.si from the QBE-self-host gate
  • Structs & field layout — struct_basic.si, struct_nested.si excluded
  • Type casts (@toFloat, @toInt) — three explicit # TODO(qbe) markers in src/codegen/qbe/lower.ts (lines 530, 764, 922)
  • @extern FFI marshaling — int64_extern_call.si excluded; no implementations provided
  • @defer cleanup expressions — defer_*.si excluded
  • Struct-based @global — uncertain call-style access

If you delete src/ before these land, you also delete the reference implementation for boot/'s QBE port (story 9.5-6). Self-hosting on native (no wasmtime) requires them.

Action: ship Phase 8 stories 8-6+ to feature parity. Each missing feature is a session of work; the list is well-scoped because each is gated by an excluded selfhost test.

4b. Dead code in src/codegen/qbe/types.ts

Severity: LOW · Type: Cleanup · Est: ~30min

WAT_TO_QBE_INSTR (lines 69–115) is unused since the AbstractOp introduction in Phase 9.5 but kept around as an intermediate. Delete before porting — boot/ shouldn't inherit confused intermediate layers.

Action: delete the unused table; rerun the test suite to confirm zero references.

4c. Pretty renderer is documented as "throwaway"

Severity: LOW · Type: Lock surface · Est: ~3h

src/errors/diagnostic.ts:6 — "pretty rendering is throwaway, so we land the records now and skip the rendering polish." The renderPretty function exists but is ~20 lines of placeholder. The diagnostic record schema is final (code/span/message/hint/snippet/severity), but the renderer that produces user-facing output is not.

Re-implementing a different style in Silicon is wasteful. Re-implementing the same throwaway is wasteful. Pick a final style now.

Action: lock the renderer surface in src/ — caret style (Rust-style ASCII vs ANSI box-drawing), color story (true-color, 256, or none in 1.0), hint formatting (inline "hint:" prefix vs separate block), "did you mean" position (above vs below the caret). Update docs/diagnostics.md with the final shape.

5. Completion debt boot/ would inherit

5a. Bootstrap parser grammar coverage (~40%)

Severity: HIGH · Type: Boot/ completion (tracked under story 3-26) · Est: ~1–2d (revising 3-26 estimate up)

boot/parser/parse.si (755 lines) currently covers ~40% of the Ohm grammar — literals, namespaces, binary expressions, function calls, parens. Missing: Block, Definition, DocComment, Assignment, VariantDecl, ArrayLiteral, ObjectLiteral, TupleLiteral, generic type params.

Note: this isn't a src/ deficiency — it's that boot/parse.si lags. But it's a hard prerequisite for src/ deletion. Don't underestimate it — story 3-26 is rated "L, ~8h" in v1-user-stories.html, but the actual surface area touches ~50 semantic actions in src/ast/toAst.ts (448 lines), each of which needs a Silicon-side equivalent.

Action: revise the story 3-26 estimate upward (XL, ~1–2d rather than L, ~8h). Sequence as several small commits that each preserve byte-equal (the self-host gate constrains this), pairing with boot/tests/ additions that mirror src/'s grammar tests.

5b. CaaS Symbol/Reference schema not future-proofed

Severity: MEDIUM · Type: Schema design · Est: ~2h

src/caas/workspace.ts:23 — "Cross-document symbol resolution (CaaS-5) is not yet implemented." Block-local scope tracking is also deferred. Both are post-1.0 acceptable per the v1 plan, but they shape the Symbol/Reference types in the public API.

If you ship symbol types that don't account for cross-doc and scope, then v1.1 breaks the stability contract when adding them. Optional fields are free at the type level; mandatory schema breaks are expensive.

Action: finalize the Symbol/Reference type shape now with optional containingScope?: Scope and crossDocumentReferences?: Reference[], even if the implementation returns undefined / empty array in 1.0. Future-proof the schema without doing the work.

6. What is already safe to port

Not everything is at risk. The audit found these subsystems architecturally ready to be mirrored in boot/ today:
  • HM-lite type inference core — src/types/unify.ts + typechecker.ts are self-contained, 33 e2e + 38 unit tests passing. Phase 2 boot/ port (docs/phase-2-typechecker-port.md) is the faithful target.
  • Sum types with type parameters — Option[T], Result[T,E] ship cleanly. Multi-parameter types work post-Phase-5 grammar fix.
  • IR (post-9.5 abstract opcodes) — IRBinOp.op: AbstractOp is backend-agnostic. WAT emitter and WASM binary emitter consume via the same intrinsics registry lookup.
  • Intrinsics registry — single source of truth for WASM ops; wasmIntrinsics + ABSTRACT_OP_TO_QBE are the only tables. Already mirrored in boot/.
  • Diagnostic record schema — phase/code/span/message/hint/ severity is final. Only the renderer is throwaway.
  • Phase hooks integration — all 5 hooks (decl, callSite, annotation, lower, module_finalize) fire from src/ir/lower.ts in production today. Boot/ Phase 7 firing helpers exist (3-10 ✅); only lowering-pass integration is opt-in.
  • Tier model — T0/T1/T2 tier ordering enforced; cycle detection shipping. @@before / @@after across strata is the only partial part (registration order suffices for MVP).
  • sgl CLI shape — subcommand routing, manifest parser, init/build/check/fmt all stable. Stub commands (test/eval/add) formally deferred to Phase 12.

7. Recommended sequence

Four phases. Total ~2–3 weeks of src/ work before boot/ port can proceed safely. Each phase has a clear acceptance gate.

Step 1 — Architectural locks (src/, ~1 week)

The most expensive decisions to defer. Each is a yes/no question with downstream consequences.

  1. Decide @generic monomorphization (1d) — ship it in src/ or formally move to Phase 11 (interpreter-only). Update docs/strata-authoring-guide.md §8 either way. Unskip or delete generic-monomorph.test.ts + generic-e2e.test.ts.
    Acceptance: zero skipped tests in those two files; or both files deleted with rationale in commit message.
  2. Rewrite all in-tree strata to new &compiler::* form (4–6h) — mechanical rewrites of src/strata/modules/*.si and boot/strata/builtin/*.si. Delete src/elaborator/legacyBlockTranslator.ts. Unskip the 17 strata2 tests as the acceptance gate.
    Acceptance: legacyBlockTranslator.ts deleted; strata2.test.ts shows 0 skipped tests.
  3. Lock comptime engine architecture (1d) — answer "wasmtime-as-library vs WASM-in-WASM." Either complete src/comptime/imports.ts to total coverage (and update story 3-22 to reflect the import-shim path), or commit to wasmtime FFI (and document the C-library binding shape in story 8-9).
    Acceptance: docs/compiler-api.md lists every host-import the comptime engine exposes; no "deferred to interpreter" footnotes.
  4. Add Backend<T> interface (4h) — single contract with three implementations (WAT text, WASM binary, QBE IR). Wire src/sigil_cli.ts through the interface.
    Acceptance: sigil_cli.ts has zero direct imports from src/codegen/qbe/ or src/codegen/wasm-emitter.ts; only the Backend interface.

Step 2 — Surface stability (src/, ~3 days)

Lock the shapes that boot/ will inherit forever. Cheap now, expensive later.

  1. Refactor public Map / WeakMap → arrays / HashMap-shaped types (6h) — three type definitions (ModuleRegistry, Workspace.documents, SemanticModel type annotations) and ~10 call sites. Internal Maps can stay; the surface must be portable.
    Acceptance: zero Map<...> or WeakMap<...> in the public exports of src/api.ts or src/caas/index.ts.
  2. Finalize Symbol / Reference schema with optional fields (2h) — add containingScope? and crossDocumentReferences? now, return undefined / empty array in 1.0. Documents the v1.1 extension points without doing the work.
    Acceptance: stability.md §3 lists Symbol/Reference fields with their 1.0 contract; CaaS-5 story explicitly references the existing optional fields.
  3. Lock renderPretty caret / color / hint formatting (3h) — pick a final style. Re-implement src/errors/diagnostic.ts lines 167–185 with the locked behaviour. Update docs/diagnostics.md with worked examples.
    Acceptance: docs/diagnostics.md shows three rendered examples (type error, parse error, hint with did-you-mean); src/errors/diagnostic.ts comment no longer says "throwaway."

Step 3 — Completion debt (src/, ~5 days)

Things known-broken in src/ that boot/ would inherit. Each one removes a footnote from the boot/ port stories.

  1. Complete QBE backend (4–5d) — strings, structs, type casts, @extern marshaling, @defer cleanup, struct-based @global. Each feature is gated by a currently-excluded selfhost test (src/e2e/selfhost.test.ts lines 87–95 list them).
    Acceptance: src/e2e/selfhost.test.ts excludeList shrinks to ≤ 2 entries; each remaining exclusion has a tracked v1.1 story.
  2. Finish direct WASM emitter funcref support OR formally retire it (1d) — pick one. If complete, add type/table/elem/call_indirect sections to src/codegen/wasm-emitter.ts. If retire, delete the direct path and route everything through WAT → binaryen until v1.1.
    Acceptance: zero compileToWat → watToWasmSync fallbacks remain in compileToWasm; or compileToWasm itself is deleted from the public API.
  3. Delete dead intermediate layers (30min) — WAT_TO_QBE_INSTR in src/codegen/qbe/types.ts lines 69–115; any other unused tables.
    Acceptance: grep for WAT_TO_QBE_INSTR returns no matches outside git history.

Step 4 — Boot/ port can proceed safely

With Steps 1–3 done, the existing Phase 7 + 9.5-6 + Phase 11 work has a stable target. No re-litigation of architecture in Silicon syntax.

  1. Phase 7 (21 open stories) — Strata 2.0 in boot/, by then has a stable target. Estimated ~18 days as currently scoped.
  2. Phase 9.5-6 — QBE abstract opcode dispatch in boot/strata/builtin. Unblocked once Phase 7 is substantially complete and Step 3 has shipped full QBE features in src/ as reference.
  3. Phase 11 (Silicon interpreter) — can begin in parallel with Phase 7 endgame. Sequencing constraint: needs Phase 9 (QBE) ✅ so interpreter compiles to native for usable performance.
  4. Phase 12 (interpreter-backed sgl subcommands) — closes the loop after both Phase 8 ✅ and Phase 11 are done.

8. Gate criteria for deleting src/

Do not delete src/ until all of the following hold simultaneously:
  1. Self-host byte-equal gate passes — ./build.sh check produces a byte-identical wasm-bin/stage1.wasm using only boot/ sources, with no src/ involvement.
  2. Full src/e2e/ corpus passes via boot/ — every test currently in src/e2e/e2e.test.ts (and selfhost / backends tests) has a Silicon-side translation in boot/tests/ that passes under wasmtime.
  3. All three backends emit from boot/ — boot/ produces WAT text, WASM binary, and QBE IR with byte-equal output to src/'s outputs on the example corpus.
  4. CaaS surface implemented in boot/ — Workspace, Document, SemanticModel, Symbol, Reference types and their methods callable from Silicon code. At least one consumer (sgl fmt) proves the surface works.
  5. Phase 7 (Strata 2.0 in boot/) is at 32/32 — including story 5a-7 (Vec/HashMap/Option internal data structure migration).
  6. Comptime engine has shipped its production answer — either the import-shim is total in boot/, or the wasmtime FFI binding works in the native sigilc binary. No more "deferred to interpreter" fallbacks.
  7. Stability contract honoured — every exported name in src/api.ts has an equivalent in boot/ with the documented contract from docs/stability.md.

Until all seven hold, src/ stays in the tree as the reference implementation. The day all seven pass, src/ can be removed in a single commit — and that commit is the formal cut-over to permanent self-hosting.

10. CaaS API reference

Full reference for the public Silicon CaaS surface, exported from src/api.ts. All names listed here are covered by the 1.0 stability promise (docs/stability.md §2) unless tagged @internal or port hazard. This is the contract boot/ must mirror before src/ can be deleted.

10.1 Pipeline functions

Pure functions, one per phase. None throw on user errors — failures land in the returned diagnostics array. Compose freely; each takes the previous stage's output as input.

parse(source, options?) → ParseResult stable

function parse(source: string, options?: ParseOptions): ParseResult

Parse Silicon source text into a SyntaxTree. Parse failures are captured as Diagnostic records with phase='parse' and code E0000; a minimal valid tree is returned so callers don't have to null-check.

interface ParseOptions { file?: string } // span attribution; defaults to '<input>'

buildRegistry(tree, extraSources?) → ElaboratorRegistry stable

function buildRegistry(tree: SyntaxTree, extraSources?: string[]): ElaboratorRegistry

Walk the tree's @stratum declarations and build the elaboration registry. Call after parse() and before elaborate(). In a long-lived workspace, build once and reuse across edits — only call again when strata themselves change.

elaborate(tree, registry, options?) → ElabResult stable

function elaborate(tree: SyntaxTree, registry: ElaboratorRegistry, options?: ElabOptions): ElabResult

Resolve operators and keywords against the strata registry. Returns a new SyntaxTree with the elaborated program. Elaboration errors are captured as Diagnostic records with phase='elaborate' and code E0001.

interface ElabOptions { extraSources?: string[] } // additional strata source text to merge in

typecheck(tree, registry, options?) → CheckResult stable

function typecheck(tree: SyntaxTree, registry: ElaboratorRegistry, options?: CheckOptions): CheckResult

Type-check an elaborated tree using HM-lite inference. Builds a SemanticModel with queryable type and symbol information. All type errors are captured as Diagnostic records with phase='typecheck' and stable codes (E0001–E0009; see TYPE_ERROR_CODES).

interface CheckOptions { moduleRegistry?: ModuleRegistry } // pre-registered module signatures (WASI, etc.)

lower(tree, registry, model, options?) → LowerResult stable

function lower(tree: SyntaxTree, registry: ElaboratorRegistry, model: SemanticModel, options?: LowerOptions2): LowerResult

Lower a type-checked tree to WAT (WebAssembly text format). Lowering errors land in the result's diagnostics array with phase='lower' and code E0010.

interface LowerOptions2 extends LowerOptions { moduleRegistry?: ModuleRegistry // required for WASI / external module calls _functions?: Map<string, FunctionSig> // @internal — port hazard 🔴 3b }

compile(source, options?) → CompileResult stable

function compile(source: string, options?: ParseOptions & ElabOptions & LowerOptions2): CompileResult

Convenience wrapper running the full pipeline: parse → buildRegistry → elaborate → typecheck → lower. Returns at the first phase that produces diagnostics. On success, wat holds the emitted WAT text.

interface CompileResult { readonly wat: string readonly model: SemanticModel | undefined readonly diagnostics: readonly Diagnostic[] }

10.2 Workspace API (CaaS-4)

Multi-document project state. A Workspace owns a shared strata registry and a set of open Documents, each with its own versioned source, trees, semantic model, and diagnostics.

class Workspace stable

new Workspace(options?: WorkspaceOptions) interface WorkspaceOptions { registry?: ElaboratorRegistry } // pre-built registry; else built lazily from first opened doc

Construct a workspace. The strata registry is built lazily from the first opened document unless a pre-built one is provided.

workspace.openDocument(uri, source) → Document stable

openDocument(uri: string, source: string): Document

Open a new document and run all compilation phases on its source. Throws if uri is already open — use editDocument for subsequent versions. Fires a 'opened' change event.

workspace.editDocument(uri, newSource) → Document stable

editDocument(uri: string, newSource: string): Document

Apply a full-text replacement to an open document. Re-parses, re-elaborates, and re-typechecks against the shared registry. Bumps version by 1. Fires a 'changed' change event. Throws if uri is not open.

workspace.closeDocument(uri) → void stable

closeDocument(uri: string): void

Remove a document from the workspace. No-op if not open. Fires a 'closed' change event.

workspace.onDidChange(listener) → unsubscribe stable

onDidChange(listener: (event: DocumentChangeEvent) => void): () => void interface DocumentChangeEvent { readonly kind: 'opened' | 'changed' | 'closed' readonly uri: string readonly document: Document | undefined // undefined when kind === 'closed' }

Subscribe to document lifecycle events. Returns an unsubscribe function. Listeners fire synchronously after the workspace state mutates.

workspace.findDefinition(uri, line, col) → CaaSSymbol | undefined stable

findDefinition(uri: string, line: number, col: number): CaaSSymbol | undefined

Find the symbol whose definition or reference covers the 1-based (line, col) cursor in uri. Returns undefined if the document is not open or no symbol covers the position. Use symbol.definitionSpan to jump to the declaration site.

workspace.findReferences(uri, line, col) → readonly SourceSpan[] stable

findReferences(uri: string, line: number, col: number): readonly SourceSpan[]

Find all reference spans for the symbol at (line, col) in uri. Returns an empty array if the position covers no symbol or location info was not recorded. Cross-document references are deferred (CaaS-5).

workspace.documents port hazard 🔴 3b

get documents(): ReadonlyMap<string, Document>

All currently open documents, keyed by URI. Port hazard: Silicon has no Map type. See §3b — should be redesigned to return a Silicon-portable shape (array-of-pairs or named opaque type) before boot/ port.

10.3 Document type

interface Document stable

interface Document { readonly uri: string // path / URI key readonly source: string // current text readonly version: number // monotonically increasing, starts at 1 readonly tree: SyntaxTree // parse-phase output readonly elabTree: SyntaxTree // elaborate-phase output readonly model: SemanticModel // typecheck-phase output; queryable overlay readonly diagnostics: readonly Diagnostic[] // union of all per-phase diagnostics, in pipeline order }

Immutable snapshot of one source file's compiled state. Every edit produces a new Document instance (the workspace replaces the map entry); never mutated in place.

10.4 SemanticModel — queryable overlay

Built by the type-checker; consumed by lowering, the formatter, the LSP, and any other tool that needs to ask "what's the type of this node?" or "where is this symbol defined?".

model.typeOf(node) → SiliconType | undefined stable

typeOf(node: object): SiliconType | undefined

Look up the inferred Silicon type of any AST node. Returns undefined if no type was recorded for the node (e.g. a statement-position node).

model.symbolAt(node) → Symbol | undefined stable

symbolAt(node: object): Symbol | undefined

Resolve a Namespace reference node (or any node whose identity was recorded during typechecking) to its declared Symbol.

model.symbolNamed(name) → Symbol | undefined stable

symbolNamed(name: string): Symbol | undefined

Look up a symbol by its declared name (the fully-qualified namespace string, e.g. "module::function").

model.symbolAtPosition(line, col) → Symbol | undefined stable

symbolAtPosition(line: number, col: number): Symbol | undefined

Find the symbol whose definition or reference covers the 1-based (line, col) position. Searches definition sites first, then references. Returns undefined if no span covers the position.

model.referencesTo(symbol) → readonly object[] stable

referencesTo(symbol: Symbol): readonly object[]

All AST nodes that reference symbol. Use referenceSpans instead if you only need source locations.

model.referenceSpans(symbol) → readonly SourceSpan[] stable

referenceSpans(symbol: Symbol): readonly SourceSpan[]

Source spans of every reference to symbol. Returns an empty array when location info was not recorded (pre-Ohm ASTs in unit tests).

model.diagnosticsIn(range?) → readonly Diagnostic[] stable

diagnosticsIn(range?: SourceRange): readonly Diagnostic[]

All diagnostics whose span overlaps the given range. Pass undefined to get all diagnostics for the document.

model.allSymbols / model.allDiagnostics stable

get allSymbols(): IterableIterator<Symbol> readonly allDiagnostics: readonly Diagnostic[]

Flat iteration over every declared symbol and every recorded diagnostic in the document.

10.5 Diagnostic record schema

interface Diagnostic schema stable renderer in flux 🟡 4c

interface Diagnostic { phase: 'parse' | 'elaborate' | 'typecheck' | 'lower' | 'emit' code: string // stable identifier (E0001…); never reused span: SourceSpan // where in source the diagnostic points message: string // short human message; no "Error:" prefix hint?: string // optional secondary advice ("did you mean…") notes?: Diagnostic[] // optional related sub-diagnostics snippet?: string // optional verbatim source line for caret rendering } interface SourceSpan { file: string // empty when synthesised line: number // 1-based col: number // 1-based length: number // byte length; 0 = point span }

Authoritative diagnostic record. Every error site SHOULD emit a Diagnostic rather than throwing. The schema is final; only the pretty renderer (renderPretty) is documented as throwaway in src/ and must be locked before porting (see §4c).

10.6 SyntaxTree class

class SyntaxTree stable

class SyntaxTree { readonly program: Program // the underlying AST readonly source: string // original source text readonly file: string // file name for diagnostic spans constructor(program: Program, source: string, file?: string) withText(newSource: string, options?: ParseOptions): ParseResult }

Immutable wrapper around a parsed program; the currency between pipeline stages. withText re-parses new source without rebuilding the strata registry — the typical LSP / REPL incremental-edit path. Strata added or removed in the new source are invisible until buildRegistry runs again.

10.7 Symbol type

interface Symbol schema stable extension points pending 🟡 5b

type SymbolKind = 'function' | 'variable' | 'type' | 'parameter' | 'stratum' interface Symbol { readonly name: string // fully-qualified declared name readonly kind: SymbolKind readonly definitionNode: object // Definition AST node readonly type: SiliconType | undefined // inferred or annotated type readonly definitionSpan?: SourceSpan // source span of the name; undefined if unknown // — pending additions before 1.0 (§5b) — // readonly containingScope?: Scope // readonly crossDocumentReferences?: readonly Reference[] }

Definition-site record. See §5b — the two commented-out fields should be added (as optional) to the 1.0 schema even if their implementations are deferred, so v1.1 doesn't break the stability contract when scope and cross-document tracking land.

Re-exported types stable

// From the type system export type { SiliconType } // From the elaborator export type { ElaboratorRegistry } // From the module loader export type { ModuleRegistry } // From the AST/semantic model export type { SourceRange, CaaSSymbol, SymbolKind } // Result types export type { ParseResult, ElabResult, CheckResult, LowerResult, CompileResult } // Option types export type { ParseOptions, ElabOptions, CheckOptions } // Shared export type { Diagnostic, SourceSpan, Phase, SemanticModel }

10.9 What boot/ must mirror

When porting to boot/, every name in §10.1–§10.7 needs a Silicon-side equivalent with the documented contract. The mechanical surface is ~22 functions / ~12 record types / 2 classes — bounded and well-scoped.

Per-section porting notes:
  • §10.1 Pipeline functions — pure functions, port directly. Boot/ Phase 7 already has parse, elaborate, typecheck, lower as separate passes; wrapping them in the CaaS shape is a thin API layer.
  • §10.2 Workspace — needs a Silicon HashMap (shipped 5a-6 ✅) or array-of-pairs. The onDidChange callback model needs Silicon closures (Phase 11) or a table-of-handlers indirection if porting before Phase 11.
  • §10.3 Document — pure record, port directly.
  • §10.4 SemanticModel — the WeakMap<node, type> shape doesn't translate. Use HashMap[i32, i32] keyed by AST node id instead.
  • §10.5 Diagnostic — pure record shape, port directly. Renderer comes after.
  • §10.6 SyntaxTree — port directly; class becomes a record + free functions.
  • §10.7 Symbol — add the optional extension fields before porting so the boot/ side starts with the future-proof shape.
