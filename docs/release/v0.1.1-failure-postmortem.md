# Postmortem — the v0.1.1 release that never published

## Summary

The `v0.1.1` tag was pushed and the **Release** workflow started, but no
GitHub Release was ever published. The run was eventually cancelled after
**24 hours**. Root cause: the `build-macos-x64` job targeted the retired Intel
macOS runner (`runs-on: macos-13`), which never became available, and the
`release` job `needs:`-depended on it — so the publish step never ran.

This was fixed in `release.yml` for `v0.1.2`.

## Evidence

From `gh run view 26849193004` (the `v0.1.1` Release run):

```
X Build — macOS x86-64 in 24h0m0s        (ID 79176732111)
✓ Build — macOS ARM64 (Apple Silicon) in 22s
✓ Build — Linux x86-64 in 18s
✓ Build — Linux ARM64 in 17s
- Publish GitHub Release in 0s           (never started)

ANNOTATIONS
X The job has exceeded the maximum execution time while awaiting a runner
  for 24h0m0s  —  Build — macOS x86-64
```

The three other build jobs finished in ~20s each. Only the Intel macOS job
hung — "awaiting a runner" — and because the release job declared

```yaml
needs: [build-linux-x64, build-linux-arm64, build-macos-arm64, build-macos-x64]
```

it stayed blocked until the whole run was cancelled.

## Root cause

GitHub has been **retiring the Intel (`x86_64`) macOS hosted runners**. The
`macos-13` label is the last Intel image and its capacity is shrinking; a job
requesting it can sit in the "awaiting a runner" queue indefinitely. The old
workflow used a separate native runner per platform precisely so each job could
smoke-test its own binary — but that made the release fatally dependent on a
runner that no longer reliably exists.

## Fix

`bun build --compile` cross-compiles to **every** target triple from a single
host (verified locally: from Linux x86_64 it produces valid Mach-O `arm64` and
`x86_64` binaries and an ELF `aarch64` binary). So the four per-OS build jobs
were collapsed into **one `build` job on `ubuntu-latest`** — an
always-available runner — that cross-compiles all four Tier 1 targets:

```sh
bun run cli/scripts/build-binary.ts \
  linux-x86_64 linux-aarch64 macos-x86_64 macos-aarch64
```

The native `linux-x86_64` binary is smoke-tested in-job (it validates the
bundle + compile path; the other three are the same bundle cross-targeted).
The `release` job now `needs: [build]` only. Tarball names and the
`sgl-<tag>-<platform>/sgl` layout are unchanged, so `scripts/install.sh` and
`sgl update` keep working.

## Follow-ups (optional)

- Post-release, `acceptance.yml` already validates the install-to-running
  flow on real `ubuntu-latest` / `macos-latest` runners, so the cross-compiled
  binaries still get exercised on genuine hardware after publish.
- If per-OS native smoke at build time is ever wanted again, add a `macos-latest`
  (arm64) job — but keep it **non-blocking** for the publish step so a runner
  outage can never again wedge a release.
