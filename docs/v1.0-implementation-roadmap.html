The hard v1.0 product gate: lift web/bun coverage from ~10% to 100%. Bindgen, object handles, and Asyncify @await reach ~80% without monomorphization; the closures keystone (C0→C1→C2) needs it, and C2 closes the gate. ADR 0017 → 0018 → 0019.

Sequencing notes — the three couplings that set the order

1 · Monomorphization ↔ closures (the join)

  • M0 is a v1.0 item, not v1.1. Closures C1 specialize the captured-env type per concrete shape using exactly the ADR-0001 comptime monomorphization machinery. So the substrate (M0) must land before C1 — even though the container payoff (M1: Vec[T]/HashMap) is v1.1. The ADR-0016 combinators (vec_map[T,U](v, cb)) are the join of M1 + C1: they need container mono and closures.

2 · FFI ↔ closures (the keystone)

  • Bindgen + object handles + Asyncify @await reach ~80% with no closures. But callbacks/events (~80→97%) and the poll-reactor (~97→100%) are closures — which is why ADR 0018 pulls them forward as a v1.0 keystone. C2 (escaping, wasm-gc-only) is the highest-risk single item and the point the gate closes. It needs wasm-gc (present in Bun/JSC), not JSPI (absent) — independent of the async mechanism.

3 · Capability model ↔ v1.0 (the decoupling)

  • The capability model is off the v1.0 critical path. ADR 0019 v1.0 closures restrict to by-value/ immutable capture precisely so they need no borrow checker. Within the capability track the order is strict (K1 on::check → K2 reflection → checker/ocaps → optimizer), and the borrow checker (K8, ADR 0011) is explicitly “extremely low priority” (v1.y) — its payoff is unlocking mutable capture in escaping closures and data-race-free concurrency.
