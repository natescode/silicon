⬡ Sigil / Round 46 / Design Plan

Module System & Namespace APIs

Replace flat web_* names with web::* namespace syntax and introduce a simple file-system module system

Goal

Silicon's host API calls currently use a flat naming convention with an underscore prefix (web_console_log_str). The goal is a namespace syntax (web::console_log) backed by a real module system — so that adding a new host binding or a third-party module follows a consistent, discoverable pattern.

Before (Round 45) @extern web_console_log_str ptr:String;
&web_console_log_str result;
→
After (Round 46) &web::console_log result;

The @extern declaration disappears from user code entirely — it moves into the module's own .si file, just like a C header. The compiler reads the module file, resolves the type, and emits the WASM import automatically.

Free win: The WASM import namespace is already correct. lowerExtern in lower.ts already emits (import "web" "console_log_str" …) and the playground already provides { web: { console_log_str: … } }. The JS side needs no changes — only the Silicon source and compiler module layer change.

Design Decisions

✓ D1 — Directory: modules/ (visible)

Third-party modules live in a visible modules/ directory at the project root. Modules are downloaded and managed by hand — the directory should be easy to find and browse. Whether to commit it or gitignore it is up to the project.

✓ D2 — Imports: auto-resolve, full explicit namespaces, no aliasing

No @use declaration required. The namespace in the call site is the import — &web::console_log auto-resolves the web module. Namespaces must always be written in full; there is no aliasing (use Draw as D is a non-goal for this round). Compiler error if the namespace does not match any registered module.

✓ D3 — Layout: both single-file and folder modules supported

The compiler checks modules/Draw.si first, then modules/Draw/Draw.si. Single-file keeps small, thin host-import wrappers as simple as possible. The folder form supports larger modules that need a WAT implementation, assets, scripts, or other supporting files alongside the header. Both coexist in the same modules/ directory.

Architecture

Module resolution order

When the compiler sees web::console_log it resolves the module in this priority order:

PrioritySourceExampleNotes
1 Built-in env modules web::, env::, node:: Always available. Defined in src/strata/modules/. Map to WASM host imports.
2 User modules on disk Draw::, Math::, Json:: Downloaded by hand into ./modules/. May have a Silicon or WAT implementation.

Compiler pipeline position

Module loading slots in right after strata loading, before elaboration:

Parse → AST → buildStrataRegistry → loadModules → ModuleRegistry → elaborate → TypeCheck → Codegen → WAT

Directory layout

project/
  main.si ← user source; no @extern declarations needed
  modules/ ← third-party modules (manual download)
    Draw.si ← single-file module (thin host-import wrapper)
    Math/ ← folder module (has a WAT implementation)
      Math.si ← header: @extern and @fn declarations
      impl.wat ← WAT implementation (inlined by codegen)

src/strata/modules/ ← built-in env modules (ship with the compiler)
  web.si ← web:: namespace (browser/playground host APIs)
  env.si ← env:: namespace (raw WASM runtime, print/read)

Module file format

A module .si file is a Silicon source file containing only declarations — no @module header is needed. The folder or file name is the module name. Inside, functions are declared without the module prefix: 

; src/strata/modules/web.si  — the built-in web:: environment module
; Functions here become (import "web" "name" …) in WAT.

@extern console_log     v:Int;
@extern console_log_f   v:Float;
@extern console_log_str ptr:String;
@extern console_error   v:Int;
@extern console_warn    v:Int;
; modules/Draw.si  — a hypothetical third-party drawing module
; Host must provide { Draw: { clear, fill_rect, … } } in the WASM import object.

@extern clear      () -> Void;
@extern fill_rect  (x:Int, y:Int, w:Int, h:Int) -> Void;
@extern set_color  (r:Int, g:Int, b:Int) -> Void;

Calling a module function

In user Silicon code, no extern declaration is needed — just use the namespaced call:

@let greet msg:String := {
    @local result := ('Hello, ' ++ msg);
    &web::console_log_str result;   ; no @extern needed — compiler reads web.si
    result
};
Grammar note: The Silicon grammar's namespace rule already handles web::console_log_str via the ("::" | ".") identifier repetition. And FunctionCall = "&" FunctionCallBody already accepts a namespace as the callee. No grammar changes are needed.

Implementation Steps

1
ModuleEntry type and ModuleRegistry
src/modules/registry.ts

Define a ModuleEntry — the parsed form of a module .si file — and a ModuleRegistry that maps module names to their entries. 

export interface ModuleEntry {
  name: string                   // "web", "Draw", …
  kind: 'env' | 'user'          // env = host import; user = disk module
  functions: Map<string, FnSig>  // "console_log_str" → { params, result }
}

export interface FnSig {
  params: WasmValType[]
  result: WasmValType | undefined
}

ModuleRegistry is a Map<string, ModuleEntry>. Lookup: registry.get('web')?.functions.get('console_log_str').

2
Module loader — built-in env modules
src/modules/loader.ts
src/strata/modules/web.si

Create loadModules(modulesDir?: string): ModuleRegistry. Phase A: parse all .si files in src/strata/modules/ and register them as kind: 'env'. Phase B: scan modulesDir (defaults to ./modules/) for user modules.

Parsing a module .si file reuses the existing Silicon parser — collect @extern nodes and extract their name and type signature via the same siliconTypeNameToWasm helper already used in lower.ts. 

; src/strata/modules/web.si
@extern console_log     v:Int;
@extern console_log_f   v:Float;
@extern console_log_str ptr:String;
@extern console_error   v:Int;
@extern console_warn    v:Int;
depends on step 1 (ModuleRegistry type)
3
Thread ModuleRegistry through the compiler
src/index.ts
src/codegen/compile.ts

Call loadModules() in src/index.ts after strata loading, before elaboration. Pass the registry into compile() so codegen can consult it. 

  const registry = buildStrataRegistry(ast, extraSources)
+ const moduleRegistry = loadModules()
  const ir = lowerProgram(ast, registry)
  const wat = compile(ir, registry, moduleRegistry)
depends on step 2 (loader)
4
Namespaced call resolution in the IR lowerer
src/ir/lower.ts

When the lowerer encounters a FunctionCall whose callee is a Namespace node with multiple path segments (e.g. ['web', 'console_log_str']), it looks up the function in the ModuleRegistry to obtain the type signature and emits an IRImport (or reuses an existing one).

Replace the current web_* underscore-detection regex in lowerExtern with a proper namespace split: 

- const nsMatch = /^(web|node)_(.+)$/.exec(name)
- const env   = nsMatch ? nsMatch[1] : 'env'
- const field = nsMatch ? nsMatch[2] : name
+ const sepIdx = name.indexOf('::')
+ const env   = sepIdx !== -1 ? name.slice(0, sepIdx)       : 'env'
+ const field = sepIdx !== -1 ? name.slice(sepIdx + 2)      : name

Auto-generated imports from module registry calls (step below) go through the same IRImport path — deduplication is handled by the existing module-level import collector in codegen.

depends on step 3 (registry in scope)
5
Auto-emit WASM imports for namespaced calls
src/ir/lower.ts
src/codegen/compile.ts

When codegen sees &web::console_log_str msg, the callee namespace path is ['web', 'console_log_str']. Look up web in the module registry → found as an env module → look up console_log_str → get its signature → emit (import "web" "console_log_str" (func $web__console_log_str (param i32))).

The internal WAT name for a namespaced function is $moduleName__funcName (double underscore avoids collision with user-defined functions). The call site emits (call $web__console_log_str …). 

; emitted WAT for &web::console_log_str result
(import "web" "console_log_str" (func $web__console_log_str (param i32)))
…
(call $web__console_log_str (local.get $result))
depends on step 4 (namespace resolution)
6
Update built-in Silicon source — remove @extern declarations
playground/index.html
src/e2e/ test fixtures

  @let greet msg:String := {

docs / README

Non-Goals

Deliberately out of scope for this round:

  • Package management (download, install, lock files)
  • Versioning or semver
  • Circular dependency detection
  • Module-namespaced types or structs
  • Module-level strata definitions (a module that ships its own operators)
  • Re-exporting — a module cannot re-export another module's functions
