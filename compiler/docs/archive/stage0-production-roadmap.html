Sigil Stage 0 — Production-Grade Roadmap

Ordered user stories grouped by feature. Goal: take the TypeScript Stage 0 compiler from "works for the maintainer" to "an outside developer picks it up and ships software with it." Companion to bootstrap-audit.html; this doc focuses on Stage 0, the bootstrap audit on stage1.

Scales

PriorityMeaning

P1High — developer hits within first non-trivial project. Required for serious adoption.
P2Medium — quality-of-life. Differentiates a "respectable" compiler from a "modern" one.
P3Low — nice-to-have, post-1.0. Polish, ecosystem, advanced features.

1. Error reporting & diagnostics

The single most user-visible part of a compiler. Modern languages (Rust, Elm, Swift) compete on diagnostic quality; Silicon's current messages are functional but plain. Stage 0 already has line/col info; this group makes that info actionable.

DIAG-1 Source-line excerpt with caret pointer P0 S

As a Silicon developer, I want each error to include the source line and a caret pointing at the offending token, so that I don't have to open the file and count to find the location.

Today: [Mismatch] 12:5: expected Int, got Float

Target:

error[E0102]: type mismatch
  --> src/main.si:12:5
   |
12 |     @let count:Int := 3.14;
   |     ^^^^^^^^^^^^^^^^^^^^^^ expected Int, got Float
   |
  1. Error output includes file path, line, column.
  2. Source line is rendered with line number gutter.
  3. Caret span matches the AST node's source range (start col → end col).
  4. Multi-line spans render the first + last lines with ... elision.
DIAG-2 Color-coded terminal output P0 XS needs DIAG-1

As a Silicon developer, I want errors color-coded (red for errors, yellow for warnings, blue for notes, dim grey for gutters), so the important information jumps out at a glance.

  1. ANSI escape codes when stderr is a TTY; plain text when piped.
  2. --color=auto|always|never flag respects user preference.
  3. NO_COLOR env var disables color (industry standard).
DIAG-3 Stable error codes with documentation pages P1 M

As a Silicon developer, I want every diagnostic to carry a stable E#### code that I can look up online, so I can find the canonical explanation and example fix.

  1. Every TypeError / ElaborationError / parse error gets a code.
  2. docs/errors/E0001.md … one page per code with: explanation, example failing code, example fix, related codes.
  3. Error output includes a link: For more, see https://silicon-lang.dev/errors/E0102.
  4. silicon explain E0102 prints the doc page locally.
DIAG-4 "Did you mean" suggestions for typos P1 M

As a Silicon developer, I want the compiler to suggest the closest valid identifier / type / keyword when I make a typo.

Example: @let x:Itn := 5; → note: did you mean Int?

  1. Levenshtein distance ≤ 2 against in-scope identifiers / known types / registered keywords.
  2. Suggestion rendered as a separate note line under the main error.
  3. Suggestion only fires when there is exactly one obvious candidate (avoid noise).
DIAG-5 Cross-file error chains for @use imports P1 S

As a Silicon developer, I want errors from @use'd files to show the import chain that led to them, so I can navigate from "consumer.si compile failed" to the root cause in a dependency.

  1. Errors record the @use stack at the time of import.
  2. Output prefixes each cross-file error with note: imported from a.si:3 → b.si:7 → c.si.
DIAG-6 Machine-readable JSON output P2 S

As an editor integrator (LSP, CI, custom tools), I want a --message-format=json flag that emits one JSON object per diagnostic on stdout, so I can consume diagnostics programmatically without parsing the pretty-printed text.

  1. Schema documented at docs/diagnostic-schema.json.
  2. Includes file, span (line/col + byte offsets), severity, code, message, hint, source label.
DIAG-7 GitHub Actions annotation format P2 XS needs DIAG-6

As a CI maintainer, I want --message-format=github to emit errors as GitHub Actions annotations, so failures land as inline annotations in the PR diff view.

  1. Outputs ::error file=src/main.si,line=12,col=5::expected Int, got Float.
  2. Documented in .github/workflows/test.yml example.
DIAG-8 Parse-error recovery for multiple errors per file P2 L

As a Silicon developer, I want the parser to recover from one syntax error and continue, so the compiler reports as many errors as it can find in a single run.

  1. Parser sync points: ;, top-level keyword, }.
  2. Reports up to N errors per file (default 16, configurable) before bailing.
  3. Doesn't cascade — a parse error doesn't generate fake type errors downstream.

2. Developer tooling

Beyond the compiler binary itself. Modern language ecosystems expect a formatter, scaffolder, doc generator, and watch mode in the box.

TOOL-1 silicon fmt — canonical formatter P0 M

As a Silicon developer working on a team, I want a deterministic formatter so we stop bikeshedding style and diffs stay minimal.

  1. silicon fmt src/ formats every .si file in place.
  2. silicon fmt --check src/ exits non-zero on any deviation (for CI).
  3. Idempotent: fmt(fmt(x)) == fmt(x).
  4. Preserves # line comments and ## doc comments.
  5. No configuration knobs (Go-style — opinionated).
TOOL-2 silicon new — project scaffolding P1 S

As a new Silicon user, I want one command to generate a working starter project, so I'm not copying boilerplate from a README.

  1. silicon new my-app creates a directory with: silicon.toml, src/main.si (hello-world template), .gitignore, README.md.
  2. silicon new --lib my-lib for libraries (different template).
  3. Generated project compiles + runs immediately: silicon run.
TOOL-3 silicon watch — rebuild on save P1 S amortized by PERF-1

As a Silicon developer, I want the compiler to watch source files and rebuild on save, so my dev loop is < 500 ms.

  1. silicon watch [path] uses chokidar (or Bun's --watch) to react to .si changes.
  2. Re-runs the full pipeline; later combined with incremental cache (PERF-1) for sub-second response.
  3. Clear-screen on each rebuild; persistent error list at the bottom until clean.
TOOL-4 silicon check — typecheck without codegen P1 XS

As a Silicon developer in an editor, I want a fast "did this compile?" command that skips codegen and just runs parse → elaborate → typecheck.

  1. Outputs diagnostics only; no WAT/WASM artifacts.
  2. 2–3× faster than full silicon build on cold cache.
  3. Used by the LSP for save-time validation.
TOOL-5 silicon doc — HTML docs from ## comments P2 M

As a library author, I want a tool that generates browseable HTML documentation from my source's ## doc comments, so consumers can read my API without opening the source.

  1. Walks the AST, extracts ## blocks preceding each definition.
  2. Renders Markdown inside doc bodies.
  3. Cross-links references to types and functions.
  4. Output: docs/api/ static HTML.
TOOL-6 silicon repl — interactive prompt P2 L

As a Silicon learner, I want a REPL where I can type expressions and see their values, so I can experiment without writing a full program.

  1. Wraps each input in a synthetic _start, compiles to WASM, runs under an in-process wasmtime, prints result.
  2. State accumulates: @let x := 5 then x + 1 works.
  3. Handles incomplete input (newline continuation on unmatched brace).
TOOL-7 silicon explain — offline error code lookup P3 XS needs DIAG-3

As an offline Silicon developer, I want silicon explain E0102 to print the local doc page for an error code without opening a browser.

  1. Embeds docs/errors/*.md into the binary.
  2. Renders Markdown to ANSI in the terminal.

3. Standard library

Today std.wat ships only a bump allocator + a few I/O primitives. A modern stdlib covers strings, collections, math, I/O abstractions, serialization, and testing. Each entry below is a separate Silicon module under std/*.si.

STD-1 Robust string operations P0 M

As a Silicon developer, I want std::string with concat, length, slice, indexOf, replace, split, trim, toUpper/toLower, repeat, contains, startsWith/endsWith, parseInt, parseFloat.

  1. All operations UTF-8 correct (not just byte arrays).
  2. Indices in bytes by default; chars() iterator for code points.
  3. Documented edge cases for malformed UTF-8 input.
STD-2 Collections: List, Map, Set P1 L needs LANG-2 (generics) for ergonomics

As a Silicon developer, I want generic List/Map/Set in the stdlib, so I don't reach for raw arrays for every collection task.

  1. List<T>: dynamic array with push/pop/get/set/len/iter.
  2. Map<K, V>: hash map; K constrained to hashable types.
  3. Set<T>: hash set.
  4. Iterator protocol (std::iter) so they compose with for-each.
STD-3 Testing framework: std::test P1 M

As a Silicon developer, I want a built-in testing framework so I don't roll my own harness.

  1. @test attribute marks a function as a test case.
  2. silicon test discovers + compiles + runs every @test in the project.
  3. Assertion API: assert_eq, assert_ne, assert, assert_panics.
  4. Pretty output: green checkmarks, red X with assertion message + line.
  5. silicon test --filter foo runs matching tests only.
STD-4 Math: std::math P1 S

As a Silicon developer, I want std::math with sqrt, pow, log, exp, sin/cos/tan, abs, min, max, floor, ceil, round, PI, E.

  1. Float-typed; uses WASM intrinsics where they exist (sqrt, abs, floor, ceil).
  2. Pure-Silicon implementations for transcendentals (Taylor / Pade / Remez).
STD-5 I/O abstractions: std::io P2 M

As a Silicon developer, I want higher-level I/O abstractions (open/read/write/close, file iterators, formatted print) rather than raw WASI fd_write calls.

  1. std::io::open(path, mode) -> File.
  2. std::io::read_to_string(path) -> String.
  3. println(fmt, args...) (depends on STD-6 format strings).
  4. Automatically declares the WASI externs so users don't have to.
STD-6 Format strings: std::fmt P2 M

As a Silicon developer, I want printf-style formatting ({}, {:d}, {:.2f}) so I can compose output without manual concatenation.

  1. std::fmt::format("x = {}, y = {:.2f}", x, y) returns a String.
  2. Type-checked at compile time: number of args matches placeholders, types match specs.
  3. Supports Debug trait equivalent for user types (depends on LANG-7).
STD-7 JSON encode / decode: std::json P2 M

As a Silicon developer, I want JSON support so Silicon programs can talk to the web / tooling without inventing a wire format.

  1. std::json::parse(s) -> Value; Value is a sum type of Null/Bool/Number/String/Array/Object.
  2. std::json::stringify(v) -> String.
  3. Schema-derived encode/decode for user types (depends on LANG-7).
STD-8 Date / time: std::time P3 S

As a Silicon developer, I want monotonic + wall-clock time access, basic Duration arithmetic, and a Date type.

  1. Backed by WASI clock_time_get.
  2. std::time::now(), std::time::Duration, basic ISO 8601 formatting.
STD-9 Random: std::random P3 S

As a Silicon developer, I want a seedable PRNG with uniform int / float / range / pick helpers.

  1. WASI random_get for seeding entropy; xoshiro256** for the algorithm.
  2. Reproducible from a seed for testing.

4. Module system & package management

Today @use handles relative-path imports only. To go from "single developer" to "ecosystem", Silicon needs a package manifest, a way to depend on remote packages, and a lockfile. These stories define what that looks like.

PKG-1 Package manifest: silicon.toml P1 M

As a Silicon developer, I want a project-level manifest that declares package name, version, entry point, and dependencies, so the compiler knows the project structure without flag-passing.

  1. TOML format at the project root.
  2. Keys: name, version, main (or lib), silicon (compiler version), dependencies.
  3. silicon build discovers it walking up from cwd.
PKG-2 Lockfile: silicon.lock P1 S needs PKG-1, PKG-3

As a team, I want a lockfile committed alongside the manifest so every build pins the exact same dependency versions.

  1. Resolved dependency graph + content hashes for each package.
  2. silicon build uses the lock; silicon update regenerates it.
PKG-3 Remote dependencies via git P2 L needs PKG-1

As a Silicon developer, I want to depend on a package hosted in a git repo, identified by URL + git ref, so I can share code across projects.

  1. silicon.toml accepts foo = { git = "...", tag = "v1.0" }.
  2. silicon build clones into ~/.silicon/cache/ and adds to the @use search path.
  3. SHA pinned in lockfile.
PKG-4 Workspaces: multi-package repos P2 M needs PKG-1

As a large-project maintainer, I want to define a workspace with multiple packages in one repo so they can depend on each other locally.

  1. Root silicon.toml with [workspace] members = ["pkg-a", "pkg-b"].
  2. Member packages can declare { path = "../pkg-a" } dependencies.
  3. silicon build --workspace builds all members.
PKG-5 Public registry: silicon-lang.dev/packages P3 XXL needs PKG-1, PKG-3, infra

As an ecosystem participant, I want a central registry where I can publish + discover Silicon packages, so adoption isn't gated on finding each other on GitHub.

  1. Registry service (npmjs-style) accepting silicon publish uploads.
  2. silicon.toml accepts foo = "^1.0" short-form.
  3. Web UI for browsing + searching.
  4. Cargo / npm-style version resolution (semver, ranges).

5. Language features

Silicon's surface is intentionally minimal and grammar-frozen; new features ride the strata system. Some features below are achievable as strata; others need core compiler support. Each story flags which.

LANG-1 Generic types (parametric polymorphism) P1 XL core compiler change

As a Silicon developer, I want to write @fn first<T> xs:List<T> := { ... }, so collection types and helpers don't require copy-paste-per-type.

  1. Type-parameter syntax in @fn, @let, @type.
  2. Monomorphization at call sites (no boxing / dynamic dispatch).
  3. Type inference at the call site doesn't require explicit ::<T>.
  4. Bounded generics (T: Hashable) require LANG-3.
LANG-2 First-class functions / closures P2 XL core compiler change

As a Silicon developer, I want to pass functions as values and write closures that capture lexical scope, so I can write list.map(fn).

  1. Function type syntax: (Int, Int) -> Int.
  2. Anonymous function literal: fn(x) { x + 1 } or similar.
  3. Closures box captured locals; cleanup handled by the bump arena (not GC).
  4. WASM lowering via call_indirect + a function table.
LANG-3 Traits / interfaces P2 XL needs LANG-1

As a library author, I want to define traits (Display, Hashable, Eq) and implement them per type, so generic code can constrain its type parameters.

  1. @trait Display { @fn fmt:String self; } declares.
  2. @impl Display for Int := { ... } provides.
  3. Trait bounds: @fn show<T: Display> x:T := ....
  4. Monomorphized; no vtables unless explicitly boxed.
LANG-4 Pattern-match guards P2 S

As a Silicon developer, I want when clauses on match arms so I can constrain pattern matches with arbitrary boolean predicates.

&@match x,
  $Some n when n > 0, { ... },
  $Some n,             { ... },
  $None,               { ... }
  1. Parser accepts when expr after a pattern.
  2. Lowering: guard becomes an inner @if.
  3. Exhaustiveness check ignores guarded arms.
LANG-5 Records / named tuples P2 M

As a Silicon developer, I want a way to declare a struct-like type with named fields without going through @type's sum-with-single-variant form.

  1. @record Point := x:Int, y:Int;.
  2. Field access via dot: p.x.
  3. Construct via $Point x:1, y:2 (consistent with variant syntax).
LANG-6 Result / Option in std (Rust-style) P3 S needs LANG-1

As a Silicon developer, I want Option<T> and Result<T, E> in std as the canonical way to model absence and failure.

  1. Defined as @type Option<T> := $None | $Some T;.
  2. Standard methods: unwrap, unwrap_or, map, and_then, is_some, is_none.
  3. ? operator for early-return propagation (sugar, parser-level).
LANG-7 Iterators / for-each P3 L needs LANG-1, LANG-3

As a Silicon developer, I want a for x in collection form that desugars to iterator-protocol method calls.

LANG-8 Async / await P3 XXL

As a Silicon developer targeting WASI 0.3 or the Component Model, I want async functions so I can integrate with async hosts. Deliberately late-stage — async is the single hardest feature to get right and shouldn't land before the core is stable.

6. Compiler performance

Stage 0 is fast enough today for projects under a few kLoC. These stories matter once Silicon projects get to tens of thousands of lines + editor integration.

PERF-1 Incremental compilation cache P2 L

As a Silicon developer with a large project, I want unchanged files to skip recompilation, so editing one file doesn't rebuild the world.

  1. Cache key: SHA256 of source bytes + compiler version + dependency cache keys.
  2. Cache at target/incremental/<hash>.json: AST + types + emitted WAT.
  3. Invalidation: a file's downstream consumers re-typecheck when the file's exported signatures change (not its body).
PERF-2 Parallel module compilation P2 M needs PERF-1

As a Silicon developer, I want independent modules to compile in parallel, using all available cores.

  1. Build a module dependency DAG; compile in topo order, with siblings in parallel.
  2. Worker pool sized to nproc by default.
PERF-3 WAT output optimizer P3 L

As a Silicon developer shipping a binary, I want the compiler to apply standard WAT-level optimizations (constant folding, dead code elimination, inlining hot calls) so my output is competitive in size and speed.

  1. silicon build --release runs optimizer passes.
  2. Optionally pipe through binaryen's wasm-opt for the heavy lifting.

7. Distribution & packaging

Today users install Sigil by cloning the repo + running bun install. A production compiler is one binary download.

DIST-1 Single-binary releases per OS P1 M

As a new Silicon user, I want one binary download per platform (macOS arm64/x64, Linux arm64/x64, Windows x64) so I'm not installing a JS runtime.

  1. Bun's bun build --compile emits standalone binaries.
  2. GitHub Releases artifact for each platform.
  3. Binary embeds the grammar + std.wat as bundled assets.
DIST-2 npm package: silicon-cli P1 S

As a developer with Node already installed, I want npm install -g silicon-cli to install the compiler.

  1. npm package wraps the platform binary (postinstall fetches the right one).
  2. silicon command on PATH.
DIST-3 Homebrew formula P2 XS needs DIST-1

As a macOS / Linux user, I want brew install silicon.

  1. Tap or core formula references the GitHub Release binary.
  2. Auto-updated on each release via a release hook.
DIST-4 Scoop manifest (Windows) P2 XS needs DIST-1

As a Windows user, I want scoop install silicon.

DIST-5 Docker image P3 XS

As a CI user, I want a pinned ghcr.io/natescode/silicon:v0.3 image with the compiler + wasmtime preinstalled.

DIST-6 Release automation P2 S

As a maintainer, I want git tag v0.3.0 && git push --tags to trigger: build binaries, publish npm, push Docker, update Homebrew/Scoop, generate changelog, draft GitHub Release.

8. Testing & quality assurance

Today Stage 0 has unit tests + property tests + a fuzz harness. Production needs an integration corpus, snapshot tests for WAT output, and continuous coverage measurement.

TEST-1 Integration test corpus P0 M

As a compiler contributor, I want a corpus of .si programs each with an expected exit code / stdout, so regressions in code generation get caught automatically.

  1. tests/integration/<name>.si + tests/integration/<name>.expected.
  2. Test runner compiles + runs each + diffs.
  3. Initial corpus ≥ 50 programs covering: arith, control, strings, arrays, sum types, generics (once landed), I/O, std lib.
  4. Catches the bootstrap-audit findings C-2 / C-3 / C-4 if anyone tries to delete src/ again.
TEST-2 WAT snapshot tests P1 S

As a codegen reviewer, I want each fixture program's WAT output snapshotted under git, so semantic-neutral output changes still surface in review.

  1. SILICON_SNAPSHOT_UPDATE=1 bun test regenerates snapshots.
  2. Default bun test diffs against committed snapshots.
TEST-3 Coverage measurement P1 XS

As a maintainer, I want test coverage measured per PR so I can spot drops.

  1. Bun's --coverage emits LCOV.
  2. CI publishes the report; failing if a PR drops total coverage by > 1%.
TEST-4 Performance benchmarks + regression detection P2 M

As a maintainer, I want compile-time / output-size benchmarks measured per PR, so perf regressions surface before merge.

TEST-5 Differential testing vs stage1 P2 S

As a maintainer, I want the integration corpus also run through stage1.wasm + diff against Stage 0 output, so the two pipelines stay in sync (or divergences are explicit).

9. WASM target maturity

Silicon targets wasm32 + WASI preview1 today. These stories bring it into the modern WASM ecosystem.

WASM-1 Source maps for debugging P2 M

As a Silicon developer debugging in Chrome DevTools / Firefox, I want WASM source maps so stack traces point at .si lines, not WAT instructions.

  1. Emit a sourceMappingURL custom section.
  2. Each IR node carries its SourceLocation through to the emitter.
WASM-2 DWARF debug info P3 L needs WASM-1

As a Silicon developer using lldb / gdb on wasmtime, I want full DWARF debug info including local variable names and line tables.

WASM-3 Tail-call optimization P3 S

As a Silicon developer writing recursive functions, I want guaranteed tail-call elimination via the WASM tail_call proposal, so I can write iterative algorithms recursively without blowing the stack.

WASM-4 WASI preview2 / Component Model P3 XXL

As a Silicon developer targeting modern WASM hosts, I want preview2 + Component Model support so Silicon-compiled components compose with other-language components.

WASM-5 WASM-GC reference types P3 XL

As a Silicon developer, I want strings and aggregates to use WASM-GC reference types instead of the bump arena, so my programs play nice with host GC.

WASM-6 SIMD intrinsics P3 M

As a Silicon developer writing numerics, I want WASM::v128_* intrinsics so I can hand-vectorize hot loops.

10. Documentation & learning resources

Compilers without docs don't get adopted. These stories cover what a new developer reads in their first week.

DOC-1 Getting Started tutorial P0 S

As a new Silicon user, I want a 15-minute tutorial that takes me from install → hello world → a small CLI tool with tests + dependencies, so I can decide whether to invest more time in the language.

  1. Single page; runnable code samples at every step.
  2. Ends with "what to read next" links.
DOC-2 Language reference P1 L

As a Silicon developer, I want a formal language reference that documents every syntactic form + its semantics, so I can resolve ambiguities without reading the compiler source.

  1. One page per construct.
  2. EBNF grammar fragment at the top of each construct's page.
  3. Cross-references to the relevant strata + lowering.
DOC-3 Cookbook / examples library P1 M

As a Silicon developer, I want a cookbook of recipes (read a file, parse JSON, walk a directory, simple HTTP, etc.), so I can copy-paste a starting point instead of figuring out the std API from scratch.

DOC-4 Strata authoring guide P2 S

As a library author, I want a guide that walks me through adding a new keyword via @stratum_keyword, with worked examples (@nz, @const, @loc) and the available &Compiler::* surface.

DOC-5 Compiler internals doc P2 M

As a compiler contributor, I want a document explaining the pipeline (parse → AST → elaborate → typecheck → IR → emit) with file-level pointers, so I can find where to make a change.

  1. CLAUDE.md already covers the bootstrap side; this is the Stage 0 equivalent.

11. CI / DevOps

The infrastructure that makes "production grade" durable rather than point-in-time.

CI-1 GitHub Actions: test on every PR P0 XS

As a maintainer, I want CI to run on every PR, blocking merge until tests pass.

  1. .github/workflows/test.yml runs bun test on Ubuntu + macOS + Windows.
  2. Required status check on the main branch protection.
CI-2 Type-check + format-check + lint on PR P1 XS needs TOOL-1

As a maintainer, I want silicon fmt --check + tsc --noEmit + ESLint enforced in CI, so style stays consistent automatically.

CI-3 Bootstrap fixed-point check in CI P1 XS

As a maintainer, I want ./build.sh check run in CI on every PR, so the byte-equal self-host gate doesn't quietly break.

CI-4 Dependency / security audit P2 XS

As a maintainer, I want Dependabot + bun audit on every PR + weekly scheduled scans, so CVEs in dependencies surface promptly.

CI-5 Changelog automation P2 S

As a maintainer, I want commit messages following Conventional Commits to drive automatic CHANGELOG generation on release, so I'm not hand-writing release notes.

Stories ordered within each group by priority then by what unblocks the most downstream work. An ambitious team could close the P0 set (DIAG-1, DIAG-2, TOOL-1, STD-1, TEST-1, DOC-1, CI-1) in roughly three engineer-weeks; that alone moves Sigil from "interesting prototype" to "credibly usable for a small project." P1 stories bring the experience to early-adopter parity with modern languages; P2+ is post-1.0 polish and ecosystem work.
