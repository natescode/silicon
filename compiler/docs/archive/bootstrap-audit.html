Sigil Bootstrap — Staff Engineer Audit

Honest assessment of wasm-bin/stage1.wasm as a production compiler. Written at the level a senior compiler engineer would deliver to an eng-leadership audience after a half-day deep-dive. Date: 2026-05-19. Subject commit: natescode/sigil@phase-7.

TL;DR

Not currently feasible as a production compiler. Genuinely impressive as a proof-of-concept bootstrap. Path to production is real and bounded (~6–10 focused engineer-weeks), but the gap is bigger than "polish".

  • REAL WIN The byte-equal self-host fixed point holds. Stage1 compiles its own 10.6 kLoC source to byte-identical stage1.wasm. That gate is rare in the industry and is load-bearing for trust in the binary.
  • CRITICAL The parser does not report syntax errors. Garbage input silently produces a no-op WAT module. This is the single most disqualifying issue for a self-respecting compiler.
  • CRITICAL Surface language features that worked in the TypeScript-era Stage 0 are broken in stage1: @match destructuring, @enum variant constants, half of the std helpers (arr_load_i32, alloc_array, print_int). Anyone trying to port a real program will hit these immediately.
  • HIGH Type diagnostics have no line/column. The typechecker is also opt-in and warn-only — it never refuses to compile.
  • HIGH Subprojects (silicon-playground, silicon-vscode) carry frozen copies of the TS compiler. That works today but creates three sources of truth for the language; coordinating changes is manual.

Use the bootstrap for: demonstrating self-hosting, the strata extension system, the WASI compile pipeline. Do not use it for: compiling user programs of any non-trivial size, anyone who isn't deeply familiar with the project's internals, or as the canonical compiler for the Silicon language until the issues below are closed.

1. Scope & methodology

This audit evaluates wasm-bin/stage1.wasm — the Silicon-implemented Silicon compiler — against the standard a staff compiler engineer would apply when deciding whether to declare it the canonical implementation of the Silicon language. Out of scope: the TS Stage 0 (deleted), the playground, the LSP/VS Code extension. The language design itself is taken as given.

The session:

  • Counted lines of Silicon source per module (with and without tests)
  • Read the parser (boot/parser/parse.si), the IR + WAT emitter (boot/ir/lower.si, boot/emit/wat.si), and the typechecker port we just landed (boot/types/*.si)
  • Compiled and ran 7 deliberate user programs — recursion, loop+ mutation, sum types, enums, arrays, type-error, string I/O, ill-formed garbage — through stage1.wasm + wat2wasm + wasmtime end-to-end
  • Disassembled the emitted stage1.wat to inspect the std-runtime surface it actually ships
  • Cross-checked against the TS-era src/codegen/std.wat that lives in the playground snapshot to identify what regressed
  • Ran the project's own ./test.sh (20 tests, all green) and ./build.sh check (byte-equal fixed point holds)

3. What's genuinely good

S-1. Byte-equal self-host gate SHIP

stage1.wasm recompiles its own source byte-for-byte. This is non-trivial — it requires deterministic codegen (no hash-randomized iteration order, no timestamp leakage, no allocator- address-dependent output). Hitting this gate at all is a stronger signal of compiler correctness than any number of unit tests, and the Silicon authors should be proud of it.

S-2. The strata extension system works PROVEN

The three user-defined keywords landed via strata only (@nz, @const, @loc) demonstrate that the "add a feature without touching the compiler" loop is real. This is genuinely rare; most "extensible" languages still need core changes for anything interesting. The body interpreter (boot/elab/body_rich.si) is the load-bearing piece and it dispatches cleanly.

S-3. Modular file layout CLEAN

The 30 source files break down sensibly: parser/, ir/, emit/, elab/, strata/, types/, compiler_api/, modules/, std/. No file is egregiously large (largest is ir/lower.si at 1,764 LoC, which is reasonable for an IR lowering pass). A new contributor can navigate the tree without an architecture diagram.

S-4. Generated WAT is correct (for the subset that lowers) PASSES

The programs that do compile produce valid WAT that wat2wasm accepts under strict validation and wasmtime executes correctly. fib(10) → 55, sum 1..100 → 5050, nested if/else, mutating loop with assignment — all behave correctly with the right exit codes. When stage1 commits to lowering a program, the lowering is sound.

S-5. Bootstrap pipeline is clean shell REPRODUCIBLE

./build.sh check on a fresh clone needs only wasmtime, wat2wasm, and bash. No npm, no Bun, no toolchain drift. This is what bootstrap is supposed to feel like.

4. Critical findings BLOCKING

Each of these would, individually, prevent a staff engineer from endorsing the compiler for external use.

C-1. Parser silently accepts arbitrary input boot/parser/parse.si

SeverityCritical — correctness Reproecho 'this is utter garbage that is not silicon' | wasmtime stage1.wasm ObservedExit 0. 42-line WAT module containing just the std prelude. ExpectedNon-zero exit, diagnostic to stderr identifying line and unexpected token.

parse.si appears to recover silently on every unexpected token, treating unrecognised top-level constructs as no-ops. There is no panic_stderr or equivalent on the parser path. Even obviously malformed programs (@let MISSING_TYPE := 5; @let x := MISSING_TYPE + ;, dangling operator, missing brace) produce a nominally-valid WAT module that just lacks the user's code.

A compiler that doesn't refuse malformed input is not a compiler — it's a code-shaped lossy filter. Users will spend hours wondering why their program "compiled" but does nothing. This is the single most urgent fix.

Fix sketch: introduce a panic_stderr at every parser-error site, with the source span. Aim for "every branch in parse_* that hits an unexpected token writes a diagnostic and proc_exit's with a non-zero code". ~200 LoC across parse.si.

C-2. @match destructuring does not lower boot/strata/builtin/match.si + boot/ir/lower.si

SeverityCritical — feature regression ReproSee audit/match.si — sum-type w/ @match on Circle/Rectangle. ObservedEmitted WAT has call $r / call $w / call $h where pattern-variable locals should be. wat2wasm rejects: "undefined function variable $r".

The pattern-variable bindings introduced by @match arms ($Circle r:Int) are lowered as if r were a function call rather than a local-variable bind. The codegen for @match's payload destructuring is incomplete.

Why this matters: sum types with payloads were a flagship language feature in the TS stage 0 (see docs/strata.md). Their loss in stage1 is not a "won't fix later" — it's a regression against an advertised feature.

C-3. @enum variant constants do not lower boot/ir/lower.si

SeverityCritical — feature regression Repro@enum Color := Red | Green | Blue; then refer to Green. ObservedEmitted WAT has call $Green where i32.const 1 should be.

The EnumDef stratum dispatches to &IR::def_type_sum, but the corresponding lowering in lower.si does not emit the per-variant immutable globals. Variant names then fall through to lower_call and become function references.

Same severity reasoning as C-2: @enum is a documented feature. Until this works, sum types are essentially absent.

C-4. std.wat is truncated — major helpers missing boot/codegen / emit_program

SeverityCritical — runtime regression ObservedStage 1 ships 6 std helpers. TS Stage 0 shipped 17. Missing: alloc_array, alloc_string, arr_len, arr_load_i32, arr_store_i32, arr_load_f32, print_int, print_bool, print_float, str_concat (partial).

The bootstrap appears to have inlined only the helpers stage1's own source needs. Anything user code might reasonably want — arrays of any kind, formatted integer output, string concatenation — is absent from the emitted runtime. The typechecker we just landed even pre-registers these helpers, suggesting they're expected to be available, but the codegen surface doesn't include them.

Fix sketch: std.wat lives in the compiler binary as a string; expand it to match the TS-era 17 helpers. The runtime cost is small (each helper is a handful of WAT instructions); the inclusion criterion currently looks like "needed for self-host" rather than "intended public API".

C-5. WASI extern declaration leak in boot/std/io.si boot/std/io.si

SeverityCritical — usability blocker DocumentedYes — this audit's prior conversation noted the issue and added a TODO comment. Not yet fixed.

Every user program that does @use '../boot/std/io.si' must manually @extern-declare wasi_snapshot_preview1::fd_write / fd_read / proc_exit, because io.si calls them without declaring them. This is the first thing every new user trips on (literally happened earlier today when compiling delfina.si).

The right fix is either to inline the extern declarations into io.si (currently blocked by duplicate-declaration when build.sh's stub injection runs) or to teach the lowerer to treat repeated identical @extern as idempotent. Until one of those lands, every example, every README snippet, every blog post about Silicon has to carry six lines of WASI boilerplate.

5. High-severity findings SHOULD-FIX-BEFORE-USERS

H-1. Diagnostics have no source location boot/types/errors.si

Every diagnostic err_write_msg produces is of the form [Mismatch] expected Int, got Float — no file:line:col prefix. The error record stores src_off / src_len (byte offsets into P_SRC), but they're never converted to line/column or printed. For a multi-file program this makes diagnostics essentially unusable — the user knows that there's a type error somewhere but not where.

The TS Stage 0 produced file.si:12:5: [Mismatch] ... format diagnostics consistently. Recovering that in stage1 needs a byte-offset → line/col converter (one pass over P_SRC building a line table) plus a format change in err_write_msg. ~50 LoC.

H-2. Typechecker is opt-in and warn-only

--typecheck is a flag that defaults off, and even when enabled it prints diagnostics to stderr without changing the compile-exit code. The intent (per the slice 2f commit) was to keep the byte-equal gate working while the checker matures; that's defensible at this stage but is not the steady state.

A real compiler refuses to emit code that it knows is type-incorrect. Path to production needs the flag flipped to default-on plus a --no-typecheck escape hatch, and a non-zero exit on diagnostic count > 0.

H-3. Typechecker misclassifies parse errors as unbound identifiers boot/types/check.si:check_namespace

Repro: echo 'totally not silicon at all' | stage1.wasm --typecheck produces five [UnboundIdentifier] diagnostics, one per bare word. The parser accepted the input (see C-1) and built an AST whose top-level elements are bare namespace references; the typechecker dutifully reports each as unbound.

Knock-on effect of C-1: if the parser refused garbage in the first place, this misclassification wouldn't happen. Worth noting that fixing C-1 likely fixes this too.

H-4. The typechecker walks function bodies with no parameter scope

preregister_defs seeds the top-level function table with each @fn's signature, but the inline check_inline_def path only handles expression-position definitions (@let / @loc inside a body). When the main walk descends into an @fn's body, the parameters themselves (a:Int, b:Int) are not pushed into the symbol table. Result: every reference to a parameter inside a function body is flagged as UnboundIdentifier.

Manifests immediately on real source. Try --typecheck on audit/fib.si — you'll get a flood of "unbound n" errors. The checker is currently only useful for top-level expressions.

Fix: check_fn_def wrapper that pushes params into the scope before recursing into the body and pops on exit. ~30 LoC.

H-5. No incremental compilation, no caching

Every stage1.wasm invocation re-parses, re-elaborates, re-typechecks, and re-lowers everything from scratch. This was fine for the bootstrap (compile-once, ship the .wasm); it scales badly as soon as you have a project with imports and an editor polling on save. The LSP already feels this pain — the silicon-vscode bundled server re-runs the full pipeline on every keystroke debounce.

Path forward: serialize an intermediate cache (P_AST + types keyed by source file hash). Not urgent, but blocks "Silicon feels nice in an editor."

6. Medium-severity findings

M-1. Error messages lack hints / fix suggestions

Modern compilers (Rust, Elm, Clang) competed for years to produce diagnostics that suggest fixes. Silicon's are bare: [Mismatch] expected Int, got Float. No "did you mean &@toInt x?" hint. Field is in the TypeError record for future use but never populated.

M-2. Test corpus emphasizes internal data structures over generated-code semantics

Of the 20 tests in ./test.sh: 7 verify generated-code semantics (the keyword tests, the runtime/data-structure tests — vec_test, arena_test). 13 verify the typechecker's internal data structures (does the SiliconType arena round-trip, does the symbol table shadow correctly). That's a meaningful test suite but it's mostly white-box. A black-box "compile this Silicon program, check the exit code or stdout" corpus (the Rust compiletest model, the Ghc/Haskell testsuite model) would catch C-2, C-3, C-4 within minutes of writing.

Recommendation: tests/integration/*.si directory, one program per language feature, with an expected exit code or stdout. ./test.sh integration would compile + run each and diff stdout.

M-3. Lots of known-bug TODOs left in source

Comments in boot/strata/builtin/logic.si document a "latent bug" where two keyword strata sharing the same first-call marker confuse the loader. boot/std/io.si documents the WASI extern issue (C-5). boot/types/check.si contains TYPE_NONE placeholder propagation that the comments acknowledge as imperfect. These are honest, but they're "incidents waiting to happen for an external user."

M-4. No tooling beyond the compiler

No formatter, no linter, no documentation generator. The silicon-vscode extension is a real win, but VS Code users get syntax highlighting + LSP and nothing else. For comparison: gofmt ships with Go 1.0; Rust started with rustdoc pre-1.0. Sigil at this maturity needs at minimum a silicon-fmt built atop the existing AST.

M-5. Vendored compiler copies in three subprojects

silicon-playground and silicon-vscode both carry frozen src/ copies of the TS Stage 0 compiler (the latter bundled into the VSIX, the former for the playground server). Any future improvement to the canonical Silicon compiler (this repo's boot/*.si) has to be ported by hand to the other two. The long-term plan is to point both at stage1.wasm, but until then we have three independently-drifting compilers branded as "Silicon."

7. Architectural risks (longer horizon)

R-1. Grammar lives in code, not in a spec

The canonical grammar is whatever boot/parser/parse.si accepts. There's no .ohm file or BNF document. "Reading parse.si to learn the language" is technically feasible but is not a substitute for a spec when implementing alternate parsers, linters, or syntax-aware tools. The TS Stage 0 had a .ohm grammar; that's gone.

R-2. No formal versioning or stability story

silicon-vscode is now v0.2.0 on the Marketplace. The compiler's boot/ source has no version stamp. When you ship breaking changes, you'll have no way to tell which .si programs were written against which compiler. Pragmatic fix: a @platform stratum that takes a silicon:1.0 argument and refuses to compile under a newer compiler unless overridden.

R-3. WASM target is implicitly wasm32

Int = i32 is hardcoded throughout lower.si. The CLAUDE.md docs cleanly describe an eventual wasm64 transition where Int = i64. Today, that transition would require touching dozens of lowering sites that pattern-match on 'i32'. Not urgent, but the abstraction is leakier than the docs suggest.

R-4. Memory growth is heap-only, never reclaimed

The bump allocator in std.wat grows the wasm memory but never shrinks. Long-running stage1 processes (LSP, watch mode) will accumulate memory across compiles. For a one-shot CLI compiler this is fine; for the LSP it'll eventually cause issues. No arena reset between compiles either — each compile sits on top of the previous.

8. Roadmap to production

Rough effort estimate assuming one focused engineer. These are ordered by what unblocks the most downstream work.

PhaseWorkEffortUnblocks
P1 Fix C-1: parser refuses malformed input with a real error 2-3 days Everything else. Without this, no honest dogfooding.
P2 Fix C-5 (WASI extern) + H-1 (line/col in diagnostics) + H-4 (fn body scope). These are individually small but block any non-trivial Silicon authoring. 2-3 days External users. At this point the docs/examples are accurate.
P3 Black-box integration test corpus + CI. ~30 programs covering arith, control, strings, arrays, sum types. Wire to ./test.sh integration. 3-4 days Catches every other regression in this audit + future ones.
P4 Fix C-2 (@match) and C-3 (@enum) — flagship features back to working. Likely 1-2 days each once P3's tests are in place to drive them. 3-4 days Sum types story restored.
P5 Fix C-4: full std.wat surface. Mechanical port from the playground's frozen copy. 1-2 days Real user programs.
P6 H-2: make --typecheck default-on, exit non-zero on errors, add --no-typecheck. Triage every check that fires on boot/*.si itself (sigil's own source needs to typecheck clean). 4-5 days Calls the compiler "type-safe" without lying.
P7 M-4: silicon-fmt built on the existing AST. 2-3 days Editor experience parity with peers.
P8 M-2 polish: hints / "did you mean" / fix suggestions in diagnostics. 5-7 days Diagnostic-quality competitive with modern languages.

Total: ~22-31 engineer-days, or 5-7 calendar weeks at a typical pace with reviews. After P5 the compiler is usable for real programs; after P6 it's defensible as the canonical implementation; after P8 it's competitive.

The work that won't go on this list because the architecture already supports it: more language features. The strata system genuinely works — anything ridable on @stratum_keyword / @stratum_operator can land without compiler changes. Future work like async, generators, or richer types is surface-level and orthogonal to the issues above.

9. Reproduction commands

Everything in this report is reproducible in < 5 minutes on a clean clone with wasmtime + wat2wasm installed. Try these to validate the findings:

# C-1: parser accepts garbage
echo 'totally not silicon' | wasmtime wasm-bin/stage1.wasm
# → exit 0, ~40-line WAT module

# C-2: @match broken
cat > audit/match.si <<'EOF'
@type Shape := $Circle r:Int | $Rectangle w:Int, h:Int;
@fn area:Int s:Shape := {
  &@match s,
    $Circle r:Int, { (r * r * 3) },
    $Rectangle w:Int, h:Int, { (w * h) }
};
EOF
wasmtime --dir . wasm-bin/stage1.wasm audit/match.si > audit/match.wat
bin/wat2wasm audit/match.wat -o /tmp/x.wasm
# → wat2wasm error "undefined function variable $r"

# C-3: @enum broken
echo '@enum Color := Red | Green | Blue;' > audit/e.si
wasmtime --dir . wasm-bin/stage1.wasm audit/e.si > audit/e.wat
bin/wat2wasm audit/e.wat -o /tmp/x.wasm
# → wat2wasm error "undefined function variable $Green"

# C-4: missing std helpers
bin/wasm2wat wasm-bin/stage1.wasm | grep -E '^\(func \$arr_load|^\(func \$alloc_array|^\(func \$print_int'
# → empty

# H-1 + H-4: typechecker UX
echo '@fn add:Int a:Int, b:Int := { (a + b) };' | wasmtime wasm-bin/stage1.wasm --typecheck
# → diagnostics with no line/col + unbound on params

# S-1: byte-equal self-host (works correctly)
./build.sh check
# → ✓ fixed point — rebuilt stage1.wasm byte-equal to seed

Methodology note: this audit took ~90 minutes of investigation across source reading, code execution, and disassembly. Time-box matters for these reports — deeper investigation would surface more, but the issues above are individually showstoppers and don't need further triage to decide whether to land the work.
