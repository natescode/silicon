Stage 0 Cleanup Plan

Six focused workstreams to harden the TypeScript/Bun Sigil compiler before the self-host begins · Companion to: docs/bootstrap-plan.html

1. Executive Summary

Premise. Stage 0 (the TS compiler) is going away — but not before it bootstraps Stage 1. Every shortcut left in Stage 0 either propagates into the bootstrap source or makes the bootstrap measurably harder. This plan picks the cleanups that pay for themselves during the bootstrap, and explicitly skips the cleanups that don't.

Scope. Six workstreams, sequenced so each unblocks the next. None require new language design beyond what the bootstrap plan already lists in Phase −1.

Non-goal. Polishing Stage 0 for its own sake. Anything that doesn't survive into Stage 1 or de-risk the bootstrap is explicitly excluded (see §10).

2. Why Now (Sequencing Against the Bootstrap)

Each workstream is a precondition for a later bootstrap phase, or it shrinks a risk register row, or both:

WorkstreamBootstrap dependency
1 — DeterminismPhase 9 fixed-point check. Latent non-determinism caught now is one bug; caught after Stage 2 self-compiles, it's a maze.
2 — UTF-8 stringsResolves §6.2 row D entirely — bootstrap parser reads UTF-8 source and compares to UTF-8 string literals with no convert step.
3 — Enum/sum typesPhase −1.A is on the Phase −1 critical path. Bootstrap source can't represent its own AST without this.
4 — Structured errorsPhase 5's error-format-parity gate is meaningless if Stage 0 errors are just strings. Spans must propagate through every IR layer or Stage 1 inherits a flat error world.
5 — DocsSoft dependency: prevents new contributors from leaning on outdated patterns (f32-sniffing, single-keyword @type_sum) that disappear post-bootstrap.
6 — FuzzingEvery property the fuzzer enforces in Stage 0 becomes a fixture Stage 1 must also satisfy. Test corpus grows for free.

WS 2UTF-8 String Migration
Goal
Replace UTF-16 LE string encoding with UTF-8 throughout. Source files, string literals, and stdlib string helpers all speak the same encoding.
Deliverable
Updated allocString (src/ir/lower.ts:79), updated std.wat length-header semantics, updated stdlib string fns, fixture refresh.
Touches
ir/lower.ts, codegen/std.wat, any test that asserts on string byte layout.

2.1 Why

  • Matches source-file encoding — bootstrap parser reads UTF-8 from fd_read, compares to UTF-8 string literals, no convert step. Eliminates §6.2 row D of the bootstrap plan entirely.
  • Matches WASI/WASIX syscall surface — no convert at I/O time.
  • Half the size for ASCII-heavy content, which is ~all source and ~all log output.
  • Matches every other tool Silicon will interoperate with.

2.2 Layout

# Old (UTF-16 LE):
[byte_len:i32][utf16_le_bytes...]   # byte_len = char_count * 2

# New (UTF-8):
[byte_len:i32][utf8_bytes...]       # byte_len = exact byte count

The header stays 4-byte little-endian length-prefix; only the payload encoding changes. std.wat's $str_concat is already byte-based (copies bytes blindly) — no change needed there. $str_len still returns i32.load 0(ptr).

2.3 Char Type — Don't Add One

The spec is right: there is no single "character" in Unicode. Stdlib gets three explicit views over a string buffer:

  • str_bytes(s) -> Iter<Int> — raw bytes (cheap, what the parser actually wants)
  • str_codepoints(s) -> Iter<Int> — decoded U+ codepoints
  • str_graphemes(s) -> Iter<String> — user-perceived characters (post-bootstrap)

None of these are needed by the bootstrap; document them as the public API and implement only str_bytes in Stage 0.

Done when   All existing tests pass with UTF-8 layout. New test asserts a string literal of 'hello' produces exactly [5, 'h', 'e', 'l', 'l', 'o'] in the data segment. Bootstrap-plan §6.2 row D removed.
WS 3Enum / Sum-Type Split with Payload Layout
Goal
Rename today's enum-only @type_sum to @enum; introduce @type as the general sum-type-with-payloads form. Pattern-match supports variant destructure.
Deliverable
New @enum + @type strata in defkinds.si; extended @match; AST/IR support for tagged-record constructors; round-trip fixture.
Touches
grammar (no change — same definition form), strata/defkinds.si, strata/match.si, defExpanders.ts, typechecker.ts, ir/lower.ts.

3.1 Terminology

ConstructPayloadsLayoutKeyword
EnumNoSingle i32 constant per variant (today's behaviour)@enum
Sum type / Tagged union / ADTYes(tag:i32, field0:i32, ..., fieldN:i32) padded to max variant@type
Untagged unionOverlapping—Excluded

Two keywords intentionally: @enum tells the reader the variants are guaranteed payload-free, enabling cheaper codegen and exhaustive switch-on-i32. @type is the general form — variants may have zero fields (degenerate enum) or many.

3.2 Layout Decision: Pad-to-Max, Not Boxed

For each @type, compute max_fields = max(arity(v) for v in variants). Every value is laid out as:

[tag:i32][f0:i32][f1:i32]...[fmax-1:i32]

Variants with fewer fields leave the trailing slots uninitialised (or zero — pick one and stick with it for determinism). Total size is 4 * (1 + max_fields) bytes per value, allocated contiguously in the arena.

Why not boxed: a pointer + heap allocation per variant doubles arena pressure during AST construction, and the bootstrap AST is allocation-heavy. Stage 1 doesn't need mixed-size variants (everything's i32 in the bootstrap).

Future: when post-Stage-3 work adds non-i32 fields (strings as (ptr, len) pairs, nested structs), the layout generalises to "max of sizeof(variant)" — same principle, different arithmetic.

3.3 Pattern Destructure

@type AstNode :=
  | Literal value:Int
  | BinOp   op:Int, left:Int, right:Int
  | Call    name:Int, args:Int;

@fn evalNode n:Int -> Int := {
  @match n {
    Literal {value}        => value;
    BinOp {op, left, right} => binop op, (evalNode left), (evalNode right);
    Call {name, args}      => callFn name, args;
  };
};

Compiles to a chain of if (tag == TAG_Literal) { bind value := load(node+4); ... } via expandMatchChain.

3.4 Round-Trip Fixture

The acceptance gate is a single end-to-end program that exercises every operation:

@type Shape :=
  | Circle    r:Int
  | Rectangle w:Int, h:Int;

@fn area s:Int -> Int := {
  @match s {
    Circle {r}       => r * r * 3;
    Rectangle {w,h} => w * h;
  };
};

@export main := {
  @local c := Circle 5;
  @local r := Rectangle 4, 6;
  (area c) + (area r);  # 75 + 24 = 99
};

Constructors are function-call syntax (form 3 lookup resolves Circle in scope). area compiles, runs, returns 99.

Done when   The fixture above compiles, runs under wasmer, returns 99. @type_sum renamed to @enum everywhere; @type handles the new general case. Bootstrap-plan Phase −1.A and −1.B move from pending to done.
WS 4Structured Errors + Source Spans
Goal
Errors are structured records with source spans, not formatted strings. Rendering becomes a thin disposable layer.
Deliverable
src/errors/diagnostic.ts defining the Diagnostic record; every error site emits a Diagnostic; CLI dumps them as JSON by default with a --pretty flag for the throwaway TS pretty-printer.
Touches
types/errors.ts, ir/lower.ts, parser/parser.ts, elaborator/elaborator.ts, sigil_cli.ts, every test that asserts on error text.

4.1 The Split: Structural in Stage 0, Rich Rendering in Stage 1

This is the deliberate scope cut. Source spans are infrastructure — painful to retrofit later. Pretty rendering is throwaway — anything you write in TS gets replaced when Stage 1 reformats. So: do the infrastructure, skip the rendering polish.

Stage 0 (now)Stage 1 (post-bootstrap)
  • Every AST node has sourceLocation: {file, line, col, span}
  • Every IR node has back-ref to origin AST node
  • Structured Diagnostic records: { phase, code, span, message, hint? }
  • CLI default: dump diagnostics as JSON
  • 50-line disposable pretty-printer behind --pretty
  • Multi-line context rendering with carets ^^^
  • Color codes, terminal width awareness
  • "Did you mean…" suggestions
  • Error recovery (continue parsing past first error)
  • LSP JSON format (free — already structured)

4.2 The Diagnostic Record

interface Diagnostic {
  phase:    'parse' | 'elaborate' | 'typecheck' | 'lower' | 'emit';
  code:     string;       // e.g. 'E0042' — stable identifiers for matchers/tests
  span:     SourceSpan;   // {file, line, col, length}
  message:  string;       // human, short
  hint?:    string;       // optional, secondary
  notes?:   Diagnostic[]; // related secondary diagnostics
}

4.3 Why This Strengthens the Bootstrap

  • The Phase 5 format-parity gate becomes meaningful. Today it'd diff "type error" strings; with structured diagnostics it diffs (phase, code, span) tuples — a real semantic guarantee.
  • The Stage 1 error implementation has spans to render. Otherwise it'd be retrofitting spans into a self-hosted compiler — exactly the kind of unbounded work that derails bootstraps.
  • Test fixtures get tighter — assert code == 'E0042' instead of substring-matching on prose.
Done when   All error paths emit Diagnostic records. CLI default output is JSON. Existing text-asserting tests updated to assert on code + span instead. The --pretty renderer is <100 LoC.

WS 6Parser Fuzzing Harness
Goal
Find latent parser bugs continuously, and grow the test corpus the bootstrap depends on.
Deliverable
tests/fuzz/parser.fuzz.ts + nightly CI job; corpus of seed inputs in tests/fuzz/corpus/.
Tool
fast-check generators reused from WS 1.

6.1 Three Fuzz Targets (Land in Order)

TargetInput generatorAssertion
Random bytes Arbitrary Uint8Array Parser must either succeed or return a structured error. Must never throw, crash, infinite-loop, or allocate unbounded memory.
Random token streams fast-check generator that emits valid tokens in arbitrary order Parser either produces a parse tree or returns an error pointing at the first bad token.
Generative round-trip Generate random valid AST, pretty-print, re-parse Re-parsed AST equals original AST (modulo trivial differences).

6.2 Budget

  • Local bun test:fuzz — 60 seconds per target.
  • Nightly CI — 30 minutes per target. Failures auto-file an issue with the minimised input.
  • Every reproducer minimised by fast-check gets committed to tests/fuzz/corpus/ as a permanent regression test.

6.3 Bootstrap Bonus

Every fixture the fuzzer minimises is added to tests/corpus/ (the bootstrap's structural-equivalence harness from bootstrap-plan.html §9.4). The fuzzer is therefore a corpus-growth engine for free — by the time Phase 1 starts, the bootstrap parser has hundreds of edge-case fixtures to validate against.

Done when   All three targets run for 60s in bun test:fuzz with zero failures. Nightly CI green for a week. Corpus seeded with at least 50 minimised inputs.

4. Out of Scope (Throwaway in Stage 0)

Explicitly not doing in Stage 0:

  • Error recovery (continue parsing past first error). Useful for IDEs; implement in Stage 1 where it survives.
  • Rich error rendering (color, carets, suggestions). Stage 0 dumps JSON; Stage 1 renders. Per WS 4.
  • LSP support. Free once errors are structured + spans are everywhere. Build it in Silicon, not TS.
  • Optimisation passes (const-fold, DCE). The IR pass middleware loop lands as Phase 6.5 of the bootstrap; passes themselves come post-Stage-3.
  • Generics / type parameters. Bootstrap uses monomorphic i32-handles. Generics post-Stage-3.
  • Annotations (@@), comptime calls (&&), lambdas (\). Post-Stage-3 per bootstrap-plan.html §3.4.
  • WASIX _start export — actually that is in scope, but lives in bootstrap-plan.html Phase −1.E, not here. Different workstream.
  • Performance work on Stage 0. It's going away. As long as compile times are tolerable, leave it alone.

7. Open Questions

  1. For @type variants with zero fields: do we require Circle 0-arg call syntax or allow a bare Foo name to construct? (Recommendation: bare name; calling a zero-arg constructor with empty parens is noise.)
  2. Pad-to-max with unused slots zero-init or undefined-init? Zero-init is deterministic but slightly more codegen; undefined-init is faster but a footgun for code that reads the wrong slot. (Recommendation: zero-init in Stage 1; revisit when performance matters.)
  3. Should @enum variants be auto-numbered (0, 1, 2…) or require explicit values like Red := 0xFF0000? (Recommendation: auto-numbered by default; explicit values via @enum Foo := { Red := 1, Green := 2 } when the user needs control.)
  4. Diagnostic codes — pick a scheme now. E0042-style (rustc) vs parse/unexpected-token-style (eslint)? Both work; the first is shorter, the second self-documenting. (Recommendation: E0042-style for compactness; maintain a separate docs/diagnostics.md with descriptions.)
  5. Fuzzing seed strategy: time-based (fast feedback, non-reproducible) or commit-hash-based (reproducible, predictable)? (Recommendation: commit-hash for CI, time-based for local bun test:fuzz.)
