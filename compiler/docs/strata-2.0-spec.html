Strata 2.0 — Capability Specification

The extension system as the product. Language features ride atop a small set of compiler-side capabilities; this document defines that set. Date: 2026-05-19. Status: spec proposal, no implementation yet.

Why this document exists

Sigil's bet is that a small, frozen grammar plus a powerful extension system beats a feature-rich grammar with no extensibility. docs/strata.md introduced the original Strata; this spec is the v2 redesign that enables that bet at production scale. In particular: generics, traits, pattern guards, derive-style macros, and full comptime evaluation can all be authored as strata under v2 without core compiler changes.

Companion documents: bootstrap-audit.html (what's broken in stage1 today) and stage0-production-roadmap.html (the production push for Stage 0). This spec replaces the "Language Features" section of the latter — those user stories become outcomes of Strata 2.0 rather than separate work.

1. Design tenets

Decisions accumulated through design discussion. Each tenet has knock-on effects on the rest of the spec; they're listed first so subsequent sections can reference them by ID.

T-1 Strata is the product

Language features (generics, traits, async, derive) are tests of the extension system, not work items. If a feature can't be authored as a stratum, the feature is on hold while Strata gains the capability that unblocks it. Adding compiler-internal special cases is the explicit anti-pattern.

T-2 The grammar is frozen

The Ohm grammar (Stage 0) and the hand-written grammar (boot/parser/parse.si) accept five extension axes: keywords, operators, annotations, methods, and sigils. Strata register entries on these axes; the grammar itself doesn't grow. In rare cases where the existing axes provably can't accommodate a feature, grammar change is the last resort and requires explicit discussion.

T-3 Single @stratum keyword

Today's @stratum_keyword / @stratum_operator collapse into one @stratum form. A stratum's body executes once at load time and registers handlers via a DSL. The body is itself a Silicon program — strata are compile-time programs, not declarations.

T-4 Tiered load order with monotonic reference

Three tiers: built-in (T0), user (T1), third-party (T2). Load in tier order; intra-tier in source order modulated by @@before / @@after annotations. T0 may name-resolve other T0 exports; T1 and T2 may only name-resolve lower tiers (T0). Composition between strata at the same tier or across T1/T2 happens via shared events (the observer pattern), never via direct references.

T-5 Universal late-binding error model

Semantic errors (everything past parsing) emit a structured diagnostic at compile time and a runtime trap at the offending site in the emitted WASM. The build never fails on semantic errors; binaries always emit. Executing the affected code path crashes with the original diagnostic text. Syntax errors remain hard fails because you can't lower unparseable input. Internal compiler errors (ICE) crash — they're bugs, not user errors.

Matches Roc + Zig. Enables stub-driven development, partial fixes, interactive exploration. Replaces the audit's H-2 finding ("typechecker is warn-only") — that warning was the right instinct, this is the principled version.

T-6 Cycle-free Strata graph

The directed graph of "stratum A references stratum B" must be acyclic. Cycles detected at load time produce a structured error; the loader breaks one edge (recorded in the diagnostic) and continues. Per T-5, the build still emits — references that were on the broken edge become runtime traps. Cycles are almost always bugs; revisit if real use cases force the issue.

T-7 Comptime is the long-arc goal

Today's body interpreter dispatches a hardcoded set of &Compiler::* calls. Strata 2.0 grows that interpreter incrementally toward full Silicon evaluation at compile time. Every capability added is designed to fit a future world where strata bodies can run arbitrary Silicon code. Implementation phased; principle is fixed.

2. The four-layer model

A stratum participates in compilation across four orthogonal layers. Treating them independently — rather than mashing them into "kinds of strata" — is what makes a single @stratum keyword expressive enough to subsume the five extension axes of T-2.

┌─ LAYER 1: SYNTACTIC SURFACE ───────────────────────────────────┐ │ │ │ What the grammar will recognize as belonging to this stratum │ │ │ │ • keyword '@foo' │ │ • operator '+' │ │ • annotation '@@derive' │ │ • method 'map' │ │ • sigil '$json' │ └────────────────────────────────────────────────────────────────┘ ┌─ LAYER 2: LIFECYCLE PHASES ────────────────────────────────────┐ │ │ │ When the stratum's handlers fire │ │ │ │ on::decl on::call_site on::module_ │ │ on::typecheck on::lower finalize │ │ on::annotation on::method on::parse (rare) │ └────────────────────────────────────────────────────────────────┘ ┌─ LAYER 3: EFFECTS ─────────────────────────────────────────────┐ │ │ │ What a handler is allowed to do │ │ │ │ • Read AST nodes, source spans, doc comments │ │ • Manipulate types as values (construct, substitute, equate) │ │ • Synthesise AST (capture template, clone, substitute, │ │ re-elaborate) │ │ • Emit new top-level definitions / globals / imports │ │ • Read/write per-stratum or per-invocation state │ │ • Push structured diagnostics (compile-time + runtime trap) │ │ • Emit IR (existing Compiler::ir::* surface) │ └────────────────────────────────────────────────────────────────┘ ┌─ LAYER 4: COMPOSITION ─────────────────────────────────────────┐ │ │ │ How multiple strata interact at the same event │ │ │ │ • Multiple handlers on one event = observer pattern │ │ • Firing order: (tier ASC, source-order, @@before/@@after) │ │ • No "owner" required for an event — any stratum can observe │ │ • Intra-tier name reference: forbidden (except T0) │ │ • Inter-tier name reference: only downward │ └────────────────────────────────────────────────────────────────┘

A given stratum can sit anywhere on each layer. A simple stratum might participate on Layer 1 (one keyword) + Layer 2 (one phase, on::lower) + Layer 3 (just emit IR) + Layer 4 (owner pattern, no other strata involved). A generics stratum participates on Layer 1 (one keyword) + Layer 2 (three phases) + Layer 3 (types, AST synthesis, module mutation, state) + Layer 4 (defines events that observer strata can hook).

3. Tier model

Tier determines load order, name visibility, and resource sandboxing.

Identification

TierIdentified byLocation convention
T0 — built-in Strata shipped with the compiler binary boot/strata/builtin/*.si
T1 — user silicon.toml declares [package] and the strata are in strata/ <project>/strata/*.si
T2 — third-party The manifest lists this stratum's package under [dependencies] (any form: git, path, registry) <cache>/deps/<pkg>/strata/*.si

Crucial: tier is set by the manifest's declaration, not by where the file ended up on disk. A locally-pathed dependency (foo = { path = "../foo" }) is still T2 because the manifest calls it a dependency. Matches developer intent: vendoring a stratum into your own strata/ directory promotes it to T1; declaring it as a dep keeps it at T2.

Reference rules

From / ToT0T1T2
T0✓ allowed✗ forbidden✗ forbidden
T1✓ allowed✗ forbidden✗ forbidden
T2✓ allowed✗ forbidden✗ forbidden

"Reference" means name resolution against another stratum's exports. It does NOT mean event observation — any stratum at any tier can register handlers on events fired by any other stratum at any other tier. The two paths are orthogonal.

Enforcement

Lexical scoping. Each tier loads into a separate namespace. The registry exposes a tier's symbols only to higher-tier resolvers. Trying to name-resolve a forbidden cross-tier reference returns "unbound" with the same diagnostic shape as any other unbound identifier (per T-5, that becomes a runtime trap in the emitted binary).

Load order

1. Walk boot/strata/builtin/ → parse each .si file → run @stratum body Result: T0 handlers registered, T0 names visible to T0/T1/T2 2. Read <project>/silicon.toml → locate strata/*.si in the project → parse each, run @stratum body Result: T1 handlers registered, T1 names visible to T1 only (self) 3. Read <project>/silicon.toml → walk [dependencies] → pull each package's strata/, parse, run @stratum body Result: T2 handlers registered, T2 names visible to that package only 4. Begin main compile. Events fire as the AST is walked. Handlers run in (tier ASC, source order, @@before/@@after) order.

Cycles

Per T-6, the references-between-strata graph at T0 must be a DAG. Detection runs at the end of step 1; on cycle, the loader picks an edge (deterministically: lexicographic by stratum name pair), drops it, pushes a structured diagnostic, continues. References that traversed the dropped edge become runtime traps in the affected emitted code. T1 and T2 cannot form cycles because they have no intra-tier references.

4. The @stratum registration DSL

A stratum is declared with the @stratum definition keyword: 

@stratum Name = {
  &Compiler::register::<axis> 'token';
  &Compiler::on::<phase> ...args... = { ...handler body... };
  ...
};

The body is executed once when the stratum loads. Each register::* call adds an entry to the syntax registry; each on::* call deferred-binds a handler that fires on the named phase.

The body is a Silicon expression

Not a special declaration form. This matters for T-7: as the comptime interpreter grows, you can have conditionals, loops, and helper function calls inside the body. A stratum is a program that registers other programs. 

@stratum ConditionalRegister = {
  &Compiler::register::keyword '@generic';

  @if &Compiler::feature_flag 'derive', {
    &Compiler::register::annotation '@@derive';
  };

  &Compiler::on::decl '@generic', Node = {
    ...
  };
};

Handler bodies are deferred

An inner { ...body... } following an on::* call is NOT executed at registration. It is captured as a closure and run later when the matching event fires. This is the load-vs-run distinction; the outer body builds the registry, the inner bodies populate it with code that runs later.

5. Capability API surface

Concrete entry points strata can call. Grouped by the layer they contribute to. Each entry tagged MVP (must ship in v1) or DEFER (later).

5.1 Layer 1 — Syntactic registration

&Compiler::register::keyword token:String MVP Adds token (e.g. '@generic') to the keyword registry. Naked registration is valid; an on::decl handler can be supplied by any tier or a later-loaded stratum. Per T-5, an unhandled keyword occurrence becomes a runtime trap.
&Compiler::register::operator token:String MVP Adds token (e.g. '+') to the operator registry. Operators are typed-dispatched — multiple strata can register the same token for different operand types and the right one fires based on operand inference.
&Compiler::register::annotation token:String MVP Adds token (e.g. '@@derive') to the annotation registry. Annotations attach to the immediately-following definition; the on::annotation handler receives both the annotation node and its target definition.
&Compiler::register::method token:String DEFER Adds token (e.g. 'map') to the method registry. Methods are dispatched on the receiver's static type. Deferred because no MVP user needs it; landing it is straightforward once Layer-3 type-as-data is in.
&Compiler::register::sigil token:String DEFER Adds token (e.g. '$json') to the sigil registry. Sigils introduce new literal-like forms. Deferred to keep MVP scope tight.

5.2 Layer 2 — Lifecycle phases

&Compiler::on::decl token:String, node = { body } MVP Fires for each definition AST node whose keyword equals token. node binds to the AST node. Handler runs at elaboration time, before lowering.
&Compiler::on::call_site node = { body } MVP Fires for every function call in the program. The handler is responsible for filtering — typically by inspecting &Compiler::callee::kind node. Used by the monomorphization engine to redirect calls to specialized instances.
&Compiler::on::annotation token:String, ann_node, target_def = { body } MVP Fires when an annotation token precedes a definition. Both nodes bound for inspection. Annotation handlers can modify the target definition via the synthesis API before lowering proceeds.
&Compiler::on::module_finalize = { body } MVP Fires once per module, after every other node has been walked but before code emission. Used to flush deferred work — monomorphization queues, derived implementations, accumulated diagnostics.
&Compiler::on::lower node = { body } MVP The "today" semantics — fires at IR lowering for a matching syntactic form. The current @stratum_keyword body model maps to this phase.
&Compiler::on::typecheck node = { body } DEFER Fires after type inference for a matching form. Lets a stratum participate in / override type-checking for its syntactic surface. Deferred — generics doesn't strictly need this if Approach A monomorphization re-elaborates from AST templates.
&Compiler::on::method receiver_type, token, call_node = { body } DEFER Paired with register::method. Deferred.
&Compiler::on::parse node = { body } DEFER Rare; runs at parse time before any semantic info exists. Used by token-level sugar. Deferred — most cases handled by later phases.

5.3 Layer 3 — Effects (AST read)

&Compiler::arg node, idx:Int MVP Today's API. Returns the idxth argument of node.
&Compiler::ast::children node MVP Returns the list of child AST nodes for general traversal.
&Compiler::ast::span node MVP Returns the (file, line, col, length) source span of node. Required for any stratum that pushes diagnostics with location info.
&Compiler::ast::doc node MVP Returns the ## doc-comment string immediately preceding node, or empty. Used by @@derive(Display)-style strata that want to surface docs.
&Compiler::ann_present node, token:String MVP Returns true if the annotation token (e.g. '@@inline') precedes node. Lets one stratum's handler test for the presence of another stratum's annotation.
&Compiler::ann::args ann_node MVP Returns the argument list of an annotation node. For @@derive(Display, Debug), returns the two identifiers.

5.4 Layer 3 — Effects (Types as data)

&Compiler::type::int / &Compiler::type::int64 / &Compiler::type::float / &Compiler::type::bool / &Compiler::type::string / &Compiler::type::void MVP Singleton type values. Used to construct generic instantiations: type::substitute tmpl, $T, &Compiler::type::int.
&Compiler::type::array elem_type MVP Build an Array<elem_type> type value.
&Compiler::type::function params:List, result MVP Build a function type value.
&Compiler::type::variable name:String MVP Build a type variable (a placeholder used in generic templates). Pre-elaboration AST capture preserves these; substitute replaces them.
&Compiler::type::equals A, B MVP Structural equality on type values.
&Compiler::type::infer_args call_node MVP Returns a list of inferred type values for each argument at a call site. Used by generics to determine the concrete instantiation.
&Compiler::type::substitute tmpl_type, bindings:Map MVP Walk tmpl_type, replace any type variable matching a key in bindings with its bound value, return the substituted type.
&Compiler::type::format type MVP Returns a human-readable string for a type value. Used in diagnostics.

5.5 Layer 3 — Effects (AST synthesis)

&Compiler::ast::capture_template node, kind:String MVP Returns a template handle representing node's subtree. kind is 'pre' (un-elaborated parse AST) or 'post' (elaborated AST with names resolved and operators dispatched). Per design discussion, both are supported.
&Compiler::ast::clone template_handle MVP Returns a fresh, mutable copy of the template. Required because substitution mutates.
&Compiler::ast::substitute template_handle, bindings:Map MVP Walk the template, replace every reference to a key in bindings with its bound value. Bindings can map type variables to type values OR identifiers to other identifiers (for name mangling of monomorphs).
&Compiler::ast::re_elaborate template_handle MVP For pre-elaboration templates: run the captured AST through elaboration (name resolution + operator dispatch) using the current symbol environment. Required after substitute on a pre-template.
&Compiler::ast::patch_types template_handle, bindings:Map MVP For post-elaboration templates: walk the elaborated AST, replace type-variable annotations in-place. Lighter than re_elaborate; works when the template's structure doesn't depend on the substituted types.

5.6 Layer 3 — Effects (Module mutation)

&Compiler::module::push_definition def_node MVP Adds def_node as a new top-level definition in the current module. Used by generics to emit monomorphized function instances. The new definition is elaborated, type-checked, and lowered as if it had been in the source.
&Compiler::module::push_global name:String, type, init MVP Adds a new global value to the current module.
&Compiler::module::push_import mod:String, name:String, type DEFER Adds a new WASM import. Deferred — current strata don't need this; the WASI stub injection in build scripts covers it for now.

5.7 Layer 3 — Effects (State)

&Compiler::state scope:String MVP Returns a state bucket. scope is 'instance' (fresh per handler firing) or 'stratum' (shared across all handlers of the same stratum, across all firings, for the duration of one compilation). 'global' (cross-stratum) deferred. State buckets are Silicon objects with set / get / has / each methods.

5.8 Layer 3 — Effects (Diagnostics)

&Compiler::diag::error code:String, span, message:String, hint:String MVP Push a structured error diagnostic. Renders identically to compiler-emitted errors (code, span, caret, hint). Per T-5, also emits a runtime trap at span in the resulting binary if the affected code path is reached.
&Compiler::diag::warn code:String, span, message:String, hint:String MVP Push a warning. Renders without trap insertion. For lint-style strata.
&Compiler::diag::note span, message:String DEFER Push an informational note. Used for "see also" links in complex diagnostics.

5.9 Layer 3 — Effects (IR emission)

Existing &Compiler::ir::makeIf, makeConst, makeFunction, etc. No new entries needed for MVP; the existing surface is already adequate. Document them as Layer-3 IR entries here for completeness.

5.10 Layer 4 — Composition annotations

@@before "OtherStratumName" MVP Annotation on an on::* registration. Engine sorts so this handler fires before any handler from the named stratum. Same-tier handlers only; cross-tier ordering is fixed by tier.
@@after "OtherStratumName" MVP Mirror of @@before.

6. Universal late-binding error model

Per T-5, every semantic error in the program emits a structured diagnostic at compile time AND a runtime trap at the offending site in the binary. This section specifies how that's mechanized.

6.1 Compile-time output

Every error has: code, span, severity (error/warning), message, optional hint, optional related notes. Rendering: 

error[E0102]: type mismatch
  --> src/main.si:12:5
   |
12 |     @let count:Int := 3.14;
   |     ^^^^^^^^^^^^^^^^^^^^^^ expected Int, got Float
   |
   = note: a runtime trap at this location will fire if the program
           reaches this assignment.  Fix the source or constrain the
           code path to avoid it.
   = help: convert with `&@toInt 3.14` to truncate.

Built with 1 error.  Run with caution.

The summary line at the end is intentional: it signals that the build succeeded (per T-5) but the binary contains a known failure mode.

6.2 Runtime trap lowering

Each error site's offending IR node lowers to: 

;; Original (would have been):
;;   local.get $value           ;; whatever the original computation was
;;   ...

;; With error binding:
i32.const <diag_id>            ;; index into the embedded diag table
call $silicon_runtime_trap     ;; never returns
unreachable                    ;; validator requirement

$silicon_runtime_trap lives in std.wat: 

(func $silicon_runtime_trap (param $diag_id i32)
  ;; Look up the diagnostic in the embedded table
  (call $silicon_diag_table_load (local.get $diag_id))
  ;; Print to stderr
  (call $silicon_diag_print)
  ;; Exit with non-zero status
  (call $wasi_snapshot_preview1::proc_exit (i32.const 101)))

The diag table is a WASM data section the compiler emits alongside the module — one entry per trap site with the file:line:col, error code, message, and hint. Total overhead: ~64 bytes per error site (typically a handful per program).

6.3 Errors that bind to a value (not just a site)

For "value-level" errors — like @let x:Int := some_function_returning_string — the trap fires when x is read, not at the binding. This requires writing the diag id into the slot in place of the offending value and trapping on read. Adds complexity; for the MVP we trap at the binding site instead. Tradeoff: less precise but simpler. Revisit if "trap on use, not on bind" becomes important.

6.4 Errors that prevent codegen entirely

Some errors (a function with no body, a type alias referencing itself unbreakably, etc.) genuinely block lowering. For these, the trap replaces the entire function: any call to it traps immediately. The enclosing module still compiles; the program still produces a binary; the broken function just panics when invoked.

7. Worked example: @generic

Approach A — template monomorphization via pre-elaboration AST capture. Three handlers across two strata. 

@stratum GenericKeyword = {
  &Compiler::register::keyword '@generic';
};

@stratum Monomorphize = {
  @local templates    := &Compiler::state 'stratum';
  @local instantiations := &Compiler::state 'stratum';

  &Compiler::on::decl '@generic', DeclNode = {
    @local name := &Compiler::watId DeclNode.name.name;
    @local tmpl := &Compiler::ast::capture_template DeclNode, 'pre';
    &templates::set name, tmpl;
  };

  &Compiler::on::call_site CallNode = {
    @local callee := &Compiler::callee::name CallNode;
    @local tmpl := &templates::get callee;
    &@if (tmpl != @nil), {
      @local types := &Compiler::type::infer_args CallNode;
      @local key := &mangle callee, types;
      &instantiations::set key, $InstReq tmpl, types;
      &Compiler::ast::rewrite_call CallNode, key;
    };
  };

  &Compiler::on::module_finalize = {
    &instantiations::each, Key, Req = {
      @local clone := &Compiler::ast::clone Req.tmpl;
      @local sub := &Compiler::ast::substitute clone, Req.bindings;
      @local mono := &Compiler::ast::re_elaborate sub;
      &Compiler::module::push_definition mono;
    };
  };
};

What a user writes:

@generic first xs:List<T> := { (&list_get xs, 0) };

@fn _start:Void := {
  @local ints := &list_of_ints 1, 2, 3;
  @local x := &first ints;     ;; type::infer_args => [List<Int>]
  ...
};

What the compiler emits (conceptually):

@fn first$List_Int xs:List_Int := { (&list_get xs, 0) };

@fn _start:Void := {
  @local ints := &list_of_ints 1, 2, 3;
  @local x := &first$List_Int ints;
  ...
};

All achieved without compiler changes. Adding a second instantiation (&first on List<String>) generates a second monomorph automatically. Errors inside the template are reported per-instantiation, per @generic's re_elaborate call — a known C++-templates UX cost we accept for Approach A simplicity.

8. Worked example: @@derive

An annotation that adds trait implementations to a @type declaration. Demonstrates Layer-4 observer composition: the Derive stratum doesn't own @type — it just observes @type declarations and emits additional code if the @@derive annotation is present. 

@stratum Derive = {
  &Compiler::register::annotation '@@derive';

  &Compiler::on::annotation '@@derive', AnnNode, TargetDef = {
    @local traits := &Compiler::ann::args AnnNode;
    &loop_traits traits, Trait = {
      &@if (Trait == 'Display'), { &emit_display_impl TargetDef };
      &@if (Trait == 'Debug'),   { &emit_debug_impl TargetDef };
      &@if (Trait == 'Eq'),      { &emit_eq_impl TargetDef };
    };
  };
};

@fn emit_display_impl TargetDef := {
  @local name := &Compiler::watId TargetDef.name.name;
  @local impl_ast := &build_display_impl_for TargetDef;
  &Compiler::module::push_definition impl_ast;
};

User code:

@@derive(Display, Debug)
@type Point := $Point x:Int, y:Int;

Compiler walks past the @@derive annotation, lands on the @type declaration, fires the on::decl '@type' handler (which is in T0, the built-in handler for @type), THEN fires the Derive stratum's on::annotation handler (also T0 if you bundle it built-in, T1 if a user wrote it). Order is deterministic; the on::annotation handler runs after the target's own declaration handler so it can push_definition safely.

9. Worked example: third-party plugin

A T2 stratum from an imaginary silicon-json dependency that registers a sigil for JSON literals.

Package manifest:

[dependencies]
silicon-json = { git = "https://github.com/...json", tag = "v0.1" }

The stratum (loaded as T2):

@stratum JsonLiteral = {
  &Compiler::register::sigil '$json';

  &Compiler::on::sigil '$json', SigilNode = {
    @local raw := &Compiler::ast::raw_text SigilNode;
    @local parsed := &parse_json_at_comptime raw;
    &@if (parsed.error != @nil), {
      &Compiler::diag::error
        'E2001',
        (&Compiler::ast::span SigilNode),
        parsed.error.message,
        'Check the JSON syntax in this $json literal.';
    };
    &@if (parsed.error == @nil), {
      &Compiler::ir::make_json_value parsed.tree;
    };
  };
};

User code:

@use 'silicon-json/api.si';

@fn _start := {
  @local config := $json'{"port": 8080, "host": "localhost"}';
  ...
};

The JsonLiteral stratum is T2, cannot reference other T2 strata, and cannot be referenced by T1 user strata. But the user's program uses it through the registered sigil, which is the right kind of coupling: syntactic and capability-based, not symbol-based.

11. Migration from current strata

Current strata (boot/strata/builtin/*.si) use @stratum_keyword and @stratum_operator as one-shot declarations with implicit on-lower handlers. Migration is mechanical:

TodayStrata 2.0
@stratum_keyword Const ('@const', Node) = {
  &Compiler::ir::makeGlobal Node;
};
 
@stratum Const = {
  &Compiler::register::keyword '@const';
  &Compiler::on::lower Node = {
    &Compiler::ir::makeGlobal Node;
  };
};
 
@stratum_operator Plus ('+', Node) = {
  &IR::i32_add Node.left, Node.right;
};
 
@stratum Plus = {
  &Compiler::register::operator '+';
  &Compiler::on::lower Node = {
    &IR::i32_add Node.left, Node.right;
  };
};

The old syntax can stay as syntactic sugar that desugars to the new form — or can be removed in a single sweep. Either works; mechanical migration script in either direction is < 200 LoC.

Recommendation: keep both for one release, then drop @stratum_keyword / @stratum_operator. Gives users a window to migrate without a hard cutover.

12. Open questions / future work

Open: Compile-time evaluation scope

The body interpreter currently dispatches a fixed set of &Compiler::* calls. Strata 2.0 expands it incrementally. Open question: when does the interpreter become powerful enough to handle arbitrary Silicon code at compile time? The pragmatic answer is "when a stratum needs it" — grow the interpreter feature-by-feature as use cases force the issue. But there's a design pull toward deciding the long-arc target now (e.g., "the interpreter implements a subset of Silicon limited to expressions over comptime-known data"). Worth pinning down before too much surface accumulates.

Update (2026-05-22): direction settled. See comptime-via-compilation.md — the dissolution plan replaces the AST-walk interpreter with a Zig-style compile-then-run model: strata bodies become first-class Silicon programs lowered through the normal pipeline and invoked by the host via a compiler-as-imports surface. Closes the @if-as-intrinsic chicken-egg and gives strata bodies full Silicon at compile time. Not yet implemented; T-7 is the principle, that doc is the destination.

Open: Trait/interface dispatch under T-5

Traits-as-stratum (LANG-3) needs dispatch resolution at call sites. Under T-5, an unresolved trait method call must trap at runtime. Is that always the right behavior? For "method called on a type that doesn't implement the trait" the trap is fine. For "trait method not implemented for the inferred type", the call-site rewriting needs to emit the trap inline. Spec-out when implementation proceeds.

Resolved: type system for generic strata (HM-lite, 2026-05-22)

The spec's worked examples for @generic and @@derive assume a type system that handles parametric types and call-site inference. That layer landed as HM-lite — Hindley-Milner restricted to declared polymorphism on @fn[T] and @type[T], no let-generalisation for locals. Roc-style trajectory; full HM is the natural extension. See hm-lite.md for the data shapes, the five core primitives, and how the typechecker threads them through call sites, annotation reconciliation, and @match arm checking. <Option[T], List[Option[Int]]> etc. parse and typecheck end-to-end.

Companion: @match grew an arm-expression form &@match d, pat => body, … with per-arm pattern alternation via | — zero grammar changes because => and | were already valid BinaryOp operators. Legacy flat form still works; both coexist.

Open: Cross-compilation tier rules

When compiling for one WASM target on another host, do tier rules apply to the host's compiler or the target's runtime? Probably the former (strata run at compile time on the host), but worth checking if cross-compilation ever becomes a thing. Punt until needed.

Open: Strata versioning

A T2 stratum declares a required platform/compiler version somewhere. Today there's no version stamp on the compiler. Coupled with PKG-1 (manifest) and CI-3 (semver enforcement) in the production roadmap. No new design needed; just remember to land version metadata when PKG-1 lands.

Open: Incremental compilation + strata state

State buckets (stratum scope) persist across handler firings within one compile. What about across compiles in watch mode (PERF-1)? Probably reset between compiles — cache only the typed-AST output, not stratum scratch state. Worth formalizing when the incremental cache lands.
