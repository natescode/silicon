The problem

AI writes code in languages designed for humans.

Every major language — Python, JavaScript, Rust, Go, C++ — was designed before LLMs existed. Their grammars are ambiguous. Their error models punish iteration. Their tooling assumes a human is reading every diff. Their package ecosystems are vendor-trust minefields. Their version control systems require humans to resolve conflicts.

!

Grammars drift

Every Python or TypeScript release changes the language. AI models trained on yesterday's syntax produce code that yesterday's parsers reject.

!

Errors block iteration

A single type error stops the whole build. AI-generated code is "mostly right" by nature; languages that demand "fully right" before any output force expensive correction loops.

!

Merge conflicts don't scale

Three AI agents working in parallel on the same repo produce three branches Git can't combine without a human in the loop. The bottleneck isn't model quality — it's the VCS.

!

Generic models are spread thin

GPT-5 and Claude know a little about every language and a lot about none. For Silicon specifically, no general model is competitive.

!

Runtimes don't sandbox

AI-generated code is untrusted by default. Running it in V8 or Python interpreters means trusting it with full host access.

!

Tooling assumes humans

IDEs, linters, and code review tools optimize for the human reviewer. None optimize for the AI author.

The industry's response is to make models smarter. Ours is to make the language — and the entire stack around it — designed for the way models work.

Silicon
The language

A language designed for the model, not the human.

The design pillars

  • Frozen grammar. New features land as libraries, not syntax changes. A model trained today still understands code written in 2030.
  • Library-extensible semantics. The Strata system lets users (and AI agents) author new keywords, operators, annotations, methods, and literals without compiler changes.
  • Late-binding errors. Every error produces a diagnostic AND a binary. AI iterates on partial code without hitting build walls.
  • Explicit conversions. No implicit coercions. The model can't accidentally produce code that compiles but means something different than intended.
  • WASM-native. Output runs in any sandbox. Memory-safe by default, capability-constrained by WASI.
  • Self-hosted. The compiler is written in Silicon. A model studying the language can study the implementation in the same training pass.

What this looks like

# A user (or an AI) adds a new keyword via a stratum.
# No grammar change.  No compiler PR.  Just a library.

@stratum Generic = {
  &Compiler::register::keyword '@generic';
  &Compiler::on::decl '@generic', Node = {
    &Compiler::ast::capture_template Node;
  };
  &Compiler::on::call_site Call = {
    &monomorphize Call;
  };
};

# Now anyone (or any AI) can use it.
@generic first xs:List<T> := { &list_get xs, 0 };

Generics, traits, pattern guards, derive macros — every "language feature" in mainstream languages ships as a Silicon library in 50-200 lines.

Sigil
The compiler

The smallest production-grade compiler ever shipped.

Why "smallest" matters

Mature compilers (LLVM, GCC, V8) are millions of lines and decades of accreted assumptions. No model — no human — understands them end to end. When the model writes code, it's targeting a system it can't reason about.

Sigil is < 10,000 lines of Silicon code. An AI agent can ingest the entire compiler as context, reason about exactly what every Silicon program lowers to, and confidently propose changes.

  • Auditable in a weekend. Whole-codebase review by a senior engineer takes ~16 hours.
  • Self-hosted with byte-equal fixed point. Sigil compiles its own source to byte-identical output — the strongest possible correctness signal.
  • WASM-only output target. No optimization swamp, no CPU-specific backends, no link-time chaos.
  • Trap-on-error binaries. Every diagnostic produces a runnable artifact. Iteration loops are measured in seconds.

NIT
Content-addressed version control for parallel AI agents

Code as values, not files.

The shift

Git represents history as a directed acyclic graph of commits over text files. Every "change" mutates a file; merging means walking the graph and reconciling textual conflicts — a fundamentally human task because resolution requires understanding intent.

NIT, taking the model proven by Unison and pairing it with Silicon, represents code as a content-addressed database. Every function, type, and definition is identified by the hash of its normalized AST. Names are mutable metadata pointing at immutable hashes. Edits never mutate anything — they add a new hash and update the name to point at it.

  • Nothing ever overwrites. Conflicts as a category don't exist — two agents producing different implementations of foo just produce two different hashes. The name layer decides which one foo resolves to.
  • Refactoring is free and safe. Rename a function: update the name → hash map. No grep-and-replace, no risk of missing a reference.
  • Dependencies pin to hashes. Builds are reproducible forever. Upstream libraries can't break your code by publishing a new version — you're pinned to a specific hash.
  • History is granular. Every hash is a precise unit of change with an author + a timestamp + the AST that produced it. Audit "what did this AI agent change?" by listing the hashes it added.

Why this unlocks AI workflows

The bottleneck in agentic coding isn't model quality — it's the human reviewing and resolving merge conflicts between concurrent agents. Under content-addressing, that category of conflict doesn't exist. Ten agents working in parallel each produce new hashes; the name layer is the only place orchestration matters, and orchestration is a structured problem (cheap to automate) instead of a textual one (expensive, semantic, human-only).

NIT pairs naturally with Silicon: both treat code as values rather than text. Silicon's frozen grammar means hashes are stable across releases; the strata system means name → hash mappings can carry rich metadata; the WASM-native runtime means content-addressed code can be cached + distributed + executed without untrust assumptions.

"Code is a database, not a folder of text files. Hash it, name it, version the names independently. Merges disappear because the layer they happened on disappears." — Project thesis, after Unison
Silicon Model
The specialized AI

One language. One model. Deeply expert.

The wager

A 7B-30B parameter model trained exclusively on Silicon code + the compiler internals + the strata catalogue + the language spec outperforms any general-purpose model on every Silicon-specific task by 10x or more.

General models — GPT-5, Claude, Gemini — have to allocate their parameter budget across hundreds of languages, frameworks, and domains. Their grip on any single one is shallow. A specialized model owns its domain entirely.

  • Fluent in every stratum. Knows every keyword, operator, annotation users have authored — including ones written after the model's training cutoff, because the strata are self-describing.
  • Compiler-aware. Can predict exactly what code compiles to. Trained on AST + IR + WAT pairs, not just source text.
  • Verification-aware. Can prove invariants by emitting Silicon comptime checks — the strata system makes proof-as-code first-class.
  • Tiny inference cost. At 7-30B params, runs on a single consumer GPU or as a hosted service for cents per request.

Why this is defensible

The model gets better with use. Every Silicon program written, every stratum authored, every NIT history committed is training data. General models can't catch up because they don't have privileged access to the Silicon ecosystem's data flywheel.

Defensibility

Five reinforcing moats.

Each product is interesting alone; together they form a position no incumbent can replicate without rebuilding their entire stack.

1. Language lock-in

Code written in Silicon doesn't port out cheaply. Strata are tied to the runtime; WASM-native idioms don't translate. Once a team adopts Silicon, they stay.

2. Model data flywheel

Every Silicon program written improves the model. Every stratum authored expands its surface. Competitors building from generic web crawls can't match the signal-to-noise of curated ecosystem data.

3. VCS network effect

NIT's value scales with codebases hosted on it. Once a project moves from Git to NIT for content-addressed reproducibility + parallel-agent capability, moving back means re-inheriting every problem Git was designed around.

4. Ecosystem composability

Strata compose: third-party libraries extend the language for free. Each new stratum makes Silicon more valuable for everyone else — classic two-sided market dynamic.

5. Trust through smallness

An auditable compiler + a sandboxed runtime + a deterministic VCS is the only stack that lets enterprises trust AI-generated code in production. No competitor offers this combination.

6. Founder velocity

The stack is internally consistent because one team designed it. Incumbents trying to retrofit AI-first capabilities into Python or TypeScript hit decades of legacy friction.
