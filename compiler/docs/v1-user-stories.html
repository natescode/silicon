Sigil 1.0 — User Stories

Status note (post-boot/ removal): This document was authored when the Silicon-in-Silicon bootstrap lived under boot/. That tree has been removed and the self-hosted compiler is slated for a future rewrite from scratch. References to boot/* paths in the stories below describe the historical bootstrap layout — they need re-scoping when the rewrite begins. The TypeScript compiler in src/ is the current production compiler.

Session-sized work units to ship 1.0. Stories are grouped by phase (matching sigil-1.0-roadmap.md) with the in-progress dissolution work as Phase 1 and a cross-cutting Compiler-as-a-Service section as Phase 10. Click any story to expand its goal, acceptance criteria, and subtasks. Date: 2026-05-23.

Architectural directives (2026-05-23):

1. Compiler-as-a-Service. Sigil is designed Roslyn-style — the compiler is a library, not just a CLI. Every consumer (sgl tool, LSP server, IDE plugins, formatter, REPL, future analysers) uses the same public API surface. LSP-protocol conformance is optional — the underlying API may be richer. CaaS is split across two phases: Phase 2 — CaaS Foundation is required-for-1.0 and must land before the boot/ port (Phase 7) so the self-hosted compiler inherits the right shape. Phase 10 — CaaS Consumer APIs is the rest: some 1.0, some post-1.0 stretch. The LSP (v1.1's Phase 9b) consumes both.

2. Interpreter-first debugging. The Silicon interpreter (Phase 11) is the primary debug tool for 1.0 — it shows Silicon values as Silicon types and covers app-developer needs. Native DWARF emission (v1.1's Phase 9a) is moved out of 1.0; native binaries still get function-name symbols for free from QBE/linker, so stack traces stay readable. Full source-line DWARF + LSP + IDE integration land in v1.1 — see docs/v1.1-user-stories.html.

Phase 9c — Memory management (explicit arenas)

Phase 9c SHIPPED in src/ ✅ (2026-05-26). 9 of 10 stories complete; 9c-7 (boot/ port) moved to v1.1 alongside Phase 7. ADR 0008 moved to Accepted. Two follow-up tracks ride on top of the core arena work: 9c-3a (Sum-payload promotion + 9c-8 introspection helpers) closes Phase 9c's known gaps, and 9c-9 ships the Rc smart pointer stdlib as a composition-driven demonstration of stratum power (Rc-as-stdlib + @defer + &@with_arena = Rust's Rc / Box / Drop without compiler changes).

Today's bump allocator (src/codegen/std.wat + src/codegen/prelude-ir.ts) is a one-way ratchet: every allocation lives forever. Fine for sigilc one.si; unusable for sgl watch, servers, REPLs, or any loop bigger than a few iterations. Phase 9c lands the v1.0 instantiation of ADR 0008: explicit scoped arenas with stack-discipline freeing, plus an explicit escape hatch so heap values (most importantly String) can be returned from the arena to the enclosing scope.

Two surface constructs. &with_arena { body } saves the heap pointer at entry, lets body bump-allocate normally, and resets the heap pointer at exit — every byte allocated inside is freed at once. &move_to_parent_arena value (tail position only) memcpys value's bytes into the saved-pointer region before the unwind, so the block evaluates to a pointer that survives the reset. v1.0 supports flat heap types (String, value arrays, sum types with all-value payloads). Nested heap (Vec[String] etc.) is a v1.1 trace-and-copy follow-up (M-7 / M-8).

Why now. Servers and REPLs are 1.0 acceptance bars; without arenas they OOM in seconds. Doing it as a stratum keeps the door open for v1.1's GC stratum (same wit/allocator.wit interface) without breaking 1.0 programs. Same architectural shape as ADR 0003 (comptime engine).

Introspection primitives (&rc_count, &heap_used, …) raise E0012 under wasm-gc — they can't be honestly emulated.
