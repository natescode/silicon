Sigil 1.0 — User Stories

Session-sized work units to ship 1.0. Stories are grouped by phase (matching sigil-1.0-roadmap.md) with the in-progress dissolution work as Phase 0 and a cross-cutting Compiler-as-a-Service section as Phase 8.5. Click any story to expand its goal, acceptance criteria, and subtasks. Date: 2026-05-23.

Architectural directives (2026-05-23):

1. Compiler-as-a-Service. Sigil is designed Roslyn-style — the compiler is a library, not just a CLI. Every consumer (sgl tool, LSP server, IDE plugins, formatter, REPL, future analysers) uses the same public API surface. LSP-protocol conformance is optional — the underlying API may be richer. CaaS is split across two phases: Phase 0.5 — CaaS Foundation is required-for-1.0 and must land before the boot/ port (Phase 3) so the self-hosted compiler inherits the right shape. Phase 8.5 — CaaS Consumer APIs is the rest: some 1.0, some post-1.0 stretch. Phase 9b (LSP) consumes both.

2. Interpreter-first debugging. The Silicon interpreter (Phase 7) is the primary debug tool for 1.0 — it shows Silicon values as Silicon types and covers app-developer needs. Native DWARF emission (Phase 9a) is marked post-1.0 acceptable; native binaries still get function-name symbols for free from QBE/linker, so stack traces stay readable. Full source-line DWARF can land when there's concrete demand for systems-level debugging of compiled artifacts.
Sizing. S = 1–2h focused session · M = 3–5h session · L = 1 day · XL = 2+ days (consider splitting further). Numbers in parentheses (eg. "M, ~4h") give the estimate. Effort is for one focused engineer; subtask order matters less than total dependency chain.

CaaS-2 Immutable syntax tree (red-green tree, Roslyn-style) ✅ XL, 2d

Convert the mutable AST to an immutable/persistent representation. The "green" tree is structurally shared (content-addressed); the "red" tree is the per-document overlay with parent links and absolute positions. Required for incremental compilation, safe concurrent access from IDE tooling, and to avoid baking the mutable shape into boot/.

Acceptance:
  • Existing elaborator/typechecker/lowerer use the immutable representation.
  • A node never changes after construction; "modifications" produce new trees that share unchanged subtrees.
  • Memory footprint is bounded — repeated reparses of similar source don't grow the tree pool unboundedly.
  • All existing tests pass.
Subtasks:
  1. Audit current mutable-AST patterns (elaborator stamping hook, typechecker stamping inferredType, etc.) and design replacement.
  2. Implement green-tree node types + interning pool.
  3. Implement red-tree overlay with parent links and positions.
  4. Migrate elaborator to produce new trees rather than mutating.
  5. Migrate typechecker to surface inferredType via SemanticModel (CaaS-3) rather than node stamping.
  6. Migrate lowerer.
  7. Verify byte-equal self-host.
Depends on: CaaS-1

Phase 8.5 — CaaS Consumer APIs

Builds on Phase 0.5 (CaaS Foundation). These stories add the consumer-facing APIs — workspace, symbols, incrementality, code actions — and rewrite existing CLI / formatter tools to be the first real consumers of the foundation. Phase 9b (LSP) consumes these.

Some stories here are required for 1.0 (workspace, symbols, CLI rewrite, stability contract); others are post-1.0 acceptable (incremental parsing/typecheck, code actions) and labelled accordingly in their summaries.
CaaS-4 Workspace abstraction — multi-document project state M, ~4h [1.0]

A Workspace holds a set of Documents (files). Each Document has a content version, a parsed SyntaxTree, and a derived SemanticModel. Operations on the Workspace: open/close/edit/save documents; query across all documents.

Acceptance:
  • API: workspace.openDocument(uri, content), workspace.editDocument(uri, edits), workspace.closeDocument(uri).
  • Change events fire when documents are modified; consumers can subscribe.
  • Cross-document references (a function in file A used in file B) resolve correctly.
Depends on: CaaS-3
CaaS-5 Symbol table API — declarations, references, navigation M, ~4h [1.0]

Public Symbol type with: name, kind (function/type/variable/strata), declaration location, list of reference locations, containing scope. Powers go-to-definition, find-references, rename. Required for LSP go-to-def and hover (Phase 9b).

Depends on: CaaS-4
CaaS-7 Incremental parser — re-parse only changed documents L, ~8h [post-1.0 acceptable]

When a document is edited, re-parse only that document. Downstream artifacts (SemanticModel, diagnostics) invalidate per-document. No full-project reparse. Pure perf optimisation — 1.0 ships without it (full reparse on change) and the LSP is still usable for moderately sized projects.

Acceptance:
  • Editing one document leaves other documents' SyntaxTrees unchanged.
  • Sub-document incremental parsing (only re-parse the changed subtree) is a stretch goal; full-document re-parse is the floor.
Depends on: CaaS-2, CaaS-4
CaaS-8 Incremental typecheck — semantic delta propagation L, ~8h [post-1.0 acceptable]

When document A changes and B doesn't, B's SemanticModel only re-runs if A's exports actually changed (signatures, types, registered strata). Salsa-style query memoisation. Like CaaS-7, a perf concern — 1.0 ships without it.

Depends on: CaaS-7
CaaS-9 CLI rewrite — sgl as a library consumer M, ~4h [1.0]

Rewrite sgl build / run / check to use only the public CaaS API. Proves the API is sufficient for a real consumer. Any internal-only access needed → either expose it or change the implementation.

Depends on: CaaS-3, CaaS-6, 6b-1
CaaS-10 Stability contract for the CaaS API S, ~2h [1.0]

Part of Phase 10 stability work. Declare the CaaS API surface stable at 1.0: additive changes are fine, removals/breaks require a major version. Internal refactors are unconstrained. Document which subsystems can call which.

Depends on: CaaS-9, 10b-1
CaaS-11 Code action / quick-fix API surface M, ~4h [post-1.0 acceptable]

A CodeAction takes a diagnostic and produces zero or more TextEdits that fix it. Foundational for Roslyn-style "Press Ctrl+. to fix this error" workflows in IDEs. Strata authors can register code actions for their diagnostics. Nice IDE polish but not blocking 1.0.

Depends on: CaaS-6
CaaS-12 Formatter as a CaaS consumer (sgl fmt rewrite) M, ~4h [1.0]

Re-implement sgl fmt (originally story 6b-10) on top of the SyntaxTree API. No re-parsing in the formatter — it reads the tree. Proves the API supports format-style tooling.

Depends on: CaaS-2, 6b-10 superseded

Phase 9 — DWARF + LSP

9a — DWARF debug info

The Silicon interpreter (Phase 7) is the primary debug story for 1.0 — it operates on Silicon semantics, shows values as Silicon types, and covers app-developer needs. DWARF wins systems-level debugging of native binaries (crash analysis, profilers, gdb on sigilc itself) — a real but secondary concern. All 9a stories are marked post-1.0 acceptable; ship when there's concrete demand.

Function-name symbols in native binaries come essentially free from QBE → as → ld as long as functions are emitted with proper names — so stack traces stay readable without any DWARF work. Full source-line info, variable inspection, and IDE-level native debugging wait on Phase 9a being prioritised.
