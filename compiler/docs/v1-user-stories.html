Sigil 1.0 — User Stories

Session-sized work units to ship 1.0. Stories are grouped by phase (matching sigil-1.0-roadmap.md) with the in-progress dissolution work as Phase 0 and a cross-cutting Compiler-as-a-Service section as Phase 8.5. Click any story to expand its goal, acceptance criteria, and subtasks. Date: 2026-05-23.

Architectural directives (2026-05-23):

1. Compiler-as-a-Service. Sigil is designed Roslyn-style — the compiler is a library, not just a CLI. Every consumer (sgl tool, LSP server, IDE plugins, formatter, REPL, future analysers) uses the same public API surface. LSP-protocol conformance is optional — the underlying API may be richer. CaaS is split across two phases: Phase 0.5 — CaaS Foundation is required-for-1.0 and must land before the boot/ port (Phase 3) so the self-hosted compiler inherits the right shape. Phase 8.5 — CaaS Consumer APIs is the rest: some 1.0, some post-1.0 stretch. The LSP (v1.1's Phase 9b) consumes both.

2. Interpreter-first debugging. The Silicon interpreter (Phase 7) is the primary debug tool for 1.0 — it shows Silicon values as Silicon types and covers app-developer needs. Native DWARF emission (v1.1's Phase 9a) is moved out of 1.0; native binaries still get function-name symbols for free from QBE/linker, so stack traces stay readable. Full source-line DWARF + LSP + IDE integration land in v1.1 — see docs/v1.1-user-stories.html.
Sizing. S = 1–2h focused session · M = 3–5h session · L = 1 day · XL = 2+ days (consider splitting further). Numbers in parentheses (eg. "M, ~4h") give the estimate. Effort is for one focused engineer; subtask order matters less than total dependency chain.

5d — First-class functions

5d-1 Function reference table management ✅ M, ~4h

New module-level FuncrefTable field on IRModule tracks (a) function names tagged for inclusion in the WASM funcref table and (b) the distinct function signatures used by indirect calls. The @fnref name expression keyword appends a top-level @fn to the table on first reference and returns its slot index as an i32 (subsequent @fnrefs of the same name return the cached index). The WAT emitter materialises (type $sig) / (table N funcref) / (elem (i32.const 0) $fn1 $fn2 …) declarations from the table — only when at least one funcref feature is used, so non-funcref programs preserve byte-equal codegen. 6 e2e tests in src/codegen/funcref.test.ts cover slot-stability, distinct-index uniqueness, and end-to-end dispatch.

5d-2 call_indirect codegen ✅ M, ~4h
Depends on: 5d-1

New IRCallIndirect node (kind: CallIndirect; carries sigKey, args, tableIndex) extends the IR expression union. The WAT emitter emits (call_indirect (type $sig) ARG1 … INDEX) in the per-WASM-spec argument order. Surface syntax: &@call_indirect cb, arg with the signature fixed at i32 → i32 for the 1.0 surface (Vec.map / Result.map / callback hooks all live in this shape); multi-arg + mixed-type variants are mechanical extensions of the same machinery. Direct binary emitter parity is deferred — funcref-using modules currently route through compileToWat → watToWasmSync as a fallback; native section-level encoding (type / table / elem / call_indirect bytes) is a 1.x follow-up that doesn't affect runtime behaviour.

5d-3 Function-type as a first-class SiliconType ✅ M, ~3h
Depends on: 5d-2

Type-system side already existed pre-Phase-5: SiliconType has a Function kind with typed params and result arrays, HM-lite uses it for signature tracking, wasmTypeOf routes Function → i32 (function-table index). Surface annotation syntax landed with the Phase 5 grammar revision via the $fn sigil-type production: :$fn _:R _:T1, _:T2 mirrors the function-definition shape exactly (return-type slot + comma-separated param list, regular and consistent with Silicon's design principles). resolveTypeAnnotation in src/types/typechecker.ts recognises the $fn typename and recursively builds a FunctionOf(params, result) type from fnReturn + fnParams. 4 new tests in src/types/fntype.test.ts cover nullary / unary / n-ary callback annotations and nominal distinctness.

Multi-callback disambiguation resolved. Initially shipped with a known limitation (PEG greedy-match on the inner comma). The follow-up "parens as optional grouping" revision (docs/parens-optional-grouping.md) added optional paren-wrapped forms for both function-definition param lists and $fn type annotations: @fn dispatch a:$fn _:Int (_:Int), b:$fn _:Bool (_:Float) := 0; now parses correctly with two outer params, each a one-arg callback. Parens are a pure grouping operator — present them where you need disambiguation, omit them otherwise.

5d-4 Vec.map proof — first-class function passed as callback ✅ M, ~3h
Depends on: 5a-4 Vec[T], 5d-1 funcref table, 5d-2 call_indirect.

Capstone test that exercises the full first-class-function chain: a Silicon callback flows from &@fnref name at the call site, through the WASM funcref table, into &@call_indirect cb, x inside vec_map_i32_i32, and back out as the mapped element. 5 e2e tests in src/stdlib/vec_map.test.ts exercise: element-by-element application ([1, 2, 3] |> double = [2, 4, 6]), length preservation, source-unchanged invariant (functional semantics), distinct-callback dispatch on the same source Vec, and the empty-Vec edge case. Proves Phase 5d's central claim: Silicon supports higher-order functions through WASM-native funcref + call_indirect, with the callback type expressible via the $fn _:R _:T surface annotation (5d-3).
