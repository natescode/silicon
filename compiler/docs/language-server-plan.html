Silicon Language Server — v1 Alpha Plan

A Language Server Protocol implementation for .si files. Targets VS Code first via the existing vscode-silicon extension; the server itself speaks plain LSP and runs in any editor that supports it (vim, emacs, helix, neovim, sublime).

Status: plan only — no code shipped.
Scope: v1 alpha covers everything an editor needs to feel like it understands Silicon. Strata-aware semantics and project-config parsing are explicit later goals captured here so the architecture leaves room for them, but neither ships in v1.

1. What ships in v1 alpha

Four capabilities, in priority order:

  1. Diagnostics on save / change — surface the existing typechecker errors (already structured per docs/diagnostics.md) as textDocument/publishDiagnostics. Free win: every check the Stage 0 compiler already runs becomes a red squiggle.
  2. Document symbols — outline view + breadcrumb support via textDocument/documentSymbol: every @let, @fn, @var, @extern, @type, @stratum_* in the file.
  3. Go-to definition — textDocument/definition for &name call sites, name identifier references (locals → declaration; globals → top-level def; cross-file via @use resolution).
  4. Hover — textDocument/hover shows the source location and signature of whatever the cursor is on (function name, local, type).

Everything else (rename, references, completion, formatting, semantic tokens) is post-alpha. Capturing them in §6 so the architecture doesn't paint itself into a corner.

2. Architecture

Editor (VS Code / vim / …)
Spawns the server as a subprocess. Speaks LSP over stdio. The vscode-silicon extension grows a thin client wrapper that wires activate() → spawn server → register language client.
Language server (Node, TypeScript)
Long-running process. Loads src/parser, src/elaborator, src/types, src/modules/useResolver directly — same compiler frontend the CLI uses. Maintains an in-memory document store + symbol index. Speaks LSP via vscode-languageserver/node.
Silicon source tree
The user's project. Files are .si. Multi-file via @use 'path.si'. Server tracks the full @use graph for the open workspace.
Why TypeScript Stage 0 and not stage1.wasm? Two reasons. First, Stage 0 already has structured diagnostics (src/errors/diagnostic.ts), source-location-tracking AST nodes, and the elaborator/typechecker. Reusing it is a day-one win. Second, stage1.wasm emits WAT and exits — it doesn't expose a "give me diagnostics for this buffer without finishing compilation" API. Once it does, we can swap the backend without touching the LSP protocol layer. Today's coupling between the server process and the TypeScript frontend is a v1 implementation detail, not a contract.

The flow

documents.onDidChangeContent(({ document }) => {
    debouncedRecheck(document)
})

async function recheck(doc: TextDocument): Promise<void> {
    try {
        const ast = parseAndBuildAst(doc.getText())
        const registry = buildStrataRegistry(ast)
        const { errors: elabErrors } = elaborate(ast, registry)
        const { errors: typeErrors } = typecheck(ast, registry, moduleRegistry)
        const diagnostics = [...elabErrors, ...typeErrors]
            .map(toLspDiagnostic)
        connection.sendDiagnostics({ uri: doc.uri, diagnostics })
    } catch (e) {
        // Parse-error path — single diagnostic from the lex/parse exception
        connection.sendDiagnostics({ uri: doc.uri,
            diagnostics: [parseErrorToDiagnostic(e)] })
    }
}

Diagnostic mapping

The symbol index

Per document, build a flat list of { name, kind, range, selectionRange }. Cached and invalidated on edit.

interface SiliconSymbol {
    name:           string
    kind:           'fn' | 'let' | 'var' | 'extern' | 'type' | 'stratum' | 'local'
    range:          Range  // the entire definition
    selectionRange: Range  // just the name
    container?:     string // the enclosing @fn name, if any
}

function buildSymbolIndex(ast: Program): SiliconSymbol[] {
    const out: SiliconSymbol[] = []
    for (const el of ast.elements) {
        if (el.type === 'Definition') {
            out.push({ name: el.name.name, kind: kindFromKeyword(el.keyword),
                       range: el.sourceLocation,
                       selectionRange: el.name.sourceLocation })
            // Recurse into the body for @local declarations
            collectLocals(el.binding, el.name.name, out)
        }
    }
    return out
}

Resolution algorithm for go-to definition

  1. Find the AST node under the cursor. Must be a Namespace (identifier reference) or FunctionCall.
  2. Single-segment name: walk outward from the cursor position through enclosing @fn's scope (params → hoisted @locals → enclosing @lets of same file → other top-level defs of same file → @use'd files). First match wins.
  3. Multi-segment name (Module::fn): resolve Module against @use 'Module.si'; look up fn in that file's symbol index.
  4. Return the symbol's selectionRange as a Location in the original file.
The @use graph. v1 alpha needs to resolve @use 'path.si' across files in the workspace. Re-use src/modules/useResolver (already battle-tested in the CLI build path) but call it eagerly when the workspace opens — and re-resolve on file save when @use lines change. The graph lives in workspace.ts as an in-memory Map<uri, ResolvedUseEntry[]>.

Hover

Once the symbol index exists, hover is "find the symbol under the cursor, format it as Markdown." Show keyword + signature + the first line of the doc comment (##) preceding the definition.

function hover(uri: string, pos: Position): Hover | null {
    const sym = resolveAtPosition(uri, pos)
    if (!sym) return null
    return {
        contents: { kind: 'markdown',
                    value: `\`\`\`silicon\n${sym.signature}\n\`\`\`\n\n${sym.docComment ?? ''}` },
        range: sym.selectionRange,
    }
}

5. Later goals — strata semantics & project config

5.1 Strata-aware editor semantics

Why this is hard, and why we punt it from v1. Today the LSP can hard-code knowledge of built-in keywords (@if, @loop, @let, …). To honour user-defined strata, the server has to evaluate each project's @stratum_* declarations and register them dynamically — same job src/elaborator/strataLoader.ts does at compile time. Doable, but cross-cuts every feature (completion suggests user strata; hover explains them; definition jumps to @stratum_keyword Foo ('@foo', Node) = { ... }; rename rewrites both the declarator and every &@foo call site).

The mechanism, when it lands:

  1. On workspace open, scan for *.si files containing @stratum_keyword or @stratum_operator.
  2. Build the same ElaboratorRegistry the compiler builds (buildStrataRegistry). Cache it per workspace.
  3. Invalidate the cache when any file containing a @stratum_* changes.
  4. Feed the cached registry to each handler:
    • Completion: include user keywords / operators alongside built-ins.
    • Hover: show the stratum's first-line doc comment + the intrinsic it lowers to.
    • Definition: jump to the @stratum_* declarator.
    • Diagnostics: a &@unknown call becomes "no stratum registered for keyword @unknown" instead of a generic parse error.

This is roughly a Phase 5 the day someone wants it. Architectural pre-condition for v1 alpha: keep the workspace + symbol index as separate concerns from the per-document store, so registry-aware features can layer on without rewriting.

5.2 Project configuration file

Spec deferred to a sibling doc, but the LSP needs to read it. Sketch:

# silicon.toml — at workspace root

[project]
name    = "sigil"
version = "0.1.0"
entry   = "boot/main.si"

[build]
target  = "wasix"            # or "host", "browser", "wasi"
output  = "dist/"

[strata]
include = ["src/strata/**/*.si",
           "src/strata/modules/*.si"]
exclude = []

[lsp]
root_dirs       = ["boot/", "src/"]
diagnostic_mode = "on_save"     # or "on_change"

v1 alpha assumes the workspace root is the project root and that every .si file under it is in scope. Once a config exists, the server reads it on initialize, narrows the file watcher to root_dirs, and resolves @use paths relative to the project root rather than the file's own directory.

This is roughly a Phase 6 the day there's a config-file spec to read. Architectural pre-condition for v1 alpha: don't bake any boot/-specific assumptions into workspace.ts — keep the resolution roots overridable.

6. Open questions

How does the server find Stage 0?

v1 spawns the server with node lsp-silicon/dist/index.js; the server imports Stage 0 from a known relative path inside the monorepo. Once stage1.wasm is the canonical compiler, the server shells out to it. Until then: how do we ship the TS frontend alongside the LSP binary? Bundle with esbuild? Publish as an npm package? Punt for v1 — assume the LSP lives in this monorepo.

Granularity of re-checking

On every keystroke the server has to re-parse + re-elaborate + re-typecheck the changed file. For a small file (boot/std/io.si: 70 lines) this is sub-millisecond. For the full bootstrap source (~3500 LoC concatenated) it's tens of milliseconds. Debounce by 200ms in v1. Move to incremental re-elaboration (cache per-definition IR) only if it becomes a problem.

Cross-file diagnostics

When file A defines @fn foo and file B calls &foo, editing A's signature should invalidate B's typechecking. In v1: re-check every file in the workspace on every save (cheap because files are small). Smart invalidation via the @use graph is a v2 optimisation.

Workspace folder vs single file

VS Code can open a single .si file outside any workspace. In that case there's no @use graph and no symbol index across files. v1 degrades gracefully: diagnostics + intra-file go-to-def still work; cross-file symbols are simply absent.

7. Acceptance checklist for v1 alpha

  • must   F1 → Developer: Show Logs → Silicon LSP shows the server initialising on workspace open with no errors.
  • must   Opening any .si file shows the existing TextMate highlighting and activates the LSP (status bar shows the language as "Silicon").
  • must   A deliberate type error in boot/parser/parse.si renders a red squiggle with the exact same message bun run src/sigil_cli.ts would print.
  • must   The Outline pane on boot/parser/parse.si lists every @fn at the top level.
  • must   F12 on any &parse_* call in parse.si jumps to the matching @fn definition.
  • must   F12 on a @local's usage inside parse_definition jumps to its @local foo := ... declaration.
  • must   Hovering on a function name shows its @fn name:Type ... signature line.
  • should   Server CPU usage during idle editing is < 5% (debouncing works).
  • should   Server memory after 30 minutes of editing the full boot/ tree is < 200 MB.

8. Build order (in commit-sized slices)

  1. Slice 1   scaffolding: lsp-silicon/package.json, tsconfig.json, src/index.ts stub. No handlers, just onInitialize.
  2. Slice 2   VS Code client wrapper: vscode-silicon/client/extension.ts spawns the server; activation event on .si file open.
  3. Slice 3   document store + debounced onDidChangeContent recheck → diagnostics.
  4. Slice 4   symbol index per document; documentSymbolProvider implementation.
  5. Slice 5   @use graph in workspace.ts; cross-file symbol resolution.
  6. Slice 6   definitionProvider for single-segment names (locals + same-file).
  7. Slice 7   definitionProvider extended to cross-file via @use graph.
  8. Slice 8   hoverProvider: signature + first-line doc comment.
  9. Slice 9   published to the vscode-silicon extension as an optional capability (gated by a setting until stable).

Each slice should land on a feature branch and get a green run of the existing 30 WASIX smoke tests plus at least one new test exercising the slice's handler against a fixture .si file.

9. Non-goals for v1 alpha

  • No formatter. Style decisions haven't been made.
  • No code actions / quick fixes. Real fixes need the elaborator to suggest them; today's diagnostics are reports, not suggestions.
  • No semantic tokens. TextMate grammar carries the visual load.
  • No inline values / debug adapter. WASI debugging is upstream-wasmer territory.
  • No live preview / playground integration. The web playground already exists; embedding it in VS Code is a separate concern.
  • No incremental compilation. Re-check the whole file on each debounced edit. Smart invalidation is v2.
