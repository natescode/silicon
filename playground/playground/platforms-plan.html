⬡ Sigil / Design Plan

Silicon Platform System

A first-class way for Silicon programs to declare which runtime they target and which host APIs they need

Overview

Right now the web APIs (canvas, console, math, DOM) all live in a single flat file — src/strata/modules/web.si. A program using the web platform implicitly gets all of them. There is no way to declare "I target the web" from the program side, no way to enable only what you need, and no framework for adding a second platform (Node, native WASI, etc.).

A Platform is a named set of host API groups that defines the runtime environment the compiled WASM will run inside. Platforms are selected in project.si — a Silicon source file at the project root that acts as the project manifest. Individual API groups within a platform are called features and are opt-in.

Today No platform declaration.
All web externs implicitly available.
No project config file.
Adding Node would require a separate pass.
→
With Platforms project.si declares platform + features.
Compiler loads only the selected APIs.
Platforms::web { canvas, game }
New platforms just add a folder under src/platforms/.
Relationship to Modules: Platforms are an evolution of the module system, not a replacement. A platform is a curated, environment-specific collection of extern declarations — conceptually the same as src/strata/modules/web.si but split into features, versioned as a unit, and declared explicitly by the program.

User Stories

Game developer
I want to write a canvas game in Silicon. I need a game loop that calls my tick function automatically each frame, and canvas draw calls for rendering. I don't want to wire up requestAnimationFrame manually — the platform should handle the loop and I just export tick.
Web developer
I want to build a Silicon WASM module that manipulates the DOM and uses console_log. I only need the core web APIs — I don't want a game loop auto-starting. The platform should not pull in canvas or game APIs unless I ask.
Server developer (future)
I want to compile Silicon to WASM and run it inside Node.js with access to fs and http APIs. I should be able to declare @platform Platforms::node { fs, http } in project.si and the compiler should pull in the Node extern declarations automatically.
Compiler contributor
I want to add a new platform (node) without touching the core compiler. I should be able to drop a src/platforms/node/ folder with feature .si files and a metadata file, and the platform system will discover it.

Silicon Syntax — project.si

project.si is a Silicon source file placed at the project root. The compiler reads it before everything else. It may contain only platform declarations and top-level Silicon definitions that are scoped to project configuration — no imperative code, no function bodies.

Canonical form

; project.si — Silicon project manifest
; Declares the target platform and which feature groups to enable.

@platform Platforms::web, { Platforms::web::canvas, Platforms::web::game };

Web platform — core only (no features)

@platform Platforms::web;

Web platform with canvas and game loop

; Snake, breakout, any rAF-driven game
@platform Platforms::web, { Platforms::web::canvas, Platforms::web::game };

Node platform (future)

@platform Platforms::node, { Platforms::node::fs, Platforms::node::http };
Grammar note: @platform is a new definition kind (like @fn, @extern, @export). It maps to a new IR kind IR::meta_platform and is handled entirely as a compiler directive — it emits no WAT. The feature list { … } reuses the existing block body syntax. No grammar changes are needed — this is registered as a new defkind in defkinds.si, exactly like @export was added.

What @platform means to the compiler

  • Load the platform core file: src/platforms/web/core.si
  • For each listed feature, load its .si file: src/platforms/web/canvas.si, src/platforms/web/game.si
  • All loaded externs become available in the program's module registry under the web:: namespace
  • The game feature additionally emits a JS runtime wrapper that calls requestAnimationFrame and invokes the exported tick function
open question — short form: Should { canvas, game } be allowed as shorthand instead of the fully-qualified { Platforms::web::canvas, Platforms::web::game }? The long form is unambiguous but verbose. The short form requires the compiler to infer the platform scope from context. Recommendation: start with the fully-qualified form; add the shorthand later as a convenience desugaring.

Architecture

Relationship to existing module system

The existing module system (from modules-plan.html) loads .si extern declaration files and registers them in a ModuleRegistry. Platforms sit on top of the module system — a platform is a curated set of module files that the platform loader feeds into the same ModuleRegistry. No new registry type is needed: the compiler ends up with one unified ModuleRegistry regardless of whether the externs came from a hand-dropped modules/Draw.si or a platform feature file.

Compiler pipeline position

project.si → parsePlatformConfig → PlatformConfig
                            ↓
     Source → Parse → AST → buildStrataRegistry
                            → loadPlatform(config) → ModuleRegistry
                            → loadUserModules → ModuleRegistry
                            → elaborate → TypeCheck → Codegen → WAT

parsePlatformConfig is a tiny pre-pass: it parses only project.si, looks for @platform directives, and returns a PlatformConfig record. The full compiler pipeline is then set up with the right module files.

Platform file layout (compiler side)

src/platforms/ ← ships with the compiler; not user-visible
  web/
    core.si ← console, math, performance, date — always loaded with web
    canvas.si ← canvas_fill_rect, canvas_set_fill, etc.
    game.si ← no extra externs; game loop is JS-side only
    dom.si ← set_html, clear_html
    platform.json ← metadata: name, version, available features
  node/ ← (planned)
    core.si
    fs.si
    http.si

src/strata/modules/ ← will be superseded by src/platforms/ but kept for backward compat
  web.si ← current monolithic web externs (flatten into web platform features)

project/ ← user's project directory
  project.si ← user's platform declaration — @platform Platforms::web, {...}
  main.si
  modules/ ← third-party user modules (unchanged)

platform.json — platform metadata

{
  "name": "web",
  "version": "1.0.0",
  "features": [
    { "name": "canvas", "file": "canvas.si",  "runtime": "canvas.js" },
    { "name": "game",   "file": "game.si",    "runtime": "game.js",  "requiresExport": "tick" },
    { "name": "dom",    "file": "dom.si" }
  ],
  "runtimeHost": "web-env.js"
}

The requiresExport field lets the compiler validate that a program using the game feature actually exports tick. The runtime field names a JS snippet the playground (or build tool) should inject to support that feature.

PlatformConfig type

interface PlatformConfig {
  platform: 'web' | 'node' | 'wasi'   // from @platform directive
  features: string[]                     // e.g. ['canvas', 'game']
}

Feature file split (migrating web.si)

The current src/strata/modules/web.si is a monolith. It gets split into feature files that match the platform catalogue:

FeatureCurrent location in web.siNew file
core console_log, math_*, performance_now, date_now src/platforms/web/core.si
canvas canvas_set_fill, canvas_fill_rect, canvas_width, … src/platforms/web/canvas.si
dom set_html, clear_html src/platforms/web/dom.si
game (no externs — loop is JS-side) src/platforms/web/game.si (empty; metadata only)

The web::game Feature

web::game is the most interesting feature because it has no extern declarations — its effect is entirely on the JS host side. When the game feature is enabled, the JS runtime wraps the WASM module in a requestAnimationFrame loop and calls the WASM export tick every frame.

Silicon side — user writes a tick function and exports it

; project.si
@platform Platforms::web, { Platforms::web::canvas, Platforms::web::game };

; main.si
@fn tick _ := {
    ; called every frame by the JS game loop
    &web::canvas_clear_rect 0.0, 0.0, 400.0, 400.0;
    ; ... draw calls ...
};

@export tick;

JS host side — game.js runtime snippet

// Injected by the build tool / playground when web::game is enabled
function startGameLoop(instance) {
  function loop() {
    instance.exports.tick();
    requestAnimationFrame(loop);
  }
  requestAnimationFrame(loop);
}

The game.si feature file is empty (no externs needed). Its presence in platform.json tells the build tool to inject game.js and to validate that the program exports tick.

What changes in the playground: Currently the playground checks whether the compiled WASM exports tick and starts the rAF loop manually. With the platform system, this heuristic becomes a declared contract — the game loop only starts when web::game is enabled in project.si.

Missing project.si — Fallback Behavior

Many programs (tests, e2e fixtures, demo files) will not have a project.si. The compiler needs sensible fallback behavior:

Option A — implicit web platform (recommended)

If no project.si is found, assume @platform Platforms::web with all features enabled. Matches current behavior — existing programs just work.

Option B — no platform (strict)

If no project.si is found, emit a warning and load no platform externs. Programs that use web:: calls will fail at type-check with "unknown module web". Stricter but breaks all existing programs.

Recommendation: Option A for backward compatibility. Add a CLI flag --no-platform-fallback for strict mode in CI/new projects.

Open Design Questions

? Q1 — Should features be declared with full path or short names?

Full path: @platform Platforms::web, { Platforms::web::canvas, Platforms::web::game }
Short form: @platform Platforms::web, { canvas, game }

The full path is unambiguous and consistent with how namespace values are referenced everywhere else in Silicon. The short form is more ergonomic. Since features are always scoped to the selected platform, the short form is unambiguous in context. Recommendation: full path as canonical; short form as compile-time desugaring.

? Q2 — Should Platforms be a Silicon built-in enum type?

In other languages, build configuration is usually a separate concern from the language (Cargo.toml, package.json, build.gradle). In Silicon the config is Silicon, which means Platforms would need to be a special compile-time namespace.

One option: define Platforms as a compiler-reserved namespace (like WASM:: and IR::) — not a user-definable type, just a recognised token. The compiler pattern-matches @platform Platforms::X as a directive and never lowers it to WASM.

This keeps project.si feeling like Silicon without requiring actual enum type-checking of platform names. A typo like Platforms::webb is caught at platform-loading time with a clear error: "Unknown platform 'webb' — available: web, node".

? Q3 — Where does web-env.js go? One file or per-feature?

Currently playground/web-env.js provides all web host bindings (console, canvas, DOM, rAF) in one file. With the platform system, the canvas bindings are only needed when web::canvas is enabled, and the game loop is only needed when web::game is enabled.

Options: (a) keep one monolithic web-env.js that provides all bindings unconditionally; (b) split into per-feature snippets that the playground stitches together based on the active platform config. Option (a) is simpler. Option (b) is cleaner long-term and enables tree-shaking the host side. Recommendation: keep monolithic for now, design for per-feature later.

? Q4 — Does the playground need to read project.si?

The playground today runs the compiler on user-typed Silicon source. With platforms, the compiler will look for project.si relative to the source file's directory. In the playground there is no file system — the source lives in a textarea.

Options: (a) the playground passes a virtual project.si derived from a "Platform" dropdown in the UI; (b) the playground always runs with the implicit fallback platform (all web features enabled). Option (b) requires zero UI changes. Option (a) lets playground users experiment with platform restrictions. Recommendation: start with (b); add platform dropdown as a playground enhancement later.

Implementation Steps

1
Split web.si into feature files
src/platforms/web/

Create src/platforms/web/ and split the existing src/strata/modules/web.si into four files according to the feature table above. Add platform.json listing the features. 

src/platforms/web/
  core.si       # console_log*, math_*, performance_now, date_now
  canvas.si     # canvas_set_fill, canvas_fill_rect, canvas_width, …
  dom.si        # set_html, clear_html
  game.si       # empty — no externs; triggers game loop in JS host
  platform.json # metadata

Keep src/strata/modules/web.si intact for backward compatibility. It will be deprecated once project.si support is complete.

2
PlatformConfig type + parsePlatformConfig
src/platforms/config.ts

Write a tiny parser that reads project.si, locates any @platform definitions (matching the existing DefKind pattern), and returns a PlatformConfig. No new grammar needed — @platform is registered as a new defkind and the existing Silicon parser handles it. 

export interface PlatformConfig {
  platform: 'web' | 'node' | 'wasi'
  features: string[]   // e.g. ['canvas', 'game']
}

export function parsePlatformConfig(projectSiPath?: string): PlatformConfig
// Returns { platform: 'web', features: ['canvas', 'game', 'dom'] } if no file
3
Register @platform as a new DefKind
src/strata/defkinds.si
src/ir/irKinds.ts

Add @platform to the defkind registry as a new IR::meta_platform kind. This teaches the parser to accept @platform Platforms::web, { … }; as valid Silicon syntax without grammar changes. 

; src/strata/defkinds.si — add alongside @extern, @export, etc.
@stratum_keyword "@platform"  := { IR::meta_platform };

In the IR lowerer, encountering meta_platform is a no-op — the directive was already consumed by parsePlatformConfig in the pre-pass. The lowerer skips it with no error.

depends on step 2 (PlatformConfig)
4
Platform loader — loadPlatform
src/platforms/loader.ts

Write loadPlatform(config: PlatformConfig): string[]. It reads platform.json for the selected platform, collects the .si file paths for core plus each enabled feature, reads them, and returns them as Silicon source strings ready to pass as extraSources to buildStrataRegistry (which already feeds them into the module registry pipeline). 

export function loadPlatform(config: PlatformConfig): string[] {
  const meta = readPlatformJson(config.platform)    // platform.json
  const files = ['core', ...config.features]         // core always first
    .map(f => meta.features.find(x => x.name === f))
    .filter(Boolean)
    .map(f => readFileSync(path.join(PLATFORMS_DIR, config.platform, f.file), 'utf-8'))
  return files
}
depends on step 1 (feature files exist) and step 2 (PlatformConfig type)
5
Wire into the compiler entry point
src/index.ts

In the main compiler entry, detect project.si, call parsePlatformConfig, call loadPlatform, and pass the resulting source strings as extraSources into the existing buildStrataRegistry call. Zero other pipeline changes. 

+ const platformConfig = parsePlatformConfig(projectSiPath)
+ const platformSources = loadPlatform(platformConfig)
  const registry = buildStrataRegistry(ast, [
+   ...platformSources,
    ...extraSources,  // user modules
  ])
depends on steps 2, 3, 4
6
Add @platform validation in type-checker
src/types/typechecker.ts

When the game feature is enabled (requiresExport: "tick" in platform.json), the type-checker should verify that the program exports a function named tick. This is a new check — if the export is missing, emit a clear error: "Platform feature web::game requires an exported function named 'tick'".

depends on step 4 (platform metadata available at type-check time)
7
Update playground — read platform from project config
playground/index.html
playground/server.ts

For now the playground uses the implicit fallback (all web features). No UI changes needed. The server should pass an empty projectSiPath to the compiler so it gets the full-featured web platform by default.

As a follow-on, add a Platform selector dropdown to the playground header that lets the user toggle features (canvas, game, dom) and re-compiles with a virtual project.si.

depends on step 5 (compiler accepts platform config)

Non-Goals

  • Package management for platforms (download, versioning, lock files)
  • Multiple platforms in one program (a program targets exactly one platform)
  • User-defined platforms (only compiler-shipped platforms for now)
  • Platform-conditional code inside Silicon source (@if platform == web)
  • Node platform implementation (planning only; implementation is a future round)
  • Per-feature tree-shaking of the JS host (web-env.js stays monolithic for now)
  • Playground platform selector UI (deferred to after core implementation)
