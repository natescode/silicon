⬡ Sigil Playground / Design Plan

String Round-Trip: greet(String) → String

Passing a JS string into an exported Silicon function and reading the result back

Goal

Right now greet takes a plain integer. We want it to accept a JavaScript string as input and return a Silicon string back to JS — a full round-trip through WASM linear memory.

JavaScript greet("Alice")
→
Silicon (WASM) msg: String → String
→
JavaScript "Hello, Alice!"
Note: Steps 1–6 prove the round-trip with a simple echo function. Steps 7–10 extend Silicon with a ++ string-concat operator so greet can return "Hello, Alice!" dynamically.

How Silicon Strings Live in WASM Memory

Every Silicon String is an i32 pointer into WASM linear memory. At that address sits a length-prefixed UTF-16 LE buffer:

ptr + 0
ptr + 4
ptr + 5
ptr + 6
ptr + 7
…
i32 byteLen
'H' lo
'H' hi
'i' lo
'i' hi
…
byteLen = charCount × 2  |  each char = 2 bytes, little-endian  |  "Hi" → byteLen=4, bytes: 48 00 69 00

Static string literals (e.g. 'Hello, World!') are baked into the WASM data segment at compile time. Dynamic strings — built at runtime or written from JS — live on Silicon's bump-allocated heap (managed by $alloc in std.wat).

To pass a string from JS into Silicon, JS must write the length-prefixed UTF-16 LE bytes into WASM memory and hand Silicon the pointer. To read a string back out, JS reads from instance.exports.memory.buffer at the returned pointer. Both operations require WASM memory to be exported, and allocation to be reachable from JS.

Implementation Steps

1
Export WASM memory and the allocator
src/codegen/std.wat

JS cannot read or write string data without a handle to WASM linear memory. Similarly, allocating heap space for a new string requires calling Silicon's $alloc bump allocator. Both must be exported from the WAT module. 

(memory 1)
+(export "memory" (memory 0))     ;; exposes instance.exports.memory

(func $alloc (param $size i32) (result i32)
  …
)
+(export "alloc" (func $alloc))   ;; exposes instance.exports.alloc(byteLen)

After this, instance.exports.memory.buffer is a live ArrayBuffer that JS can wrap in a DataView or Uint8Array to read and write strings.

2
Add JS bridge helpers to web-env.js
playground/web-env.js

Two helpers are needed. readLenString already exists internally; it just needs to be exposed on the returned object. writeLenString is new. 

// Write a JS string into Silicon's heap. Returns the i32 pointer.
function writeLenString(str) {
    if (!wasmMemory || !wasmAlloc) return 0
    const byteLen = str.length * 2              // UTF-16 LE: 2 bytes per char
    const ptr = wasmAlloc(4 + byteLen)          // allocate from Silicon heap
    const view = new DataView(wasmMemory.buffer)
    view.setInt32(ptr, byteLen, true)           // write byte-length prefix (LE)
    for (let i = 0; i < str.length; i++) {
        view.setUint16(ptr + 4 + i * 2, str.charCodeAt(i), true)
    }
    return ptr
}

// bindInstance must also capture alloc:
bindInstance: function(instance) {
    if (instance?.exports?.memory) wasmMemory = instance.exports.memory
    if (instance?.exports?.alloc)  wasmAlloc  = instance.exports.alloc
},

// Expose both helpers publicly:
return {
    imports, bindInstance, flush,
    readString:  readLenString,    // (ptr) → JS string
    writeString: writeLenString,   // (jsStr) → i32 ptr
}
Depends on Step 1 — alloc must be exported before bindInstance can capture it

When a function's Silicon return type is "String", the raw i32 return value is a pointer into WASM memory. Read it back with webEnv.readString rather than displaying the raw integer. 

function formatResult(value, wasmType, siliconResult) {
    if (siliconResult === 'String') return webEnv.readString(value)

    return String(value)

7
Add $str_concat to the stdlib
src/codegen/std.wat

Add a WAT function that allocates a new buffer large enough for both strings, copies each in turn, and returns the pointer. This is a pure runtime helper — no new WASM instructions or intrinsics needed. 

;; str_concat — concatenate two Silicon UTF-16 LE strings
;; Inputs:  $a and $b are i32 pointers to length-prefixed string buffers
;; Returns: i32 pointer to a new heap-allocated string = $a ++ $b
(func $str_concat (param $a i32) (param $b i32) (result i32)
  (local $lenA   i32)   ;; byte length of $a's payload
  (local $lenB   i32)   ;; byte length of $b's payload
  (local $total  i32)   ;; combined byte length
  (local $dst    i32)   ;; pointer to the new buffer

  ;; read byte-lengths from the 4-byte headers
  (local.set $lenA (i32.load (local.get $a)))
  (local.set $lenB (i32.load (local.get $b)))
  (local.set $total (i32.add (local.get $lenA) (local.get $lenB)))

  ;; allocate 4 (header) + total payload bytes
  (local.set $dst (call $alloc (i32.add (i32.const 4) (local.get $total))))

  ;; write the combined byte-length header
  (i32.store (local.get $dst) (local.get $total))

  ;; copy $a's payload (skip 4-byte header of $a, write after 4-byte header of $dst)
  (memory.copy
    (i32.add (local.get $dst) (i32.const 4))
    (i32.add (local.get $a)   (i32.const 4))
    (local.get $lenA))

  ;; copy $b's payload immediately after $a's payload
  (memory.copy
    (i32.add (i32.add (local.get $dst) (i32.const 4)) (local.get $lenA))
    (i32.add (local.get $b)   (i32.const 4))
    (local.get $lenB))

  (local.get $dst)
)

memory.copy is a bulk-memory WASM instruction (MVP+ extension, enabled in all modern runtimes). No separate export is needed — $str_concat is called internally by the Silicon ++ stratum.

8
Define the ++ stratum for string concatenation
src/strata/strings.si

Create a new Silicon stratum file that registers ++ as an operator. The stratum body calls str_concat — a Silicon function backed by the WAT helper added in step 7. This is not a new WASM intrinsic; WASM intrinsics must map exactly 1-to-1 to a single WASM opcode. Strata bodies may span multiple lines and freely mix existing WASM:: intrinsics with ordinary Silicon function calls. 

# src/strata/strings.si
# String operator strata for Silicon.

@stratum Concat := {
    Compiler::register::operator('++');
    str_concat(a, b)
};
# `a` and `b` are the left- and right-hand operands injected by the elaborator.
# str_concat is resolved to $str_concat in std.wat at link time.

The elaborator sees x ++ y, looks up the Concat stratum, and expands the body with a = x and b = y. The result is a call to $str_concat — no intermediate WASM intrinsic registration needed.

Type rule: The type checker must enforce that both operands of ++ are String and that the result is String. This can be expressed as part of the stratum's type annotation or enforced in the type checker's operator elaboration pass.
Depends on Step 7 — str_concat must exist in std.wat
9
Extend the strata loader to handle Silicon function calls
src/elaborator/strataLoader.ts

The current strata loader recognises WASM::opcode and IR::kind tokens in stratum bodies. To support calls like str_concat(a, b), the loader must also handle plain Silicon function-call syntax and emit the appropriate IR Call node (or WAT call $str_concat) during expansion. 

// strataLoader.ts — conceptual extension

function expandStratumBody(body, bindings) {
    // Existing: WASM:: → inline instruction
    if (body.kind === 'WasmIntrinsic') {
        return emitWasmInstr(body.opcode, bindings)
    }
    // Existing: IR:: → IR node construction
    if (body.kind === 'IrKind') {
        return buildIrNode(body.irKind, bindings)
    }
    // New: plain Silicon function call → Call IR node
    if (body.kind === 'FunctionCall') {
        const args = body.args.map(arg => bindings[arg] ?? expandStratumBody(arg, bindings))
        return { kind: 'Call', callee: body.name, args }
    }
    // New: multi-statement body (sequence of the above)
    if (body.kind === 'Block') {
        return body.stmts.map(s => expandStratumBody(s, bindings))
    }
}

This change is additive — existing WASM intrinsic strata continue to work unchanged. The parser for .si stratum files just needs to recognise fnName(args…) as a FunctionCall node alongside the existing WASM::opcode and IR::kind forms.

Depends on Step 8 — the Concat stratum must be registered before the loader processes it
10
Update greet to use ++ for a dynamic greeting
playground/index.html

Once steps 7–9 are complete, the playground example can produce a real greeting by concatenating the literal prefix 'Hello, ' with the caller-supplied name. 

@fn greet msg := {
    web_console_log_str(msg);
    msg
};
@fn greet msg := {
    result := ('Hello, ' ++ msg);
    web_console_log_str(result);
    result
};

The elaborator rewrites 'Hello, ' ++ msg into str_concat('Hello, ', msg), which lowers to (call $str_concat (i32.const <literal_ptr>) (local.get $msg)) in WAT. The JS exports panel then reads the returned pointer with webEnv.readString(result) and displays "Hello, Alice!".

Depends on Steps 7, 8, 9 — stdlib, stratum, and loader must all be in place
